Testando o Stripe na Aplicação Local (Fluxo Completo)
A ideia aqui é simular o fluxo de ponta a ponta — como se você fosse o cliente, abrindo sua aplicação no navegador, clicando em "Pagar", preenchendo o formulário e recebendo a confirmação — tudo localmente.

Passo 1: Garantir que está no modo teste
No seu código, use as chaves de teste (pk_test_... e sk_test_...). Normalmente isso é feito via variáveis de ambiente:

# .env (ambiente de desenvolvimento)
STRIPE_SECRET_KEY=sk_test_SuaChaveSecreta
STRIPE_PUBLISHABLE_KEY=pk_test_SuaChavePublicavel
STRIPE_WEBHOOK_SECRET=whsec_... # será preenchido no passo 3
Com essas chaves, o Stripe sabe que tudo é simulação — nenhum dinheiro real é movimentado.

Passo 2: Subir a aplicação localmente
Inicie o seu servidor normalmente. Por exemplo:

Copy# Node.js / Express / Next.js
npm run dev
# → Rodando em http://localhost:3000

# Python / Django / Flask
python manage.py runserver
# → Rodando em http://localhost:8000

# Ruby on Rails
rails server
# → Rodando em http://localhost:3000
A aplicação já funciona localmente com o Stripe em modo teste — não precisa de domínio público para criar pagamentos. O formulário de cartão (Stripe Elements ou Checkout) funciona perfeitamente em localhost.

Passo 3: Conectar os webhooks ao localhost com o Stripe CLI
Este é o passo crucial. Quando um pagamento é concluído, o Stripe envia eventos (webhooks) para o seu servidor. Como o Stripe não consegue acessar localhost diretamente, o Stripe CLI faz essa ponte:

Terminal 1 — Sua aplicação:

Copynpm run dev  # servidor rodando em localhost:3000
Terminal 2 — Stripe CLI escutando eventos:

Copystripe listen --forward-to localhost:3000/api/webhooks/stripe
O CLI vai exibir algo assim:

Ready! Your webhook signing secret is 'whsec_1234abcd...' (^C to quit)
Copie esse whsec_... e coloque na variável STRIPE_WEBHOOK_SECRET do seu .env. Reinicie o servidor se necessário.

Passo 4: Testar o fluxo como usuário
Agora abra o navegador em http://localhost:3000 e faça o fluxo de compra normalmente. Quando chegar no formulário de pagamento, use os dados de teste:

Campo	Valor
Número do cartão	4242 4242 4242 4242
Validade	Qualquer data futura (ex: 12/34)
CVC	Qualquer 3 dígitos (ex: 123)
CEP	Qualquer valor (ex: 01001-000)
Nome	Qualquer nome
Clique em Pagar. O pagamento será processado com sucesso no modo teste.

Passo 5: Verificar que tudo funcionou
Após clicar em "Pagar", confira em três lugares:

No terminal do Stripe CLI, você verá os eventos sendo encaminhados:

2026-03-23 10:32:15  --> payment_intent.created [evt_1234]
2026-03-23 10:32:15  --> charge.succeeded [evt_5678]
2026-03-23 10:32:16  --> checkout.session.completed [evt_9012]
2026-03-23 10:32:16  <-- [200] POST http://localhost:3000/api/webhooks/stripe
No terminal da sua aplicação, verifique que o handler de webhook processou o evento corretamente (ex: logs de "Pagamento confirmado", criação de pedido no banco, etc.).

No Dashboard do Stripe (dashboard.stripe.com), no sandbox, você verá o pagamento de teste listado com todos os detalhes.

Passo 6: Testar cenários de erro
Além do caminho feliz, teste o que acontece quando as coisas dão errado usando cartões especiais:

# Cartão recusado — veja se sua UI mostra erro amigável
4000 0000 0000 0002

# Fundos insuficientes
4000 0000 0000 9995

# Requer autenticação 3D Secure — testa o fluxo de popup/redirect
4000 0025 0000 3155

# Cartão expirado
4000 0000 0000 0069
Para o cartão de 3D Secure (4000 0025 0000 3155), o Stripe vai mostrar um modal de autenticação simulado. Você pode clicar em "Complete" para aprovar ou "Fail" para recusar.

Passo 7: Testar assinaturas e eventos recorrentes (se aplicável)
Se sua aplicação trabalha com assinaturas (subscriptions), você pode simular o ciclo completo:

Copy# No Stripe CLI, avance o relógio do teste para simular renovação:
stripe trigger invoice.payment_succeeded

# Simular falha na cobrança recorrente:
stripe trigger invoice.payment_failed

# Simular cancelamento de assinatura:
stripe trigger customer.subscription.deleted
Confira no terminal da aplicação se os handlers processam esses eventos corretamente (ex: desativar acesso do usuário ao receber customer.subscription.deleted).
