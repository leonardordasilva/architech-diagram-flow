Contexto e Objetivo Geral
O MicroFlow Architect é um editor visual de diagramas de arquitetura de microsserviços com geração por IA (Google Gemini). O objetivo deste PRD é guiar a implementação de um conjunto de melhorias críticas de segurança, arquitetura, performance, qualidade de código e novas funcionalidades, transformando a aplicação em uma plataforma robusta, segura e colaborativa para equipes de engenharia.
Todas as implementações devem preservar 100% das funcionalidades existentes (drag-and-drop, geração por IA, exportação, undo/redo, dark mode, auto-layout Dagre, análise de arquitetura) enquanto adicionam as melhorias descritas abaixo.

ÉPICO 1 — Segurança: Proteção da API Key do Gemini
Prioridade: 🔴 CRÍTICA — Implementar primeiro, antes de qualquer outra mudança.
Contexto do problema: Atualmente, o arquivo services/geminiService.ts lê a chave da API diretamente via process.env.API_KEY e executa as chamadas ao Google Gemini inteiramente no browser do usuário. Em uma build Vite de produção, isso expõe a chave no bundle JavaScript, tornando-a visível para qualquer pessoa que inspecione o código-fonte.
O que deve ser implementado:
Criar uma camada de backend usando Supabase Edge Functions (já que @supabase/supabase-js está instalado como dependência) para intermediar todas as chamadas ao Google Gemini. O frontend nunca deve chamar o Gemini diretamente.
Especificação técnica:
Criar duas Supabase Edge Functions:
•	supabase/functions/generate-diagram/index.ts — recebe { description: string } no body, chama o Gemini com o prompt de geração, retorna { nodes, edges }.
•	supabase/functions/analyze-architecture/index.ts — recebe { nodes, edges } no body, chama o Gemini com o prompt de análise, retorna { markdown: string }.
Ambas as funções devem:
•	Ler GEMINI_API_KEY do ambiente do Supabase (secrets), nunca do frontend.
•	Implementar o mesmo sistema de fallback multi-modelo já existente (gemini-2.0-flash → gemini-3-flash-preview → gemini-flash-latest → gemini-1.5-flash-8b).
•	Retornar erros HTTP adequados (400, 429, 500, 503) com mensagens em português.
•	Ter cabeçalhos CORS configurados para aceitar requisições apenas da origem da aplicação.
O geminiService.ts no frontend deve ser refatorado para chamar essas Edge Functions usando o cliente Supabase já configurado, em vez de chamar o Gemini diretamente. A variável process.env.API_KEY deve ser completamente removida do frontend.
Critério de aceite: Ao abrir o DevTools do navegador e inspecionar o bundle JS ou as variáveis de ambiente do processo, nenhuma string relacionada a API_KEY, gemini, AIza ou chaves de API deve ser encontrada no código executado no browser.

ÉPICO 2 — Arquitetura: Gerenciamento de Estado Global com Zustand
Prioridade: 🟠 ALTA
Contexto do problema: Todo o estado da aplicação (nós, arestas, histórico de undo/redo, título do diagrama, estado dos modais) está centralizado em App.tsx, que também contém toda a lógica de negócio. Isso cria um componente com excesso de responsabilidades e um padrão de prop drilling extenso onde callbacks são passados por múltiplas camadas de componentes.
O que deve ser implementado:
Criar um store global com Zustand para gerenciar o estado do diagrama, e utilizar o middleware zundo para substituir o sistema de undo/redo baseado em snapshots completos por um sistema baseado em patches (deltas), mais eficiente em memória.
Especificação técnica:
Criar o arquivo store/diagramStore.ts com a seguinte estrutura de estado e ações:
Copy// Estado gerenciado pelo store:
interface DiagramState {
  nodes: ArchitectureNode[];
  edges: ArchitectureEdge[];
  diagramTitle: string;
  isAnalyzing: boolean;
  analysisResult: string | null;
  
  // Ações
  setNodes: (nodes: ArchitectureNode[]) => void;
  setEdges: (edges: ArchitectureEdge[]) => void;
  setDiagramTitle: (title: string) => void;
  addNode: (node: ArchitectureNode) => void;
  removeNode: (nodeId: string) => void;
  updateNodeData: (nodeId: string, data: Partial<CustomNodeData>) => void;
  addEdge: (edge: ArchitectureEdge) => void;
  removeEdge: (edgeId: string) => void;
  clearDiagram: () => void;
  importDiagram: (nodes: ArchitectureNode[], edges: ArchitectureEdge[], title: string) => void;
  setAnalysisResult: (result: string | null) => void;
  setIsAnalyzing: (value: boolean) => void;
}
O store deve ser envolvido com o middleware temporal do zundo para undo/redo automático com limite de 50 entradas. O App.tsx deve ser refatorado para consumir o store via hooks (useDiagramStore), removendo os useState correspondentes e a passagem de props de estado para componentes filhos.
Critério de aceite: O App.tsx não deve conter useState para nodes, edges, diagramTitle ou analysisResult. Todos os componentes que precisam desses dados devem consumi-los diretamente do store via hook, sem recebê-los como props.

ÉPICO 3 — UX: Auto-Save e Recuperação de Sessão
Prioridade: 🟠 ALTA
Contexto do problema: O usuário perde todo o trabalho se fechar a aba acidentalmente ou se o navegador travar. A única forma de salvar é exportar manualmente como JSON.
O que deve ser implementado:
Criar um hook useAutoSave que persiste automaticamente o estado do diagrama no localStorage com debounce de 1,5 segundos após qualquer modificação. Ao inicializar a aplicação, se existir um diagrama salvo, exibir um banner de recuperação no topo do canvas.
Especificação técnica:
Criar hooks/useAutoSave.ts:
•	Chave de storage: microflow_autosave_v1.
•	Estrutura salva: { nodes, edges, title, savedAt: ISO8601string, version: "1" }.
•	Debounce de 1500ms para evitar escritas excessivas.
•	Não salvar se nodes.length === 0 (evitar sobrescrever com canvas vazio).
Criar o componente components/RecoveryBanner.tsx:
•	Exibido apenas na inicialização quando um autosave existe E o canvas está vazio.
•	Texto: "📂 Diagrama recuperado: '{title}' — salvo em {data formatada}. Deseja restaurar?"
•	Dois botões: "Restaurar" (carrega o autosave) e "Descartar" (limpa o autosave e fecha o banner).
•	Design: banner horizontal no topo do canvas, com fundo amber-900/80 no dark mode, animação de entrada suave (slide down).
Exibir um indicador de status de salvamento no header, ao lado do título do diagrama:
•	Estado "Salvando..." com ícone de loading durante o debounce.
•	Estado "Salvo" com ícone de check verde após a persistência.
Critério de aceite: Ao modificar um diagrama, aguardar 2 segundos, fechar e reabrir a aba, o banner de recuperação deve aparecer e ao clicar em "Restaurar" o diagrama completo deve ser recarregado fielmente.

ÉPICO 4 — Algoritmo de Layout: Integração do ELKjs
Prioridade: 🟠 ALTA
Contexto do problema: O Dagre, motor de layout atual, não é suficientemente eficiente para evitar cruzamento de arestas em grafos complexos. Como solução temporária, o prompt da IA instrui a criar cópias duplicadas de nós para diferentes chamadores — o que é um antipadrão semântico que distorce a representação real da arquitetura.
O que deve ser implementado:
Instalar elkjs e criar um novo motor de layout no layoutService.ts baseado no ELK (Eclipse Layout Kernel), algoritmo layered, que minimiza matematicamente os cruzamentos de arestas. O Dagre deve ser mantido como fallback.
Especificação técnica:
Instalar: npm install elkjs
Adicionar ao layoutService.ts a função applyELKLayout:
Copy// Configuração recomendada do ELK para diagramas de microsserviços:
const layoutOptions = {
  'elk.algorithm': 'layered',
  'elk.direction': 'RIGHT',
  'elk.spacing.nodeNode': '80',
  'elk.layered.spacing.nodeNodeBetweenLayers': '120',
  'elk.layered.crossingMinimization.strategy': 'LAYER_SWEEP',
  'elk.layered.nodePlacement.strategy': 'BRANDES_KOEPF',
};
Atualizar o menu de auto-layout para incluir as opções:
•	"ELK — Esquerda para Direita" (novo, padrão recomendado)
•	"ELK — Cima para Baixo" (novo)
•	"Dagre — Horizontal" (existente)
•	"Dagre — Vertical" (existente)
•	"Dagre — Horizontal Invertido" (existente)
•	"Dagre — Vertical Invertido" (existente)
Atualizar o COMMON_SYSTEM_PROMPT no geminiService.ts para remover a Regra 2 (Duplicação de Nós REST), substituindo-a pela instrução: "NÃO duplique nós. Se múltiplos serviços chamam o mesmo serviço, use um único nó com múltiplas arestas apontando para ele. O layout será tratado automaticamente pelo algoritmo ELK."
Critério de aceite: Um diagrama onde três serviços chamam o mesmo serviço central deve ser gerado pela IA com apenas UM nó para o serviço chamado (não três cópias), e o layout ELK deve posicioná-los sem cruzamentos visíveis de arestas.

ÉPICO 6 — Funcionalidade: Colaboração em Tempo Real via Supabase
Prioridade: 🟡 MÉDIA-ALTA
Contexto do problema: A ferramenta é individual. Equipes de arquitetura não podem colaborar em um mesmo diagrama simultaneamente. O @supabase/supabase-js já está instalado no projeto, tornando esta implementação de baixo custo relativo.
O que deve ser implementado:
Implementar colaboração em tempo real onde múltiplos usuários podem visualizar e editar o mesmo diagrama simultaneamente, com sincronização de estado via Supabase Realtime Broadcast.
Especificação técnica:
Autenticação:
•	Implementar login anônimo via Supabase Auth (supabase.auth.signInAnonymously()). Cada usuário anônimo recebe um UUID de sessão.
•	Exibir um avatar/indicador no header com a cor do usuário (cor gerada deterministicamente a partir do UUID).
Modelo de dados (Supabase): Criar a tabela diagrams no Supabase com o schema:
CopyCREATE TABLE diagrams (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL DEFAULT 'Novo Diagrama',
  nodes JSONB NOT NULL DEFAULT '[]',
  edges JSONB NOT NULL DEFAULT '[]',
  owner_id UUID REFERENCES auth.users(id),
  share_token TEXT UNIQUE DEFAULT encode(gen_random_bytes(8), 'hex'),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
Compartilhamento:
•	Adicionar botão "Compartilhar" no header (ícone de link).
•	Ao clicar, salvar o diagrama atual no Supabase e gerar uma URL de compartilhamento no formato: https://[app-url]/diagram/[share_token].
•	Exibir a URL em um modal com botão "Copiar link".
Sincronização em tempo real:
•	Quando dois ou mais usuários estão na mesma URL de diagrama compartilhado, usar Supabase Realtime Broadcast no canal diagram:[share_token] para sincronizar mudanças de estado.
•	Evento nodes_updated: enviado quando nós são adicionados, movidos ou removidos.
•	Evento edges_updated: enviado quando arestas são adicionadas ou removidas.
•	Evento cursor_moved: enviado com a posição do cursor de cada colaborador (opcional, para "cursores ao vivo").
•	Aplicar debounce de 300ms antes de emitir eventos para evitar flood de mensagens.
•	Exibir avatares coloridos dos colaboradores ativos no header.
Critério de aceite: Dois navegadores abrindo a mesma URL de diagrama compartilhado devem ver as mesmas mudanças em tempo real (com latência máxima de 1 segundo). A URL deve ser copiável e abrível por qualquer usuário sem necessidade de login com conta.

ÉPICO 7 — Exportação: Múltiplos Formatos
Prioridade: 🟡 MÉDIA
Contexto do problema: A exportação atual suporta apenas PNG e JSON. Arquitetos precisam de formatos vetoriais (SVG) para documentação de alta qualidade e formatos de texto (Mermaid.js) para uso em READMEs e wikis.
O que deve ser implementado:
Expandir o menu de exportação para suportar quatro formatos:
Exportação para SVG:
•	Usar a biblioteca html-to-image (já instalada) com o método toSvg().
•	O SVG exportado deve ser limpo, sem referências a fontes externas ou URLs absolutas que quebrem ao abrir offline.
Exportação para Mermaid.js:
•	Criar a função exportToMermaid(nodes, edges): string em services/exportService.ts.
•	O diagrama resultante deve usar a sintaxe graph LR do Mermaid.
•	Cada tipo de nó deve ser representado com a forma correta:
•	service → [Nome do Serviço] (retângulo)
•	database → [(Nome do DB)] (cilindro)
•	queue → >Nome da Fila] (forma de fila/flag)
•	external → (Nome Externo) (stadium/pill)
•	Cada aresta deve incluir o protocolo como label: A -->|REST| B.
•	Exibir o código Mermaid gerado em um modal com campo de texto (read-only, copiável) e botão "Copiar código".
Exportação para JSON estruturado (aprimorar o existente):
•	Adicionar campos de metadados ao JSON exportado: exportedAt, version: "2", description (campo opcional que o usuário pode preencher no modal de exportação).
UI do menu de exportação:
•	Substituir o botão único de "Exportar PNG" por um dropdown com as opções: PNG, SVG, Mermaid.js, JSON.
•	Ícones distintos para cada formato.
Critério de aceite: Os quatro formatos de exportação devem funcionar para diagramas com pelo menos 10 nós e 15 arestas. O Mermaid exportado deve ser renderizável em mermaid.live sem erros de sintaxe.

1.	Épico 9 — Limpeza de dependências (base limpa para todo o restante).
2.	Épico 1 — Segurança da API Key (crítico, sem dependências dos outros épicos).

4.	Épico 3 — Auto-save (consome o store do Épico 2).
5.	Épico 4 — Layout ELKjs (independente, mas melhora os resultados do Épico 5).
6.	Épico 5 — Novos protocolos (independente, adiciona sobre a base existente).
7.	Épico 8 — Testes (deve ser implementado após os refactors dos Épicos 2, 3 e 4).
8.	Épico 7 — Exportação múltipla (independente, pode ser paralelo ao 8).
9.	Épico 6 — Colaboração em tempo real (maior complexidade, implementar por último).
10.	Épico 10 — Refinamentos de UX/a11y (finalizações, implementar por último).

Restrições e Regras Gerais para Toda Implementação
•	Não quebrar funcionalidades existentes. Toda feature atual deve continuar funcionando após cada épico.
•	Manter TypeScript estrito. Nenhum any novo deve ser introduzido sem justificativa explícita no comentário. O tsconfig.json deve manter "strict": true.
•	Dark mode obrigatório. Toda nova UI deve ter variante dark mode usando as classes Tailwind existentes (dark:bg-*, dark:text-*). A aplicação usa dark mode como modo padrão.
•	Português como idioma padrão. Todos os textos de UI, mensagens de erro, labels e tooltips devem ser em português brasileiro.
•	Sem quebra de layout em mobile. Embora a aplicação seja primariamente desktop, ela não deve ter erros de layout em viewports abaixo de 768px.
•	Commits atômicos por épico. Cada épico deve ser implementado e validado de forma isolada antes de avançar para o próximo.
