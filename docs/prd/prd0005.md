Épico 1 – Normalização de internalDatabases e internalServices Prioridade: 🔴 CRÍTICA | Arquivos: src/types/diagram.ts, src/schemas/diagramSchema.ts, src/components/ImportJSONModal.tsx
Problema detalhado: Existe uma divergência crítica entre o schema Zod e o runtime. O schema em diagramSchema.ts valida internalDatabases e internalServices como z.array(z.string()). O diagramStore.ts e o ServiceNode.tsx também operam com string[]. A IA, porém, pode retornar esses campos como { id: string, label: string }[] (formato objeto). Se um JSON com o formato objeto for importado, o ServiceNode tentará renderizar o objeto como string, resultando em [object Object] no canvas — corrupção silenciosa de dados.
Solução — dois passos:
Passo 1: Atualizar o schema Zod em src/schemas/diagramSchema.ts para aceitar string ou { id, label } e normalizar tudo para string via .transform():
Copy// src/schemas/diagramSchema.ts
const InternalItemSchema = z.union([
  z.string().min(1),
  z.object({ id: z.string(), label: z.string() }).transform((obj) => obj.label),
  z.object({ label: z.string() }).transform((obj) => obj.label),
]);

// Dentro de NodeSchema.data:
internalDatabases: z.array(InternalItemSchema).optional(),
internalServices:  z.array(InternalItemSchema).optional(),
Passo 2: Nenhuma alteração adicional necessária no ImportJSONModal.tsx — o .transform() do Zod já normaliza os dados antes de chamar onImport.
Testes a adicionar em src/schemas/diagramSchema.test.ts:
•	internalDatabases: [{ id: 'db1', label: 'Oracle Prod' }] → deve retornar ['Oracle Prod']
•	internalDatabases: ['Oracle Prod'] → deve retornar ['Oracle Prod'] inalterado
•	internalDatabases: [{ label: 'Redis Cache' }] sem id → deve retornar ['Redis Cache']
Critérios de aceite:
•	Diagrama gerado pela IA com internalDatabases no formato objeto é importado corretamente, sem [object Object] no canvas
•	Diagrama exportado via JSON e re-importado preserva todos os nomes de bancos e serviços internos
•	npm run test passa

Épico 2 – Remover criação automática de conexão ao spawnar nós Prioridade: 🔴 CRÍTICA | Arquivos: src/store/diagramStore.ts, src/store/diagramStore.test.ts
Problema detalhado: A função addNodesFromSource no diagramStore.ts cria automaticamente arestas entre o nó de origem e os nós recém-gerados, populando newEdges e adicionando-as ao estado junto com os nós. Este comportamento é indesejado: o usuário deve ter total controle sobre quais handles de saída e entrada utilizar para conectar os nós, definindo a direção e o protocolo da conexão manualmente. A criação automática de arestas também ignora o fluxo de inferência de protocolo (inferProtocol) e o modal de seleção (ProtocolSelectorModal) introduzidos nos ciclos anteriores, gerando arestas sem protocolo definido ou com protocolo inferido incorretamente.
Análise do código atual: O trecho problemático em addNodesFromSource está na seção que cria nós externos (não-embedded):
Copy// TRECHO ATUAL — a ser removido
const newEdges: DiagramEdge[] = [];

for (let i = 0; i < count; i++) {
  // ...criação do nó...

  let edgeLabel: string | undefined;
  if (sourceNode.type === 'service' && type === 'queue') edgeLabel = 'produce';
  else if (sourceNode.type === 'queue' && type === 'service') edgeLabel = 'consume';

  newEdges.push({             // ← REMOVER ESTE BLOCO INTEIRO
    id: `edge_${sourceNodeId}_${id}`,
    source: sourceNodeId,
    target: id,
    type: 'editable',
    animated: true,
    style: { strokeWidth: 2 },
    markerEnd: { type: 'arrowclosed' as any },
    data: { waypoints: undefined },
    ...(edgeLabel ? { label: edgeLabel, labelStyle: { fontSize: 11, fontWeight: 600 } } : {}),
  });
}

set((state) => ({
  nodes: [...state.nodes, ...newNodes],
  edges: [...state.edges, ...newEdges],  // ← REMOVER newEdges daqui
}));
Solução: Remover completamente a lógica de criação de newEdges dentro do bloco de nós externos, incluindo a variável newEdges, o newEdges.push(...) e a propagação ...newEdges no set. Os nós devem ser adicionados ao canvas sem nenhuma aresta associada. O posicionamento relativo ao nó de origem (offset x + 250) é mantido para que o nó apareça próximo e facilite a conexão manual pelo usuário:
Copy// CÓDIGO FINAL — addNodesFromSource (bloco de nós externos)
const newNodes: DiagramNode[] = [];

for (let i = 0; i < count; i++) {
  const id = createNodeId();
  newNodes.push({
    id,
    type,
    position: {
      x: sourceNode.position.x + 250,
      y: sourceNode.position.y + i * 120 - ((count - 1) * 60),
    },
    data: {
      label: `${labelMap[type]} ${count > 1 ? i + 1 : ''}`.trim(),
      type,
      subType,
      internalDatabases: [],
      internalServices: [],
    },
  });
}

set((state) => ({
  nodes: [...state.nodes, ...newNodes],
  // Sem alteração nas edges
}));

Copyexport async function renameDiagram(id: string, title: string): Promise<void> {
  const trimmed = title.trim();
  if (!trimmed || trimmed.length > 100) {
    throw new Error('Título inválido: deve ter entre 1 e 100 caracteres');

Épico 5 – Header Vary: Origin e Access-Control-Allow-Methods nas Edge Functions Prioridade: 🟠 ALTA | Arquivo: supabase/functions/generate-diagram/index.ts e demais Edge Functions
Problema: A função getCorsHeaders retorna Access-Control-Allow-Origin dinamicamente por origem, mas não inclui Vary: Origin nem Access-Control-Allow-Methods. Sem Vary: Origin, proxies e CDNs podem cachear a resposta de uma origem e servi-la incorretamente para outras origens.
Solução:
Copy// supabase/functions/generate-diagram/index.ts
function getCorsHeaders(req: Request) {
  const origin = req.headers.get("Origin") || "";
  const allowed = ALLOWED_ORIGINS.length === 0 || ALLOWED_ORIGINS.includes(origin);
  return {
    "Access-Control-Allow-Origin": allowed ? origin : ALLOWED_ORIGINS[0] || "",
    "Access-Control-Allow-Headers": "authorization, x-client-info, apikey, content-type, x-supabase-client-platform, x-supabase-client-platform-version, x-supabase-client-runtime, x-supabase-client-runtime-version",
    "Access-Control-Allow-Methods": "POST, OPTIONS",
    "Vary": "Origin",
  };
}
Aplicar a mesma correção em todas as Edge Functions presentes em supabase/functions/*/index.ts.
Critério de aceite: Respostas da Edge Function incluem os headers Vary: Origin e Access-Control-Allow-Methods: POST, OPTIONS.

Épico 6 – exportToMermaid prioriza data.protocol sobre edge.label Prioridade: 🟡 MÉDIA | Arquivos: src/services/exportService.ts, src/services/exportService.test.ts
Problema: exportToMermaid usa edge.label como rótulo da aresta. Em arestas onde edge.label está ausente mas edge.data.protocol está presente (diagramas legados importados), o Mermaid exporta a aresta sem rótulo, perdendo informação de protocolo.
Solução:
Copy// src/services/exportService.ts — dentro de exportToMermaid
edges.forEach((edge) => {
  const src = idMap.get(edge.source);
  const tgt = idMap.get(edge.target);
  if (!src || !tgt) return;

  const protocol = (edge.data as any)?.protocol as string | undefined;
  const label = protocol || (typeof edge.label === 'string' ? edge.label : '');

  if (label) {
    lines.push(`  ${src} -->|${escapeMermaid(label)}| ${tgt}`);
  } else {
    lines.push(`  ${src} --> ${tgt}`);
  }
});
Testes a adicionar:
•	Aresta com data.protocol = 'gRPC' e label = undefined → exporta N0 -->|gRPC| N1
•	Aresta com data.protocol = 'TCP' e label = 'consume' → exporta com protocolo TCP (protocolo tem prioridade)
•	Aresta sem data.protocol e sem label → exporta N0 --> N1
Critérios de aceite: Arestas sem label mas com data.protocol exportam corretamente para Mermaid. npm run test passa.
