PRD 0037 — Termos de Uso e Política de Privacidade (LGPD)
Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Gap legal — requisito para cobrança e operação no Brasil Plataforma: Lovable Cloud Premissas: Operação principal no Brasil (LGPD). Usuários internacionais possíveis (GDPR aplicável para UE).

## Contexto

Toda plataforma que (1) cobra usuários e (2) coleta dados pessoais é obrigada por lei a disponibilizar Termos de Uso e Política de Privacidade. No Brasil, a LGPD (Lei 13.709/2018) exige:

- Informar quais dados são coletados, como são usados e com quem são compartilhados
- Nomear uma base legal para cada tipo de tratamento (consentimento, execução de contrato, etc.)
- Disponibilizar canal de contato do encarregado (DPO)
- Informar prazo de retenção dos dados
- Permitir que o titular exerça seus direitos (acesso, correção, exclusão)

Sem esses documentos, qualquer cobrança feita via Stripe pode ser contestada como abusiva, e a empresa fica exposta a sanções da ANPD (Autoridade Nacional de Proteção de Dados) de até 2% do faturamento, limitado a R$ 50 milhões por infração.

### Estrutura do documento

O documento deve cobrir no mínimo as seguintes seções. **O texto deve ser redigido por você ou revisado por advogado** — este PRD fornece a estrutura e os pontos obrigatórios, não o texto jurídico final.

```
1. Aceitação dos Termos
   - Ao se cadastrar ou usar o serviço, o usuário aceita estes termos
   - Versão e data de vigência

2. Descrição do Serviço
   - O que é o MicroFlow Architect
   - O que está incluído em cada plano (Free, Pro, Team)
   - Limitações (maxDiagrams, maxNodes, etc.)

3. Cadastro e Conta
   - Requisitos de elegibilidade (maior de 18 anos)
   - Responsabilidade pelo sigilo da senha
   - Proibição de criação de contas falsas

4. Planos e Pagamento
   - Cobrança via Stripe (dados de cartão não são armazenados pelo MicroFlow)
   - Ciclos de cobrança (mensal, trimestral, anual)
   - Política de cancelamento (acesso mantido até o fim do período pago)
   - Política de reembolso (definir: ex. 7 dias corridos após a cobrança)
   - Política de inadimplência (grace period de 7 dias, depois downgrade para Free)

5. Conteúdo do Usuário
   - Os diagramas criados pertencem ao usuário
   - O MicroFlow não acessa, vende nem compartilha diagramas de usuários
   - Dados são criptografados em repouso (mencionar encryption)
   - Direito de excluir conta e todos os dados (funcionalidade já existe em AccountModal)

6. Propriedade Intelectual
   - A plataforma (código, design, marca) pertence ao MicroFlow
   - O usuário não pode fazer engenharia reversa ou redistribuir

7. Limitação de Responsabilidade
   - O serviço é fornecido "como está" (as-is)
   - Limite de responsabilidade ao valor pago nos últimos 12 meses

8. Modificações nos Termos
   - Notificação por e-mail com 30 dias de antecedência para mudanças materiais
   - Versão e data de cada revisão

9. Lei Aplicável
   - Foro: comarca de [cidade], estado de [estado], Brasil
   - Lei aplicável: legislação brasileira

10. Contato
    - E-mail de suporte: [suporte@dominio.com.br]
    - E-mail do DPO: [privacidade@dominio.com.br]
```

## TAREFA T2 — Criar página `/privacy`

**Arquivo:** `src/pages/Privacy.tsx` (novo)
**Route:** Adicionar `<Route path="/privacy" element={<Privacy />} />` em `src/App.tsx`
**Pilar:** Legal / LGPD

### Estrutura do documento (obrigatória pela LGPD Art. 9º)

```
1. Quem somos (Controlador dos dados)
   - Nome da empresa / razão social, CNPJ
   - Endereço
   - E-mail do DPO: [privacidade@dominio.com.br]

2. Dados que coletamos
   - Dados de cadastro: e-mail, senha (hash — nunca a senha em si)
   - Dados de uso: diagramas, nomes de nós, rótulos (criptografados)
   - Dados de pagamento: apenas token Stripe (número do cartão NUNCA armazenado pelo MicroFlow)
   - Dados técnicos: logs de acesso, erros (via Sentry — anonimizados: apenas UUID, sem e-mail)
   - Avatar: imagem enviada opcionalmente pelo usuário

3. Como usamos os dados
   - Fornecer o serviço (base legal: execução de contrato)
   - Enviar e-mails transacionais: confirmação de e-mail, reset de senha (base legal: execução de contrato)
   - Processar pagamentos via Stripe (base legal: execução de contrato)
   - Monitorar erros e melhorar o serviço (base legal: legítimo interesse)
   - NÃO vendemos, alugamos nem compartilhamos dados com terceiros para fins de marketing

4. Compartilhamento com terceiros
   - Supabase (banco de dados e autenticação — servidores AWS us-east-1)
   - Stripe (processamento de pagamentos — PCI DSS certificado)
   - Brevo (envio de e-mails transacionais — RGPD compliant)
   - Sentry (monitoramento de erros — dados anonimizados: UUID apenas)

5. Retenção de dados
   - Dados de conta: enquanto a conta estiver ativa + 90 dias após exclusão
   - Diagramas deletados (soft delete): 90 dias, depois excluídos permanentemente
   - Logs de e-mail: 90 dias
   - Dados de pagamento (Stripe): conforme política da Stripe

6. Seus direitos (LGPD Art. 18)
   - Acesso: solicitar cópia dos seus dados
   - Correção: atualizar dados incorretos (via AccountModal ou e-mail)
   - Exclusão: excluir conta e todos os dados (via AccountModal → "Excluir conta")
   - Portabilidade: exportar seus diagramas como JSON (via Export no canvas)
   - Revogação de consentimento: onde aplicável
   Para exercer: enviar e-mail para [privacidade@dominio.com.br]

7. Cookies e rastreamento
   - Não utilizamos cookies de rastreamento ou publicidade
   - Apenas cookies de sessão (Supabase Auth) e preferências locais (localStorage)

8. Segurança
   - Dados em repouso: criptografados (AES-256 via Web Crypto API)
   - Transmissão: HTTPS/TLS 1.2+
   - Senhas: hashed via bcrypt pelo Supabase Auth

9. Alterações nesta política
   - Notificação por e-mail com 30 dias de antecedência

10. Contato e DPO
    - [privacidade@dominio.com.br]
```

### Critérios de Aceite

- [ ] Rota `/privacy` existe e retorna HTTP 200
- [ ] CNPJ/razão social preenchidos (ou "a definir" explicitamente se pessoa física)
- [ ] Terceiros listados: Supabase, Stripe, Brevo, Sentry
- [ ] Seção de direitos do titular presente com canal de contato
- [ ] Data de vigência visível

## TAREFA T4 — Aceite explícito no checkout Stripe

**Arquivo:** `src/components/BillingModal.tsx` (seção do botão de upgrade)
**Pilar:** Legal

### Por que necessário

Para cobranças recorrentes, o Stripe recomenda (e alguns bancos exigem) que o usuário afirme explicitamente ter lido e aceito os termos antes de fornecer dados de cartão. Isso também reduz disputes (chargebacks) já que o usuário tem ciência do que está contratando.

### O que fazer

Adicionar um checkbox de aceite antes do botão de "Fazer upgrade" no `BillingModal`:

```tsx
// Estado local no BillingModal
const [termsAccepted, setTermsAccepted] = useState(false);

// UI — acima do botão de checkout
<div className="flex items-start gap-2 text-xs text-muted-foreground">
  <input
    type="checkbox"
    id="terms-accept"
    checked={termsAccepted}
    onChange={(e) => setTermsAccepted(e.target.checked)}
    className="mt-0.5 h-3.5 w-3.5 rounded border-border accent-primary shrink-0"
  />
  <label htmlFor="terms-accept">
    {t('billing.acceptTerms.prefix', 'Li e aceito os')}{' '}
    <a href="/terms" target="_blank" className="underline hover:text-foreground">
      {t('footer.terms', 'Termos de Uso')}
    </a>{' '}
    {t('billing.acceptTerms.and', 'e a')}{' '}
    <a href="/privacy" target="_blank" className="underline hover:text-foreground">
      {t('footer.privacy', 'Política de Privacidade')}
    </a>
  </label>
</div>

// Botão de upgrade — desabilitado até checkbox marcado
<Button
  onClick={handleUpgrade}
  disabled={!termsAccepted || upgradeLoading}
>
  {t('billing.upgradePlan')}
</Button>
```

### Critérios de Aceite

- [ ] Checkbox visível antes do botão de upgrade
- [ ] Botão de upgrade desabilitado quando checkbox não está marcado
- [ ] Checkbox reseta para `false` ao fechar e reabrir o modal (`useState` reinicia no unmount)
- [ ] Links abrem em nova aba sem fechar o modal
