Épico 1 – Validação Zod no ImportJSONModal Prioridade: 🔴 CRÍTICA
Problema identificado: O componente src/components/ImportJSONModal.tsx faz apenas JSON.parse e checa se parsed.nodes e parsed.edges existem (verificação de presença superficial). Qualquer JSON com nós malformados (ex.: nó sem id, sem type, aresta sem source) chega ao diagramStore e pode causar crashes silenciosos no React Flow ou corromper o histórico de undo/redo. A dependência zod ^3.25.76 já está instalada no projeto e não é utilizada neste fluxo.
Implementação:
Criar o arquivo src/schemas/diagramSchema.ts com os schemas Zod que refletem exatamente os tipos de src/types/diagram.ts:
Copy// src/schemas/diagramSchema.ts
import { z } from 'zod';

export const NodeTypeSchema = z.enum(['service', 'database', 'queue', 'external']);

export const InternalItemSchema = z.object({
  id: z.string(),
  label: z.string(),
});

export const DiagramNodeDataSchema = z.object({
  label: z.string().min(1),
  type: NodeTypeSchema,
  subType: z.string().optional(),
  internalDatabases: z.array(InternalItemSchema).optional(),
  internalServices: z.array(InternalItemSchema).optional(),
}).passthrough();

export const DiagramNodeSchema = z.object({
  id: z.string().min(1),
  type: NodeTypeSchema,
  position: z.object({ x: z.number(), y: z.number() }),
  data: DiagramNodeDataSchema,
}).passthrough();

export const EdgeProtocolSchema = z.enum([
  'REST', 'gRPC', 'GraphQL', 'WebSocket',
  'Kafka', 'AMQP', 'MQTT', 'HTTPS', 'TCP', 'UDP'
]);

export const DiagramEdgeSchema = z.object({
  id: z.string().min(1),
  source: z.string().min(1),
  target: z.string().min(1),
  type: z.string().optional(),
  animated: z.boolean().optional(),
  label: z.string().optional(),
  data: z.object({
    protocol: EdgeProtocolSchema.optional(),
    waypoints: z.array(z.object({ x: z.number(), y: z.number() })).optional(),
  }).passthrough().optional(),
}).passthrough();

export const ImportDiagramSchema = z.object({
  nodes: z.array(DiagramNodeSchema).min(1, 'O diagrama deve ter ao menos 1 nó'),
  edges: z.array(DiagramEdgeSchema),
  name: z.string().optional(),
  title: z.string().optional(),
});

export type ImportDiagramPayload = z.infer<typeof ImportDiagramSchema>;

Épico 2 – Inferência database→service = TCP Prioridade: 🔴 CRÍTICA
Problema identificado: src/utils/protocolInference.ts não cobre o par database → service (ex.: triggers de banco notificando serviços via CDC — Change Data Capture), caindo no fallback genérico REST, que é semanticamente incorreto. Além disso, os pares external→database, queue→external e external→queue também caem no fallback sem regra explícita.
Implementação:
Substituir o conteúdo de src/utils/protocolInference.ts pela versão expandida:
Copy// src/utils/protocolInference.ts
import type { NodeType, EdgeProtocol } from '@/types/diagram';

type ProtocolMap = Partial<Record<NodeType, Partial<Record<NodeType, EdgeProtocol>>>>;

const PROTOCOL_MAP: ProtocolMap = {
  service: {
    service:  'REST',
    database: 'TCP',
    queue:    'AMQP',
    external: 'HTTPS',
  },
  database: {
    service:  'TCP',
    database: 'TCP',
    queue:    'TCP',
    external: 'TCP',
  },
  queue: {
    service:  'Kafka',
    queue:    'AMQP',
    external: 'AMQP',
    database: 'AMQP',
  },
  external: {
    service:  'REST',
    database: 'HTTPS',
    queue:    'HTTPS',
    external: 'REST',
  },
};

export function inferProtocol(
  sourceType: NodeType,
  targetType: NodeType
): EdgeProtocol {
  return PROTOCOL_MAP[sourceType]?.[targetType] ?? 'REST';
}

Épico 3 – Paginação em loadUserDiagrams Prioridade: 🟠 ALTA

Implementação:
Adicionar suporte a paginação cursor-based na função loadUserDiagrams e adaptar MyDiagrams.tsx para usar scroll infinito:
Copy// src/services/diagramService.ts  (substituir loadUserDiagrams)
const PAGE_SIZE = 12;

export async function loadUserDiagrams(
  userId: string,
  page = 0,
): Promise<{ diagrams: DiagramRecord[]; hasMore: boolean }> {
  const from = page * PAGE_SIZE;
  const to = from + PAGE_SIZE - 1;

  const { data, error, count } = await supabase
    .from('diagrams')
    .select('*', { count: 'exact' })
    .eq('owner_id', userId)
    .order('updated_at', { ascending: false })
    .range(from, to);

  if (error) return { diagrams: [], hasMore: false };

  return {
    diagrams: (data || []) as unknown as DiagramRecord[],
    hasMore: (count ?? 0) > to + 1,
  };
}

import { PROTOCOL_CONFIGS, type EdgeProtocol, type DiagramEdge } from '@/types/diagram';

interface DiagramLegendProps {
  edges: DiagramEdge[];
}

export default function DiagramLegend({ edges }: DiagramLegendProps) {
  // Collect unique protocols from current diagram edges
  const usedProtocols = new Set<EdgeProtocol>(
    edges
      .map((e) => (e.data as any)?.protocol as EdgeProtocol)
      .filter(Boolean)
  );

  // If no protocols are used, show nothing

  const syncProtocols = (Object.entries(PROTOCOL_CONFIGS) as [EdgeProtocol, typeof PROTOCOL_CONFIGS[EdgeProtocol]][])
    .filter(([key, c]) => !c.async && usedProtocols.has(key));

  const asyncProtocols = (Object.entries(PROTOCOL_CONFIGS) as [EdgeProtocol, typeof PROTOCOL_CONFIGS[EdgeProtocol]][])
    .filter(([key, c]) => c.async && usedProtocols.has(key));

Épico 8 – Segurança: shareDiagram verifica propriedade do diagrama Prioridade: 🟠 ALTA
Problema identificado: A função shareDiagram em src/services/diagramService.ts faz update na tabela diagrams usando apenas .eq('id', diagramId), sem verificar se o owner_id corresponde ao usuário autenticado. Isso significa que qualquer usuário autenticado que conheça o id de um diagrama alheio pode gerar um share token para ele, expondo o conteúdo de terceiros. A proteção via Row Level Security (RLS) do Supabase deveria cobrir isso, mas a ausência de uma verificação explícita no cliente é uma falha de defesa em profundidade.
Implementação:
Atualizar shareDiagram para receber e verificar ownerId:
Copy// src/services/diagramService.ts
export async function shareDiagram(
  diagramId: string,
  ownerId: string,
): Promise<string | null> {
  // Fetch only if the diagram belongs to the requesting user
  const { data: existing, error: fetchError } = await supabase
    .from('diagrams')
    .select('share_token, is_shared')
    .eq('id', diagramId)
    .eq('owner_id', ownerId)   // <-- verificação de propriedade
    .single();

  if (fetchError || !existing) return null;

  let shareToken = existing?.share_token;
  if (!shareToken) {
    const randomBytes = crypto.getRandomValues(new Uint8Array(8));
    shareToken = Array.from(randomBytes)
      .map((b) => b.toString(16).padStart(2, '0'))
      .join('');
  }

  const { error: updateError } = await supabase
    .from('diagrams')
    .update({
      share_token: shareToken,
      is_shared: true,
      updated_at: new Date().toISOString(),
    })
    .eq('id', diagramId)
    .eq('owner_id', ownerId);  // <-- verificação de propriedade no update

  if (updateError) return null;
  return `${window.location.origin}/diagram/${shareToken}`;
}

•	Chamadas existentes passam user.id corretamente
