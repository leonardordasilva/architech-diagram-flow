PRD-09 — Segurança, Performance e Qualidade de Código
MicroFlow Architect
Product Requirements Document 
Documento: PRD-09
Data de criação: 01/03/2026 
________________________________________
1. Visão Geral
Este documento especifica os requisitos de melhoria derivados de uma auditoria técnica completa do repositório MicroFlow Architect, conduzida em março de 2026. A auditoria cobriu segurança, performance, qualidade de código, experiência do usuário, acessibilidade e infraestrutura. Os itens aqui descritos são obrigatórios para que a aplicação atinja nível de produção com segurança, estabilidade e manutenibilidade adequados.
O escopo total deste PRD compreende 27 itens de melhoria organizados em 5 categorias, priorizados de forma que os itens críticos de segurança sejam endereçados primeiro, seguidos pelos itens de performance de alto impacto e, por último, os ajustes de qualidade, UX e infraestrutura.
________________________________________
2. Contexto e Motivação
O MicroFlow Architect é uma aplicação SPA (Single Page Application) construída sobre React 18, Vite 5, Supabase e Google Gemini AI, destinada à criação visual de diagramas de arquitetura de microsserviços. A auditoria identificou que, apesar de uma base de código bem estruturada e de boas escolhas arquiteturais, existem falhas críticas que colocam em risco a segurança dos usuários e dos dados, bem como antipadrões que degradam a performance em diagramas de maior complexidade.
________________________________________
3. Objetivos
O PRD-09 tem como objetivos: eliminar vulnerabilidades de segurança que expõem chaves de API e dados de usuários; corrigir antipadrões de performance que causam renderizações excessivas e uso ineficiente de memória; padronizar o uso de TypeScript removendo usos de any nos pontos críticos de entrada de dados; melhorar a experiência do usuário com persistência de preferências e feedbacks de estado adequados; e garantir a acessibilidade mínima da interface para tecnologias assistivas.
________________________________________
4. Itens Fora de Escopo
Os seguintes itens foram explicitamente excluídos deste PRD e não devem ser implementados como parte desta entrega: atualização da lista de modelos de fallback do Gemini (QUA-07) e configuração de pipeline de CI/CD (INF-02). Esses itens serão tratados em PRDs separados.

5. Requisitos por Categoria
5.1 Segurança (Crítico)
5.1.1 SEC-01 — Proteção da API Key do Gemini
Problema: O arquivo geminiService.ts acessa process.env.API_KEY e é compilado para o bundle JavaScript do cliente. Qualquer visitante da aplicação pode inspecionar o código-fonte no navegador, extrair a chave e realizar chamadas diretas à API do Gemini, gerando custos para o proprietário e risco de abuso irrestrito.
Requisito: Toda comunicação com a API do Google Gemini deve ser realizada exclusivamente por uma função serverless (Supabase Edge Function). A aplicação cliente deve chamar apenas o endpoint interno, nunca a API do Gemini diretamente. A chave de API deve residir apenas em variáveis de ambiente do servidor, nunca no bundle do cliente.
Critérios de aceite:
•	Nenhuma referência a API_KEY, GEMINI_API_KEY ou process.env relacionada ao Gemini existe no código cliente após a mudança.
•	A Edge Function generate-diagram e analyze-architecture estão criadas em supabase/functions/.
•	As funções do cliente (generateDiagramFromText, analyzeArchitecture) passam a chamar supabase.functions.invoke().
•	O fluxo de geração e análise de diagramas funciona corretamente de ponta a ponta.
________________________________________
5.1.2 SEC-02 — Autorização na função saveSharedDiagram
Problema: A função saveSharedDiagram em diagramService.ts executa um UPDATE na tabela diagrams filtrando apenas pelo id, sem verificar se o usuário autenticado tem permissão de escrita naquele registro. Qualquer usuário com o UUID do diagrama pode sobrescrever seu conteúdo.
Requisito: A operação de atualização de diagramas compartilhados deve ser protegida por Row Level Security (RLS) no Supabase. As políticas de RLS devem garantir que apenas o owner_id ou usuários listados em uma estrutura de colaboradores autorizados possam executar UPDATE. A Edge Function de compartilhamento deve validar permissões antes de persistir.
Critérios de aceite:
•	A política RLS UPDATE na tabela diagrams está ativa e restringe a operação a auth.uid() = owner_id.
•	O teste manual de tentar salvar um diagrama de outro usuário via Supabase client retorna erro de permissão.
•	O fluxo de colaboração legítimo (usuários com permissão) continua funcionando.
________________________________________
5.1.3 SEC-03 — Validação de JSON importado com Zod
Problema: A função handleImport em DiagramCanvas.tsx carrega diretamente o conteúdo de um arquivo JSON no estado da aplicação sem qualquer validação de estrutura ou conteúdo. Um arquivo malformado ou malicioso pode injetar dados inválidos, corromper o estado do diagrama ou explorar falhas nos componentes de renderização de nós.
Requisito: Todo conteúdo importado via JSON deve ser validado com um schema Zod antes de ser carregado no store. A validação deve verificar tipos, formatos de ID, limites de tamanho de strings e a enumeração válida de NodeType. Erros de validação devem ser exibidos ao usuário com mensagem descritiva.
Critérios de aceite:
•	Existe um schema Zod ImportSchema que valida nodes, edges e name.
•	O campo type de cada nó é validado contra o enum z.enum(['service', 'database', 'queue', 'external']).
•	O campo label é limitado a no máximo 100 caracteres.
•	Importar um arquivo com estrutura inválida exibe uma mensagem de erro específica ao usuário.
•	Importar um arquivo válido continua funcionando normalmente.
________________________________________
5.1.4 SEC-04 — Content Security Policy no index.html
Problema: A aplicação não define nenhuma política de Content Security Policy, expondo-a a ataques de Cross-Site Scripting (XSS), particularmente relevante dado que conteúdo gerado pela IA é renderizado via react-markdown.
Requisito: Uma meta tag Content-Security-Policy deve ser adicionada ao index.html com diretivas que restrinjam fontes de scripts, conexões e imagens aos domínios necessários para o funcionamento da aplicação.
Critérios de aceite:
•	O index.html contém a meta tag CSP com as diretivas default-src, script-src, connect-src, img-src e style-src devidamente configuradas.
•	A aplicação funciona normalmente com a CSP ativa (sem erros de violação no console do navegador para uso legítimo).
•	As conexões com *.supabase.co e generativelanguage.googleapis.com estão explicitamente permitidas em connect-src.
________________________________________
5.1.5 SEC-05 — Row Level Security na função loadDiagramById
Problema: A função loadDiagramById em diagramService.ts faz um SELECT por id sem filtrar por owner_id, permitindo que qualquer usuário autenticado acesse o conteúdo de qualquer diagrama cujo UUID seja conhecido.
Requisito: A política RLS SELECT da tabela diagrams deve garantir que um usuário autenticado só possa ler registros onde owner_id = auth.uid() ou onde exista um share_token válido associado ao registro. A função loadDiagramById não precisa ser alterada no código cliente — a proteção deve ser garantida pela camada de banco de dados.
Critérios de aceite:
•	A política RLS SELECT está ativa na tabela diagrams.
•	Tentar acessar um id de diagrama de outro usuário sem share_token retorna erro ou resultado vazio.
•	O acesso a diagramas próprios e a diagramas compartilhados por token continua funcionando.

5.2 Performance (Alto)
5.2.1 PERF-01 — Substituição do padrão useMemo instável em storeActions
Problema: Em DiagramCanvas.tsx, storeActions é criado via useMemo(() => useDiagramStore.getState(), []). Embora as ações Zustand sejam estáveis, o padrão é semanticamente incorreto e pode causar problemas no Strict Mode do React com double-invocation. Além disso, getTemporalActions é invocada fora de qualquer hook ou componente.
Requisito: Substituir o padrão useMemo de storeActions pela seleção direta de ações via useShallow do Zustand. As ações temporais (undo/redo) devem ser obtidas dentro de callbacks memoizados estáveis.
Critérios de aceite:
•	storeActions é substituído por seletores individuais ou agrupados com useShallow.
•	getTemporalActions não é mais chamada no escopo de módulo; undo e redo são obtidos dentro de useCallback.
•	Nenhum warning de Strict Mode é exibido relacionado a chamadas de hook fora de componentes.
________________________________________
5.2.2 PERF-02 — Otimização do broadcastChanges com comparação de conteúdo
Problema: O useEffect que chama broadcastChanges é disparado a cada mudança de nodes ou edges por referência, incluindo mudanças puramente de posição durante drag. Isso pode gerar broadcasts excessivos para os colaboradores.
Requisito: Antes de disparar o broadcast, comparar o conteúdo serializado com o último broadcast enviado usando um ref. O broadcast só deve ser enviado se houver diferença real de conteúdo.
Critérios de aceite:
•	Um useRef armazena o último estado serializado transmitido.
•	O broadcastChanges só é chamado quando o conteúdo difere do ref.
•	Em testes manuais de drag de nó, o número de broadcasts é reduzido visivelmente no painel de logs do Supabase.
________________________________________
5.2.3 PERF-03 — Substituição de JSON.stringify por comparação de updated_at no realtime
Problema: Em useRealtimeCollab.ts, ao receber atualizações do Postgres Realtime, é feita uma comparação via JSON.stringify de todo o array de nós e arestas para detectar mudanças. Para diagramas grandes, essa operação é O(n) e pode bloquear a thread principal.
Requisito: A comparação deve ser feita prioritariamente usando o campo updated_at do payload do evento. Apenas se updated_at não estiver disponível, utilizar comparação de hash leve ou comparação por contagem de elementos.
Critérios de aceite:
•	O handler do postgres_changes compara updated_at antes de qualquer operação em arrays.
•	A comparação por JSON.stringify de arrays inteiros é removida.
•	O comportamento de sincronização em tempo real permanece correto.
________________________________________
5.2.4 PERF-04 — Carregamento dinâmico (lazy) do elkjs
Problema: A biblioteca elkjs (~1.5MB) é carregada na inicialização da aplicação, aumentando o tempo de carregamento inicial mesmo para usuários que nunca usam o auto-layout ELK.
Requisito: O import do elkjs deve ser convertido para import dinâmico dentro da função getELKLayoutedElements em layoutService.ts, garantindo que a biblioteca só seja carregada quando o usuário solicitar esse tipo de layout.
Critérios de aceite:
•	O arquivo layoutService.ts usa await import('elkjs/lib/elk.bundled.js') em vez de import estático.
•	O bundle inicial da aplicação não inclui o chunk do ELK.
•	O auto-layout ELK funciona corretamente na primeira chamada (carregamento sob demanda) e nas chamadas subsequentes (cache do navegador).
________________________________________
5.2.5 PERF-05 — Correção do closure stale em handleKeyDown
Problema: O callback handleKeyDown em DiagramCanvas.tsx referencia handleSaveToCloud sem incluí-la nas dependências do useCallback, criando um closure stale que sempre usa a versão inicial da função de save.
Requisito: Usar um useRef estável para armazenar a referência atualizada de handleSaveToCloud, eliminando o closure stale sem criar dependências circulares.
Critérios de aceite:
•	handleKeyDown não tem closure stale apontado pelo lint (regra react-hooks/exhaustive-deps).
•	O atalho Ctrl+S executa a versão correta e atualizada de handleSaveToCloud em todos os cenários (novo diagrama, diagrama existente, modo colaborador).
________________________________________
5.2.6 PERF-06 — Compressão do conteúdo salvo no localStorage pelo useAutoSave
Problema: O hook useAutoSave serializa e salva em localStorage todo o estado do diagrama a cada 1,5 segundos. Para diagramas complexos, isso pode facilmente resultar em strings acima de 2MB, bloqueando a thread principal e eventualmente atingindo o limite de 5MB do localStorage.
Requisito: Antes de persistir no localStorage, comprimir o JSON serializado usando a API nativa CompressionStream (disponível em todos os browsers modernos suportados). Implementar tratamento de erro gracioso caso o localStorage esteja cheio, exibindo notificação ao usuário.
Critérios de aceite:
•	O dado salvo em localStorage está em formato comprimido (verificável pelo tamanho reduzido).
•	A leitura com getAutoSave() descomprime o conteúdo corretamente.
•	Quando o localStorage está próximo do limite, o usuário recebe um toast informativo.
•	O comportamento de auto-save permanece transparente para o usuário final.

5.3 Qualidade de Código (Médio)
5.3.1 QUA-01 — Remoção dos campos deprecated de types.ts
Problema: O arquivo types.ts da raiz contém os campos hasDatabase?: boolean e databaseCount?: number marcados como deprecated. Sua presença aumenta a superfície de manutenção e pode causar confusão em novos contribuidores.
Requisito: Remover os campos hasDatabase e databaseCount da interface CustomNodeData. Garantir que nenhum componente ou serviço os referencie. Verificar se há migração necessária em dados persistidos no Supabase.
Critérios de aceite:
•	Os campos não existem em nenhum arquivo de tipos.
•	O TypeScript compila sem erros após a remoção.
•	Nenhum componente ou serviço referencia esses campos.
________________________________________
5.3.2 QUA-02 — Tipagem explícita em diagramService.ts (remoção de any)
Problema: Múltiplas conversões as any e as unknown as DiagramRecord são usadas no diagramService.ts ao interagir com o Supabase, eliminando a proteção do TypeScript nos pontos de entrada de dados externos.
Requisito: Gerar os tipos do Supabase via supabase gen types typescript e utilizá-los em substituição aos any. Onde o tipo gerado não for suficiente, criar interfaces intermediárias explícitas.
Critérios de aceite:
•	Nenhum uso de as any ou as unknown existe em diagramService.ts.
•	O arquivo de tipos Supabase gerado está presente em src/integrations/supabase/types.ts.
•	O TypeScript compila sem erros.
________________________________________
5.3.3 QUA-03 — Remoção do antipadrão getTemporalActions fora de componente
Problema: A função getTemporalActions é definida no escopo de módulo de DiagramCanvas.tsx e chama useDiagramStore.temporal.getState() fora de qualquer hook ou componente React, violando as regras do React e podendo causar problemas no Strict Mode.
Requisito: Mover a obtenção das ações temporais para dentro de useCallback ou de um custom hook dedicado useDiagramHistory(), garantindo que o acesso ao estado temporal seja feito dentro do ciclo de vida do React.
Critérios de aceite:
•	Não existe nenhuma chamada a useDiagramStore ou useDiagramStore.temporal no escopo de módulo (fora de componentes ou hooks).
•	Undo e Redo funcionam corretamente.
•	O lint não reporta violação de regras de hooks.
________________________________________
5.3.4 QUA-04 — Extração da lógica de save para um custom hook useSaveDiagram
Problema: A função handleSaveToCloud em DiagramCanvas.tsx contém toda a lógica de decisão de persistência (se é colaborador, se é diagrama existente, se é cópia de shared), misturada ao componente de UI. Isso viola o princípio de separação de responsabilidades e inviabiliza testes unitários.
Requisito: Criar um custom hook useSaveDiagram em src/hooks/useSaveDiagram.ts que encapsule a lógica de decisão de save, recebendo os parâmetros necessários e expondo a função save e o estado saving.
Critérios de aceite:
•	O arquivo src/hooks/useSaveDiagram.ts existe e exporta o hook.
•	DiagramCanvas.tsx usa const { save, saving } = useSaveDiagram(...) em substituição à função inline.
•	O comportamento de save em todos os cenários (novo, existente, colaborador, cópia) permanece idêntico.
•	O hook é testável de forma isolada (sem dependência do componente de UI).

5.3.5 QUA-05 — Atualização do nome do projeto no package.json
Problema: O package.json contém "name": "vite_react_shadcn_ts", o nome genérico do template de scaffolding, nunca atualizado para o nome real do projeto.

5.4 UX e Acessibilidade (Baixo)
5.4.1 UX-01 — Loading state com indicador visual adequado
Problema: A tela de carregamento em pages/Index.tsx exibe apenas o texto "Carregando..." sem qualquer animação ou indicador de progresso, podendo parecer uma tela quebrada em conexões lentas.
Requisito: Substituir o texto simples por um componente de loading com ícone animado (Loader2 do Lucide, já disponível no projeto), centralizado verticalmente na tela, com texto descritivo abaixo.
Critérios de aceite:
•	A tela de loading exibe o ícone Loader2 animado com animate-spin.
•	O texto "Carregando..." é mantido, posicionado abaixo do ícone.
•	A tela de loading é visualmente consistente com o design system da aplicação (usa classes Tailwind existentes).
________________________________________
5.4.2 UX-02 — revokeObjectURL consistente nos handlers de exportação
Problema: handleExportJSON chama URL.revokeObjectURL() após o download, mas handleExportPNG e handleExportSVG não seguem o mesmo padrão. Embora nesses casos específicos não haja vazamento de memória pois usam dataUrl direto, a inconsistência cria confusão de manutenção.
Requisito: Padronizar todos os handlers de exportação para seguir o mesmo padrão: criar blob/URL, clicar, revogar. Para os casos de dataUrl que não precisam de revogação, adicionar comentário explicativo.
Critérios de aceite:
•	Os três handlers de exportação (PNG, SVG, JSON) têm o mesmo padrão de criação e limpeza de URLs.
•	Onde a revogação não se aplica, existe um comentário explicando o motivo.

5.4.3 UX-03 — Fechamento do context menu ao perder foco
Problema: O menu de contexto (contextMenu) não fecha quando o usuário perde o foco da janela, rola o canvas ou pressiona Escape estando em um nó selecionado, podendo ficar "órfão" visualmente.
Requisito: O menu de contexto deve fechar em todos os seguintes eventos: clique fora do menu, scroll do canvas, tecla Escape e blur da janela (window.blur).
Critérios de aceite:
•	Pressionar Escape fecha o context menu.
•	Rolar o canvas fecha o context menu.
•	Mudar de aba/janela do navegador fecha o context menu ao retornar.
•	O comportamento de fechar ao clicar fora já existente permanece funcionando.
________________________________________
5.4.4 UX-04 — Persistência do Dark Mode entre sessões
Problema: O estado darkMode em DiagramCanvas.tsx é gerenciado localmente sem persistência. A cada recarga da página, o tema volta para o modo claro, ignorando tanto a preferência salva pelo usuário quanto a preferência do sistema operacional.
Requisito: Inicializar o darkMode lendo primeiro o localStorage. Se não houver valor salvo, usar a preferência do sistema (window.matchMedia('(prefers-color-scheme: dark)')). Salvar a preferência no localStorage sempre que o usuário alternar o tema.
Critérios de aceite:
•	Ao recarregar a aplicação, o tema utilizado é o mesmo que estava ativo antes.
•	Em um primeiro acesso (sem preferência salva), o tema segue a preferência do sistema operacional.
•	A chave usada no localStorage para o tema é 'microflow_theme'.
________________________________________
5.4.5 UX-05 — Tratamento de erro para autoLayoutELK assíncrono
Problema: A chamada a storeActions.autoLayoutELK() é uma função async invocada sem await ou .catch() na prop onAutoLayout da Toolbar. Erros de layout (timeout, parsing, ELK indisponível) são silenciosamente ignorados.
Requisito: Envolver a chamada a autoLayoutELK em um bloco try/catch que exiba um toast de erro descritivo ao usuário e que não deixe a UI em estado de loading indefinido.
Critérios de aceite:
•	A chamada a autoLayoutELK está dentro de um bloco try/catch.
•	Em caso de erro, um toast com variant: 'destructive' é exibido com a mensagem "Erro ao aplicar layout automático. Tente novamente.".
•	A UI não fica em estado de loading após o erro.

5.4.6 A11Y-01 — Atributo role no container principal do canvas
Problema: O div raiz em DiagramCanvasInner possui onKeyDown e tabIndex={0}, tornando-o interativamente focável, mas sem um role que permita a leitores de tela entender sua finalidade.
Requisito: Adicionar role="application" ao div raiz do canvas, que é o valor semântico correto para editores visuais interativos segundo as especificações WAI-ARIA. Adicionar também aria-label="Editor de diagramas de arquitetura".
Critérios de aceite:
•	O div raiz de DiagramCanvasInner contém role="application".
•	O div raiz contém aria-label="Editor de diagramas de arquitetura".
•	Nenhum erro de lint de acessibilidade é reportado para esse elemento.

6. Requisitos Não-Funcionais
Os seguintes requisitos não-funcionais se aplicam a todas as implementações descritas neste PRD. Toda mudança de código deve compilar sem erros de TypeScript no modo strict. Nenhuma das mudanças pode introduzir regressões nas funcionalidades existentes de criação, edição, importação, exportação e compartilhamento de diagramas. Mudanças de performance devem ser mensuráveis e verificáveis (ex.: via DevTools ou logs do Supabase). Toda nova função pública deve ter tipagem completa, sem uso de any. Os testes unitários novos criados devem passar com 100% de sucesso no ambiente de CI.

7. Dependências Entre Itens
A implementação deve respeitar a seguinte ordem de dependência: o item SEC-01 deve ser concluído antes de qualquer outro item de segurança, pois a Edge Function criada nele será reutilizada por SEC-02 e SEC-05 como ponto de entrada centralizado. O item QUA-02 (tipagem do Supabase) deve ser concluído antes de QUA-04 (custom hook de save), pois o hook usará os tipos gerados. O item PERF-04 (lazy load do ELK) deve ser concluído antes de PERF-06 (compressão do autoSave), pois ambos impactam o comportamento de carregamento inicial. Os demais itens são independentes entre si.

8. Critérios de Conclusão do PRD
O PRD-09 será considerado concluído quando: todos os 27 itens listados nas seções 5.1 a 5.5 tiverem seus critérios de aceite verificados e aprovados; o comando npm run build executar sem erros ou warnings; o comando npm test executar todos os testes com resultado de aprovação; e nenhuma regressão funcional for identificada nos fluxos principais da aplicação (criação de diagrama, geração por IA, save/load, exportação, colaboração em tempo real).
