## TAREFA T1 — Remover `saving` das dependências do `useCallback`

**Arquivo:** `src/hooks/useSaveDiagram.ts`
**Pilar:** Performance / Correctness
**Risco que resolve:** Re-criação do callback em loop durante execução do save

### Problema

```typescript
// useSaveDiagram.ts — situação atual
const save = useCallback(async () => {
  if (saving) return;    // guard avaliado na chamada
  setSaving(true);       // ← muda estado 'saving'
  // ...
  setSaving(false);      // ← muda estado 'saving' novamente
}, [user, shareToken, saving, onDiagramLimitReached]); // ← 'saving' aqui
```

Ciclo problemático:
1. `save` é criado com `saving = false` nas deps
2. Usuário dispara save → `setSaving(true)` → `saving` muda
3. React recria `save` (deps mudaram)
4. `useEffect` sincroniza `saveRef.current = save` (linha ~121)
5. `setSaving(false)` → `saving` muda → React recria `save` novamente
6. `useEffect` sincroniza de novo

Duas recriações de callback + dois efeitos por save. Com autosave frequente, acumula overhead mensurável.

### O que fazer

```typescript
// DEPOIS — 'saving' removido das deps
const save = useCallback(async () => {
  if (!user) return;
  // Guard avaliado no momento da chamada via ref, não via closure
  if (saving) return;
  // ... restante do código sem alteração
}, [user, shareToken, onDiagramLimitReached]); // sem 'saving'
```

**Justificativa técnica:** `saving` é um valor de estado lido dentro do callback apenas como guard de entrada (`if (saving) return`). Este guard é avaliado no momento da chamada, não no momento da criação — portanto não precisa estar nas deps. A regra `react-hooks/exhaustive-deps` pode alertar, mas o comentário `// eslint-disable-next-line react-hooks/exhaustive-deps` com explicação é aceitável aqui.

**Adicionar comentário explicativo:**

```typescript
const save = useCallback(async () => {
  if (!user) return;
  // NOTE: 'saving' is intentionally omitted from deps — it is read as an
  // entry guard (checked at call time), not used to derive output. Including
  // it would cause the callback to be recreated on every setSaving() call,
  // which triggers the saveRef sync effect on every save cycle. (PRD-0032 T1)
  if (saving) return;
  // ...
}, [user, shareToken, onDiagramLimitReached]);
```

### Critérios de Aceite

- [ ] `saving` não está mais no array de dependências do `useCallback`
- [ ] Comentário explicando a omissão intencional está presente
- [ ] Lint não falha (ou regra está suprimida com justificativa)
- [ ] Comportamento do save é idêntico ao anterior (nenhuma regressão)
- [ ] Teste de snapshot existente continua passando

## TAREFA T2 — Expandir cobertura de testes de `useSaveDiagram`

**Arquivo:** `src/hooks/useSaveDiagram.test.ts`
**Pilar:** Qualidade / Testes
**Risco que resolve:** Edge cases críticos sem cobertura, regressões silenciosas

### Problema

O arquivo `useSaveDiagram.test.ts` foi criado no PRD-0030, mas a cobertura atual (~50% de branches) não contempla cenários críticos: falha parcial de save, guard de cooldown, limites de plano e idempotência.

### O que fazer

Adicionar os seguintes casos de teste ao arquivo existente:

**Grupo 1 — Falha e consistência de estado:**

```typescript
describe('falha de save e consistência de estado', () => {
  it('não atualiza diagramId quando saveDiagram lança exceção', async () => {
    // Arrange: saveDiagram rejeita após ser chamado
    mockSaveDiagram.mockRejectedValueOnce(new Error('Network error'));
    const { result } = renderHook(() => useSaveDiagram({ diagramName: 'test' }));

    // Act
    await act(() => result.current.save());

    // Assert: diagramId permanece undefined
    expect(result.current.diagramId).toBeUndefined();
  });

  it('exibe toast de erro quando save falha', async () => {
    mockSaveDiagram.mockRejectedValueOnce(new Error('DB timeout'));
    const { result } = renderHook(() => useSaveDiagram({ diagramName: 'test' }));
    await act(() => result.current.save());
    expect(mockToast).toHaveBeenCalledWith(
      expect.objectContaining({ variant: 'destructive' })
    );
  });

  it('reseta saving=false mesmo quando save falha', async () => {
    mockSaveDiagram.mockRejectedValueOnce(new Error('fail'));
    const { result } = renderHook(() => useSaveDiagram({ diagramName: 'test' }));
    await act(() => result.current.save());
    expect(result.current.saving).toBe(false);
  });
});
```

**Grupo 2 — Guard de cooldown:**

```typescript
describe('guard de cooldown (SAVE_COOLDOWN_MS)', () => {
  it('ignora segundo save chamado dentro do cooldown', async () => {
    const { result } = renderHook(() => useSaveDiagram({ diagramName: 'test' }));
    await act(() => result.current.save());
    await act(() => result.current.save()); // chamada imediata
    expect(mockSaveDiagram).toHaveBeenCalledTimes(1); // apenas uma execução
  });

  it('permite segundo save após cooldown expirar', async () => {
    vi.useFakeTimers();
    const { result } = renderHook(() => useSaveDiagram({ diagramName: 'test' }));
    await act(() => result.current.save());
    vi.advanceTimersByTime(3000); // avança além do SAVE_COOLDOWN_MS
    await act(() => result.current.save());
    expect(mockSaveDiagram).toHaveBeenCalledTimes(2);
    vi.useRealTimers();
  });
});
```

**Grupo 3 — Limites de plano:**

