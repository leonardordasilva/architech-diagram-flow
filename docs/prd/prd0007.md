Épico 1 — Sincronização de label local nos quatro componentes de nó
Prioridade: 🔴 Crítico | Esforço: Baixo | Arquivos: src/components/nodes/ServiceNode.tsx, src/components/nodes/DatabaseNode.tsx, src/components/nodes/QueueNode.tsx, src/components/nodes/ExternalNode.tsx
Problema raiz: Todos os quatro nós inicializam o estado local label via useState(nodeData.label). Quando o store é alterado externamente — undo/redo, loadDiagram, ou atualização remota via useRealtimeCollab — o componente não re-sincroniza o estado local porque o React não reinicializa useState ao mudar props. O nó exibe um label desatualizado até ser desmontado.
Solução: Adicionar em cada um dos quatro componentes um useEffect que sincroniza o estado local com o valor externo apenas quando o nó não estiver em modo de edição ativa. Essa guarda evita que uma edição em andamento seja interrompida por re-renders externos, mas garante que o display sempre reflita o store quando o campo está fechado.
O padrão a implementar identicamente nos quatro nós é o seguinte:
CopyuseEffect(() => {
  if (!editing) {
    setLabel(nodeData.label);
  }
}, [nodeData.label, editing]);
Adicionalmente, o sub-componente EditableDbItem em ServiceNode.tsx tem o mesmo problema com sua prop value. Aplicar o mesmo padrão dentro de EditableDbItem:
CopyuseEffect(() => {
  if (!editing) setText(value);
}, [value, editing]);
Critérios de aceite: Após undo de renomeação, o nó exibe imediatamente o nome anterior sem necessidade de interação. Após loadDiagram, todos os nós exibem os nomes carregados. Uma edição ativa (input aberto) não é interrompida por atualizações externas. npm run build passa.

Épico 3 — Excluir atualizações remotas do histórico de undo
Prioridade: 🟠 Alto | Esforço: Médio | Arquivo: src/hooks/useRealtimeCollab.ts
Problema raiz: O handler diagram_updated chama store.setNodes e store.setEdges diretamente, o que cria entradas no histórico zundo do usuário local. O Ctrl+Z local pode desfazer mudanças de outro colaborador — comportamento indesejado.
Solução: Envolver as chamadas ao store com pause() / resume() do zundo, que suspende temporariamente a gravação do histórico durante a aplicação das atualizações remotas:
Copy.on('broadcast', { event: 'diagram_updated' }, (payload) => {
  const { nodes, edges } = payload.payload as { nodes: DiagramNode[]; edges: DiagramEdge[] };
  isRemoteUpdate.current = true;

  // Suspende o histórico: atualizações remotas não geram entradas de undo
  const temporal = useDiagramStore.temporal.getState();
  temporal.pause();
  const store = useDiagramStore.getState();
  store.setNodes(nodes);
  store.setEdges(edges);
  temporal.resume();

  setTimeout(() => { isRemoteUpdate.current = false; }, 50);
})
Critérios de aceite: Mudanças recebidas de colaboradores remotos não aparecem no histórico de undo local. O Ctrl+Z desfaz apenas ações do usuário local. npm run build passa.

Épico 6 — Permissão de edição completa para usuários com compartilhamento direto
Prioridade: 🔴 Crítico | Esforço: Alto | Arquivos: src/services/diagramService.ts, src/services/shareService.ts, src/pages/MyDiagrams.tsx, src/pages/SharedDiagram.tsx, src/components/DiagramCanvas.tsx, src/store/diagramStore.ts
Problema raiz (análise completa): Atualmente existem dois fluxos de compartilhamento independentes no sistema. O primeiro é o compartilhamento por link público (share_token + rota /diagram/:shareToken), que abre o SharedDiagram.tsx sem currentDiagramId no store. O segundo é o compartilhamento direto por diagram_shares (tabela que relaciona diagram_id, owner_id e shared_with_id), que é carregado via loadSharedWithMe e exibido na seção "Compartilhados comigo" de MyDiagrams. O problema está no segundo fluxo: ao clicar em um diagrama compartilhado diretamente, handleLoadShared chama store.setCurrentDiagramId(item.diagram_id) — o que é correto para identificação — mas ao clicar em "Salvar na nuvem", a lógica de saveDiagram tenta fazer um update com .eq('owner_id', user.id). Como o usuário logado não é o owner, o Supabase RLS bloqueia o write e o salvamento falha silenciosamente ou lança erro. O usuário com acesso compartilhado não consegue persistir suas edições.
Solução arquitetural: Introduzir o conceito de isCollaborator no store — uma flag booleana que indica que o currentDiagramId pertence a outro usuário, mas que o usuário logado tem permissão de escrita via diagram_shares. O fluxo de save deve ser diferente nesse caso: em vez de fazer update diretamente na tabela diagrams, deve usar uma Edge Function ou uma função de serviço que contorna a RLS verificando a permissão via tabela diagram_shares no lado do servidor.
A solução é implementada em seis partes coordenadas:
Parte A — Novo campo isCollaborator no store (src/store/diagramStore.ts):
Adicionar ao DiagramState e DiagramActions:
Copy// DiagramState
isCollaborator: boolean;

// DiagramActions
setIsCollaborator: (value: boolean) => void;
Inicializar como false e adicionar o setter. O campo não deve ser rastreado pelo zundo (não incluir no partialize).
Parte B — Nova função de serviço para save colaborativo (src/services/diagramService.ts):
Adicionar saveSharedDiagram — uma função que salva edições em um diagrama do qual o usuário é colaborador (não owner). Ela usa o service_role implícito via uma Edge Function dedicada (ver Parte C), ou, como alternativa viável no cliente com RLS correta, une a checagem de permissão via diagram_shares:
Copyexport async function saveSharedDiagram(
  diagramId: string,
  collaboratorId: string,
  title: string,
  nodes: DiagramNode[],
  edges: DiagramEdge[],
): Promise<void> {
  // Verifica se o usuário tem permissão de escrita via diagram_shares
  const { data: share, error: shareError } = await supabase
    .from('diagram_shares')
    .select('id')
    .eq('diagram_id', diagramId)
    .eq('shared_with_id', collaboratorId)
    .maybeSingle();

  if (shareError || !share) {
    throw new Error('Você não tem permissão de edição neste diagrama.');
  }

  // Salva diretamente — o RLS da tabela diagrams precisa permitir
  // update para shared_with_id via policy (ver Parte D)
  const { error } = await supabase
    .from('diagrams')
    .update({
      nodes: nodes as any,
      edges: edges as any,
      updated_at: new Date().toISOString(),
    })
    .eq('id', diagramId);

  if (error) throw error;
}

Parte C — Nova política RLS no Supabase para colaboradores (supabase/migrations/):
Criar o arquivo supabase/migrations/YYYYMMDD_collab_write_policy.sql com a política que permite update na tabela diagrams para usuários que possuem um registro em diagram_shares com shared_with_id = auth.uid():
Copy-- Política: colaboradores podem atualizar nós e arestas de diagramas compartilhados com eles
CREATE POLICY "Collaborators can update shared diagrams"
ON diagrams
FOR UPDATE
USING (
  EXISTS (
    SELECT 1 FROM diagram_shares
    WHERE diagram_shares.diagram_id = diagrams.id
      AND diagram_shares.shared_with_id = auth.uid()
  )
)
WITH CHECK (
  EXISTS (
    SELECT 1 FROM diagram_shares
    WHERE diagram_shares.diagram_id = diagrams.id
      AND diagram_shares.shared_with_id = auth.uid()
  )
);
Importante: A política usa USING e WITH CHECK separados para garantir que o colaborador só possa atualizar (não inserir ou deletar) diagramas que já existem e para os quais ele tem permissão. O title não é incluído no update de saveSharedDiagram (apenas nodes, edges e updated_at) para preservar o nome original do owner — renomeação permanece exclusiva do owner.

Épico 12 — Indicador aria-live no contador de caracteres do Toolbar
Prioridade: 🟢 Baixo | Esforço: Muito Baixo | Arquivo: src/components/Toolbar.tsx
Problema raiz: O contador {localName.length}/100 que aparece quando o nome ultrapassa 80 caracteres é um <span> sem aria-live. Leitores de tela não são notificados de sua aparição ou atualização.
Solução: Substituir o span condicional por uma <div> com aria-live="polite" e aria-atomic="true" sempre presente no DOM (para que o leitor de tela registre a região), mas vazia quando o contador não deve ser exibido:
Copy<div
  aria-live="polite"
  aria-atomic="true"
  className="absolute -bottom-4 right-0 text-[10px] text-muted-foreground"
>
  {localName.length > 80 ? `${localName.length}/100` : ''}
</div>
Critérios de aceite: A região aria-live está no DOM mesmo quando vazia. Ao digitar mais de 80 caracteres, o leitor de tela anuncia o contador. npm run build passa.
