## TAREFA T1 — Remover `VITE_ADMIN_EMAIL` dos arquivos de configuração

**Arquivos:** `.env`, `.env.defaults`, `.env.example`, `vite.config.ts` (e quaisquer outros)
**Pilar:** Segurança / Limpeza
**Risco que resolve:** Variável obsoleta que pode causar confusão e representa exposição desnecessária no bundle

### Problema

Após o PRD-0029 T1, a validação de admin foi migrada para o campo `is_admin` no banco de dados via hook `useIsAdmin`. A variável `VITE_ADMIN_EMAIL` não é mais lida por nenhum componente ou hook de produção, mas ainda permanece definida nos arquivos de ambiente.

Impactos de manter a variável:
- Qualquer `VITE_*` é embutida no bundle JavaScript pelo Vite durante o build, independente de ser usada
- Desenvolvedores novos podem se confundir sobre qual mecanismo de admin está ativo
- Aumenta superfície de configuração sem necessidade

### O que fazer

**Passo 1 — Buscar e remover em todos os arquivos `.env*`:**

Procurar em todos os arquivos da raiz do projeto por `VITE_ADMIN_EMAIL` e remover a linha:

```bash

.env.local
.env.defaults
.env.example
.env.production
.env.development
```

Linha a remover (variação pode ter comentário ou aspas):
```
VITE_ADMIN_EMAIL=admin@exemplo.com
# VITE_ADMIN_EMAIL=
```

**Passo 2 — Remover de `vite.config.ts` (se referenciada):**

Verificar se `vite.config.ts` carrega ou define `VITE_ADMIN_EMAIL` e remover:

```typescript
// ANTES (se existir em vite.config.ts)
'import.meta.env.VITE_ADMIN_EMAIL': JSON.stringify(env.VITE_ADMIN_EMAIL),

// DEPOIS
// linha removida completamente
```

**Passo 3 — Remover de `AdminGuard.tsx` (se ainda houver referência residual):**

```typescript

// Remover completamente qualquer linha com VITE_ADMIN_EMAIL
```

**Passo 4 — Atualizar documentação interna (se houver):**

Se `docs/admin-area-design.md` ou `docs/admin-implementation-plan.md` ainda mencionam `VITE_ADMIN_EMAIL` como mecanismo ativo, adicionar nota indicando que foi substituído por `is_admin` no banco (PRD-0029).

### Critérios de Aceite

## TAREFA T2 — Simplificar cast de tipo em `useIsAdmin`

**Arquivo:** `src/hooks/useIsAdmin.ts`
**Pilar:** Manutenibilidade
**Risco que resolve:** Cast desnecessário que mascara o tipo real e pode esconder erros futuros

### Problema

```typescript
// useIsAdmin.ts — situação atual (linha ~18)
const { data, error } = await supabase
  .from('profiles')
  .select('is_admin')
  .eq('id', user.id)
  .single();
if (error) return false;
return (data as Record<string, unknown>)?.is_admin === true; // ← cast desnecessário
```

O cast `as Record<string, unknown>` foi provavelmente adicionado para contornar um erro de tipagem do Supabase client gerado antes da coluna `is_admin` existir no schema. Com a migration `20260409_prd0029_is_admin_on_profiles.sql` aplicada e os tipos Supabase regenerados, o client já conhece `is_admin` como `boolean`.

**Risco do cast:** Ao forçar o tipo para `Record<string, unknown>`, perdemos a tipagem de `is_admin` como `boolean`. Se a coluna for renomeada ou removida futuramente, o TypeScript não detecta o erro — o cast silencia o compilador.

### O que fazer

**Passo 1 — Verificar se os tipos Supabase foram regenerados após a migration:**

Os tipos gerados ficam em `src/integrations/supabase/types.ts`. Verificar se `profiles` inclui `is_admin`:

```typescript
// src/integrations/supabase/types.ts — deve conter:
profiles: {
  Row: {
    id: string
    email: string
    plan: string
    avatar_url: string | null
    is_admin: boolean       // ← deve existir após migration
    suspended_at: string | null
    created_at: string
  }
  // ...
}
```

**Passo 2 — Se os tipos estiverem atualizados, remover o cast:**

```typescript
// DEPOIS — useIsAdmin.ts
const { data, error } = await supabase
  .from('profiles')
  .select('is_admin')
  .eq('id', user.id)
  .single();
if (error) return false;
return data?.is_admin === true; // TypeScript já sabe que is_admin é boolean
```

**Passo 3 — Se os tipos NÃO estiverem atualizados, regenerar via Supabase CLI:**

```bash
# Regenerar tipos do Supabase (requer Supabase CLI instalado)
npx supabase gen types typescript --project-id <PROJECT_ID> > src/integrations/supabase/types.ts
```

Após regeneração, os tipos refletirão as colunas `is_admin` e `suspended_at` adicionadas pelas migrations do PRD-0029.

**Passo 4 — Verificar também `suspended_at` nos tipos:**

Aproveitar a regeneração para confirmar que `suspended_at` também está tipado:

```typescript
// src/integrations/supabase/types.ts — deve conter após migration:
suspended_at: string | null // timestamptz é mapeado para string no client TS
```

Se o `AuthProvider` verifica `suspended_at`, a tipagem correta é importante para evitar erros silenciosos.

### Critérios de Aceite

- [ ] `src/integrations/supabase/types.ts` contém `is_admin: boolean` na row de `profiles`
- [ ] `src/integrations/supabase/types.ts` contém `suspended_at: string | null` na row de `profiles`
- [ ] `useIsAdmin.ts` usa `data?.is_admin` sem cast para `Record<string, unknown>`
- [ ] `npx tsc --noEmit` passa sem erros relacionados a `is_admin` ou `suspended_at`
- [ ] Comportamento de `useIsAdmin` é idêntico ao anterior
