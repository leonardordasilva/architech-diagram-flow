Contexto e Motivação:
A auditoria pós-v7.0 identificou 5 problemas de segurança onde lógica de autorização que deveria residir no backend está sendo executada no JavaScript do frontend, tornando-a bypassável por qualquer usuário com DevTools. Os três primeiros são críticos — saveSharedDiagram sem RLS, deleteDiagram sem owner_id e renameDiagram sem owner_id — e permitem que um usuário mal-intencionado modifique, delete ou renomeie diagramas de outros usuários. O quarto problema (shareDiagram com geração de token no frontend) cria um race condition e acopla a URL ao domínio do cliente. O quinto (listUsersPaginated) expõe todos os e-mails do sistema, violando privacidade (LGPD/GDPR). O v8.0 resolve todos os 5 problemas através de uma combinação de RLS policies no Supabase e uma nova Edge Function, sem alterar nenhuma funcionalidade visível ao usuário.
Invariantes (não mudar): UI em português brasileiro, TypeScript estrito sem any injustificado, dark mode funcional, npm run build e npm run test passando após cada épico.

Épico 1 — 🔴 Crítico: RLS Policies para DELETE e UPDATE na tabela diagrams
Prioridade: Crítica | Esforço: Baixo | Risco de regressão: Médio
Arquivo principal: supabase/migrations/[timestamp]_rls_diagrams_security.sql (novo arquivo)
Problema raiz: A tabela diagrams não possui Row Level Security ativa ou, se possui, não tem políticas que impeçam operações não autorizadas. Sem RLS, qualquer chamada autenticada ao Supabase pode DELETE ou UPDATE qualquer linha da tabela independentemente das verificações JavaScript do frontend.
Solução — criar migration SQL com 4 políticas RLS:
Copy-- supabase/migrations/[timestamp]_rls_diagrams_security.sql

-- 1. Garantir que RLS está habilitado
ALTER TABLE diagrams ENABLE ROW LEVEL SECURITY;

-- 2. SELECT: owner vê seus próprios; colaborador vê os compartilhados com ele
CREATE POLICY "diagrams_select_policy"
ON diagrams FOR SELECT
TO authenticated
USING (
  owner_id = auth.uid()
  OR EXISTS (
    SELECT 1 FROM diagram_shares
    WHERE diagram_shares.diagram_id = diagrams.id
      AND diagram_shares.shared_with_id = auth.uid()
  )
  OR is_shared = true  -- leitura pública de diagramas com share_token
);

-- 3. INSERT: qualquer autenticado pode criar, mas owner_id DEVE ser seu próprio uid
CREATE POLICY "diagrams_insert_policy"
ON diagrams FOR INSERT
TO authenticated
WITH CHECK (owner_id = auth.uid());

-- 4. UPDATE: owner atualiza tudo; colaborador atualiza apenas nodes/edges
--    (o banco rejeita qualquer tentativa de alterar title/owner_id por colaborador
--     pois a função saveSharedDiagram no frontend já omite esses campos,
--     mas a RLS garante a proteção independente do cliente)
CREATE POLICY "diagrams_update_policy"
ON diagrams FOR UPDATE
TO authenticated
USING (
  owner_id = auth.uid()
  OR EXISTS (
    SELECT 1 FROM diagram_shares
    WHERE diagram_shares.diagram_id = diagrams.id
      AND diagram_shares.shared_with_id = auth.uid()
  )
)
WITH CHECK (
  -- Owner pode alterar qualquer campo
  owner_id = auth.uid()
  OR (
    -- Colaborador: owner_id e title não podem mudar
    EXISTS (
      SELECT 1 FROM diagram_shares
      WHERE diagram_shares.diagram_id = diagrams.id
        AND diagram_shares.shared_with_id = auth.uid()
    )
    AND owner_id = (SELECT owner_id FROM diagrams d2 WHERE d2.id = diagrams.id)
    AND title    = (SELECT title    FROM diagrams d2 WHERE d2.id = diagrams.id)
  )
);

-- 5. DELETE: apenas o owner pode deletar
CREATE POLICY "diagrams_delete_policy"
ON diagrams FOR DELETE
TO authenticated
USING (owner_id = auth.uid());
Copy
Verificação: após aplicar a migration, um usuário autenticado que tente DELETE em um diagrama alheio deve receber erro 42501 (insufficient privilege) do Postgres. O Supabase client retorna { error: { code: 'PGRST301' } }.
Critérios de aceite:
•	Migration executada sem erros no Supabase Dashboard
•	npm run build passa (arquivo SQL não afeta compilação TypeScript)
•	Comportamento normal do owner (criar, salvar, deletar, renomear) continua funcionando
•	Colaborador consegue salvar nodes/edges mas não consegue alterar title ou owner_id

Épico 4 — 🔴 Crítico: Simplificar saveSharedDiagram — remover verificação JavaScript, depender da RLS
Prioridade: Crítica | Esforço: Baixo | Risco de regressão: Baixo
Arquivo: src/services/diagramService.ts
Problema raiz: a função atualmente executa uma SELECT de verificação de permissão no frontend antes de executar o UPDATE. Com a RLS do Épico 1 ativa, essa verificação JavaScript é redundante como mecanismo de segurança e é um falso gate — pode ser bypassada no DevTools. A RLS do banco é o único gate confiável.
Solução — src/services/diagramService.ts:
Copy// ANTES (verificação JavaScript bypassável)
export async function saveSharedDiagram(
  diagramId: string,
  collaboratorId: string,
  title: string,
  nodes: DiagramNode[],
  edges: DiagramEdge[],
): Promise<void> {
  const { data: share, error: shareError } = await supabase
    .from('diagram_shares')
    .select('id')
    .eq('diagram_id', diagramId)
    .eq('shared_with_id', collaboratorId)
    .maybeSingle();

  if (shareError || !share) {
    throw new Error('Você não tem permissão de edição neste diagrama.');

  const { error } = await supabase
    .from('diagrams')
    .update({
      nodes: nodes as any,
      edges: edges as any,
      updated_at: new Date().toISOString(),
    })
    .eq('id', diagramId);

  if (error) throw error;
}

Épico 5 — 🟠 Alto: Mover geração de share_token para Edge Function share-diagram
Prioridade: Alta | Esforço: Médio | Risco de regressão: Médio
Arquivos: supabase/functions/share-diagram/index.ts (novo), src/services/diagramService.ts, src/components/ShareDiagramModal.tsx (verificar uso), src/pages/MyDiagrams.tsx (verificar uso)
Problema raiz: a geração de share_token acontece no cliente com crypto.getRandomValues e a URL final usa window.location.origin, acoplando a geração de tokens ao domínio do navegador e criando um race condition teórico entre dois clientes que gerem tokens simultaneamente antes de persistir.
Solução — criar supabase/functions/share-diagram/index.ts:
Copyimport { serve } from "https://deno.land/std@0.168.0/http/server.ts";
import { createClient } from "https://esm.sh/@supabase/supabase-js@2";

const ALLOWED_ORIGINS = (Deno.env.get("ALLOWED_ORIGINS") || "")
  .split(",").map((o) => o.trim()).filter(Boolean);

function getCorsHeaders(req: Request) {
  const origin = req.headers.get("Origin") || "";
  const allowed = ALLOWED_ORIGINS.length === 0 || ALLOWED_ORIGINS.includes(origin);
  return {
    "Access-Control-Allow-Origin": allowed ? origin : ALLOWED_ORIGINS[0] || "",
    "Access-Control-Allow-Headers":
      "authorization, x-client-info, apikey, content-type",
    "Access-Control-Allow-Methods": "POST, OPTIONS",
    "Vary": "Origin",
  };
}

serve(async (req) => {
  const corsHeaders = getCorsHeaders(req);
  if (req.method === "OPTIONS") {
    return new Response(null, { headers: corsHeaders });
  }

  try {
    // 1. Autenticar
    const authHeader = req.headers.get("Authorization");
    if (!authHeader?.startsWith("Bearer ")) {
      return new Response(JSON.stringify({ error: "Unauthorized" }), {
        status: 401,
        headers: { ...corsHeaders, "Content-Type": "application/json" },
      });
    }

    // Usar service role para operações privilegiadas de share_token
    const supabaseAdmin = createClient(
      Deno.env.get("SUPABASE_URL") ?? "",
      Deno.env.get("SUPABASE_SERVICE_ROLE_KEY") ?? "",
    );

    // Verificar token do usuário (anon client)
    const supabaseUser = createClient(
      Deno.env.get("SUPABASE_URL") ?? "",
      Deno.env.get("SUPABASE_ANON_KEY") ?? "",
      { global: { headers: { Authorization: authHeader } } },
    );

    const { data: { user }, error: userError } = await supabaseUser.auth.getUser();
    if (userError || !user) {
      return new Response(JSON.stringify({ error: "Invalid token" }), {
        status: 401,
        headers: { ...corsHeaders, "Content-Type": "application/json" },
      });
    }

    // 2. Extrair e validar diagramId
    const body = await req.json();
    const { diagramId } = body;
    if (!diagramId || typeof diagramId !== "string") {
      return new Response(JSON.stringify({ error: "diagramId required" }), {
        status: 400,
        headers: { ...corsHeaders, "Content-Type": "application/json" },
      });
    }

    // 3. Verificar que o usuário autenticado é o owner do diagrama
    const { data: diagram, error: fetchError } = await supabaseAdmin
      .from("diagrams")
      .select("id, share_token, owner_id")
      .eq("id", diagramId)
      .eq("owner_id", user.id)   // <-- só o owner pode gerar share_token
      .single();

    if (fetchError || !diagram) {
      return new Response(
        JSON.stringify({ error: "Diagrama não encontrado ou sem permissão" }),
        { status: 403, headers: { ...corsHeaders, "Content-Type": "application/json" } },
      );
    }

    // 4. Reutilizar token existente ou gerar novo (server-side, sem race condition)
    let shareToken = diagram.share_token;
    if (!shareToken) {
      // crypto.randomUUID() disponível no Deno runtime — 128 bits de entropia
      shareToken = crypto.randomUUID().replace(/-/g, "");
    }

    // 5. Persistir com service role (atomicamente)
    const { error: updateError } = await supabaseAdmin
      .from("diagrams")
      .update({
        share_token: shareToken,
        is_shared: true,
        updated_at: new Date().toISOString(),
      })
      .eq("id", diagramId)
      .eq("owner_id", user.id);

    if (updateError) throw updateError;

    // 6. Construir URL a partir de variável de ambiente (não de window.location)
    const siteUrl = Deno.env.get("SITE_URL") || "https://your-app.lovable.app";
    const shareUrl = `${siteUrl}/diagram/${shareToken}`;

    return new Response(JSON.stringify({ shareUrl, shareToken }), {
      headers: { ...corsHeaders, "Content-Type": "application/json" },
    });
  } catch (e) {
    console.error("share-diagram error:", e);
    return new Response(
      JSON.stringify({ error: "Erro ao gerar link de compartilhamento." }),
      { status: 500, headers: { ...corsHeaders, "Content-Type": "application/json" } },
    );
  }
});

Ordem de Implementação Recomendada:
A sequência abaixo minimiza risco de regressão e garante que cada épico seja testável de forma independente antes de avançar.
O primeiro passo é o Épico 1 (RLS na tabela diagrams), pois é a fundação de segurança que os demais épicos dependem. Sem a RLS ativa, os épicos 2 e 3 são apenas "defesa em profundidade" — com a RLS ativa, eles passam a ser a segunda camada correta. O segundo passo são os Épicos 2 e 3 em paralelo, pois ambos são mudanças de assinatura de função sem dependência entre si. O terceiro passo é o Épico 4, que simplifica saveSharedDiagram agora que a RLS do Épico 1 está ativa — sem a RLS, a simplificação seria insegura. O quarto passo é o Épico 7 (RLS em diagram_shares), que pode ser aplicado a qualquer momento sem depender dos anteriores. O quinto passo é o Épico 5 (Edge Function share-diagram), que é independente dos anteriores mas requer deploy de infraestrutura. O sexto passo é o Épico 6 (remover listUsersPaginated), que é uma limpeza de código sem dependências. O sétimo e último passo é o Épico 8 (testes), executado depois que todos os contratos de assinatura estão estáveis.
