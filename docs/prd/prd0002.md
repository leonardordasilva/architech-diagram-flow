ÉPICO A — Segurança: Corrigir CORS Wildcard nas Edge Functions
Prioridade: 🔴 CRÍTICA — Implementar antes de qualquer outro item.
Contexto do problema: A Edge Function supabase/functions/generate-diagram/index.ts usa "Access-Control-Allow-Origin": "*" (wildcard), permitindo que qualquer domínio na internet invoque a função com um token de autenticação válido. O PRD v1.0 especificava que as origens deveriam ser restritas à URL da aplicação.
O que deve ser implementado:
Substituir o objeto corsHeaders fixo em todas as Edge Functions existentes (generate-diagram e qualquer outra que exista em supabase/functions/) pelo padrão de origin dinâmica abaixo:
Copy// Substitua o objeto corsHeaders estático pelo seguinte padrão:
function getCorsHeaders(req: Request): Record<string, string> {
  const origin = req.headers.get("origin") ?? "";
  const allowedOrigins = (Deno.env.get("ALLOWED_ORIGINS") ?? "").split(",").map(s => s.trim());
  
  const isAllowed = allowedOrigins.some(allowed =>
    allowed === "*" || origin === allowed || origin.endsWith(`.${allowed}`)
  );

  return {
    "Access-Control-Allow-Origin": isAllowed ? origin : allowedOrigins[0] ?? "",
    "Access-Control-Allow-Headers": "authorization, x-client-info, apikey, content-type",
    "Access-Control-Allow-Methods": "POST, OPTIONS",
    "Vary": "Origin",
  };
}
A variável de ambiente ALLOWED_ORIGINS deve ser configurada no painel do Supabase com o valor https://[dominio-do-app].lovable.app (ou múltiplos domínios separados por vírgula para ambientes de staging/preview). Adicionar ao repositório o arquivo supabase/functions/.env.example com o conteúdo:
ALLOWED_ORIGINS=https://seu-app.lovable.app,https://staging.seu-app.lovable.app
LOVABLE_API_KEY=seu-segredo-aqui
Atualizar todas as invocações de corsHeaders dentro das funções para usar getCorsHeaders(req) em vez do objeto estático, inclusive no handler de OPTIONS preflight.
Critério de aceite: Ao chamar a Edge Function a partir de um domínio diferente do configurado em ALLOWED_ORIGINS com um token válido, a resposta deve retornar Access-Control-Allow-Origin com valor vazio (bloqueando o browser de usar a resposta). Requisições do domínio autorizado devem continuar funcionando normalmente.

ÉPICO D — Resiliência: Fallback Multi-Modelo na Edge Function
Prioridade: 🟠 ALTA
Contexto do problema: A Edge Function generate-diagram usa um único modelo fixo (google/gemini-3-flash-preview). O sistema de fallback multi-modelo que existia no geminiService.ts original foi perdido na migração para o backend serverless. Em caso de rate limit (429) ou indisponibilidade (503), a requisição falha diretamente sem retentativas.
O que deve ser implementado:
Refatorar a lógica de chamada à IA dentro de supabase/functions/generate-diagram/index.ts para implementar retentativa em cascata pelos modelos disponíveis no gateway do Lovable, com delay inteligente entre tentativas.
Copy// Adicionar ao topo da Edge Function:
const MODELS = [
  "google/gemini-2.0-flash",
  "google/gemini-3-flash-preview",
  "google/gemini-1.5-flash-8b",
];

const sleep = (ms: number) => new Promise((resolve) => setTimeout(resolve, ms));

async function callWithFallback(
  apiKey: string,
  messages: Array<{ role: string; content: string }>,
): Promise<string> {
  let lastError: unknown;

  for (const model of MODELS) {
    try {
      const response = await fetch("https://ai.gateway.lovable.dev/v1/chat/completions", {
        method: "POST",
        headers: {
          Authorization: `Bearer ${apiKey}`,
          "Content-Type": "application/json",
        },
        body: JSON.stringify({ model, messages }),
      });

      if (response.status === 429 || response.status === 503) {
        lastError = new Error(`Model ${model} returned ${response.status}`);
        await sleep(1500);
        continue;
      }

      if (response.status === 404 || response.status === 402) {
        lastError = new Error(`Model ${model} not available (${response.status})`);
        continue; // Sem delay — modelo não existe ou requer crédito
      }

      if (!response.ok) {
        throw new Error(`AI gateway error: ${response.status}`);
      }

      const result = await response.json();
      return result.choices?.[0]?.message?.content ?? "";
    } catch (err) {
      lastError = err;
      continue;
    }
  }

  throw lastError ?? new Error("All models failed");
}
Copy
A função callWithFallback deve substituir a chamada direta ao fetch que existe atualmente na Edge Function. O retorno é a string de conteúdo da IA, que segue para o parser JSON existente.
Aplicar a mesma lógica de fallback a qualquer outra Edge Function que faça chamadas à IA (ex: analyze-architecture, se existir).
Critério de aceite: Ao forçar o primeiro modelo a retornar 429 (via mock ou configurando um modelo inválido como primeiro), a Edge Function deve automaticamente tentar o próximo modelo e retornar a resposta com sucesso. O log do Supabase deve mostrar a tentativa e a troca de modelo.

export interface CollaboratorPresence {
  userId: string;
  color: string;    // cor determinística baseada no userId
  initial: string;  // primeira letra do email ou "?" para anônimos
}

// Função de cor determinística (sem dependência externa):
function getUserColor(userId: string): string {
  const colors = ['#3B82F6','#22C55E','#F59E0B','#EC4899','#A855F7','#06B6D4','#F97316'];
  let hash = 0;
  for (let i = 0; i < userId.length; i++) {
    hash = userId.charCodeAt(i) + ((hash << 5) - hash);
  }
  return colors[Math.abs(hash) % colors.length];
}
