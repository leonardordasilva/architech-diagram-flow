PRD 0038 — E-mails Transacionais com Brevo
Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Gap de produção — infraestrutura de e-mail dependente do Lovable Platform Plataforma: Lovable Cloud / Supabase Edge Functions / Brevo (ex-Sendinblue)

## Contexto

A arquitetura atual de e-mail funciona da seguinte forma:

```
Supabase Auth → auth-email-hook (edge fn)
                  ↓ renderiza React Email template
                  ↓ enfileira em pgmq (auth_emails)
                    ↓
              process-email-queue (edge fn, CRON)
                  ↓ chama sendLovableEmail() ← @lovable.dev/email-js
                      ↓ envia via Lovable Platform API
```

O problema é o último passo: `sendLovableEmail()` usa a infraestrutura de e-mail proprietária da **plataforma Lovable**. Essa dependência:

1. **Não é controlada:** se o projeto for movido para fora do Lovable ou a Lovable mudar sua política, os e-mails param de funcionar
2. **Não tem SLA explícito** para projetos em produção fora do plano Lovable
3. **Não tem painel de analytics:** não é possível ver taxa de entrega, bounces, spam reports
4. **Viola controle LGPD:** não sabemos onde os e-mails são armazenados ou processados pela Lovable

A solução é substituir `sendLovableEmail()` por chamadas diretas à **API do Brevo** (ex-Sendinblue), que oferece:
- API REST simples e bem documentada
- Domínio remetente verificado (SPF/DKIM)
- Painel de analytics (entregas, aberturas, bounces)
- RGPD/LGPD compliant (servidores na Europa)
- Gratuito até 300 e-mails/dia

A renderização dos templates (React Email → HTML) e a fila pgmq **permanecem inalterados** — apenas o passo de envio muda.

## TAREFA T1 — Criar função utilitária `sendBrevoEmail`

**Arquivo:** `supabase/functions/_shared/brevo.ts` (novo)
**Pilar:** Correctness

### O que fazer

Criar o arquivo compartilhado que encapsula a chamada à API transacional do Brevo:

```typescript
// supabase/functions/_shared/brevo.ts

export interface BrevoEmailPayload {
  to: string;
  from: string;        // ex: "MicroFlow <noreply@microflow.com.br>"
  subject: string;
  html: string;
  text?: string;
  messageId?: string;  // para idempotência (header Message-ID)
}

export interface BrevoSendResult {
  messageId: string;
}

/**
 * Envia um e-mail via Brevo Transactional Email API v3.
 * Documentação: https://developers.brevo.com/reference/sendtransacemail
 *
 * Lança erro se a API retornar status >= 400.
 */
export async function sendBrevoEmail(
  payload: BrevoEmailPayload,
  apiKey: string,
): Promise<BrevoSendResult> {
  // Parsear "Nome <email>" do campo from
  const fromMatch = payload.from.match(/^(.*?)\s*<(.+)>$/) ?? [null, payload.from, payload.from];
  const senderName = fromMatch[1]?.trim() || 'MicroFlow';
  const senderEmail = fromMatch[2]?.trim() || payload.from;

  const body = {
    sender: { name: senderName, email: senderEmail },
    to: [{ email: payload.to }],
    subject: payload.subject,
    htmlContent: payload.html,
    ...(payload.text ? { textContent: payload.text } : {}),
    // Message-ID como header para deduplicação
    ...(payload.messageId ? { headers: { 'Message-ID': payload.messageId } } : {}),
  };

  const response = await fetch('https://api.brevo.com/v3/smtp/email', {
    method: 'POST',
    headers: {
      'api-key': apiKey,
      'Content-Type': 'application/json',
      'Accept': 'application/json',
    },
    body: JSON.stringify(body),
  });

  if (!response.ok) {
    const errorText = await response.text().catch(() => 'unknown error');
    // Estruturar o erro para que isRateLimited() funcione no process-email-queue
    const err = new Error(`Brevo API error ${response.status}: ${errorText}`);
    (err as any).status = response.status;
    throw err;
  }

  const result = await response.json() as { messageId: string };
  return { messageId: result.messageId };
}
```

### Por que `(err as any).status`

A função `isRateLimited()` em `process-email-queue` verifica `error.status === 429`. Ao colocar o status HTTP no objeto de erro, o backoff exponencial existente funciona automaticamente para erros de rate limit da API do Brevo — sem precisar alterar a lógica de retry.

### Critérios de Aceite

- [ ] Arquivo `supabase/functions/_shared/brevo.ts` existe
- [ ] Função `sendBrevoEmail` exportada com a assinatura correta
- [ ] Erro com `status` 429 é lançado para o caller quando Brevo retorna 429
- [ ] Erro com `status` 400 é lançado quando payload inválido

## TAREFA T3 — Configurar domínio remetente no Brevo

**Pilar:** Segurança / Entregabilidade

### Por que crítico

Sem SPF e DKIM configurados, e-mails enviados pelo Brevo usando o domínio `microflow.com.br` (ou o domínio real do produto) caem em spam em ~60% dos provedores. Isso é especialmente crítico para e-mails de confirmação de conta e reset de senha.

### Passos no Brevo Dashboard

1. **Acessar:** Brevo Dashboard → Senders & IP → Domains
2. **Adicionar domínio:** `microflow.com.br` (ou o domínio real)
3. **Configurar registros DNS** (fornecidos pelo Brevo após adicionar o domínio):

```
# Registro SPF (TXT no domínio raiz ou subdomínio)
TXT  @   "v=spf1 include:spf.brevo.com mx ~all"

# Registro DKIM (TXT fornecido pelo Brevo — exemplo de formato)
TXT  mail._domainkey   "v=DKIM1; k=rsa; p=<chave-publica-brevo>"

# Registro DMARC (recomendado)
TXT  _dmarc   "v=DMARC1; p=none; rua=mailto:dmarc@microflow.com.br"
```

4. **Verificar no Brevo:** após propagação DNS (até 48h), clicar em "Verify" no Brevo Dashboard
5. **Configurar o remetente padrão:**
   - Name: `MicroFlow`
   - Email: `noreply@microflow.com.br`

### Configurar endereço de resposta para suporte

```
Reply-To: suporte@microflow.com.br
```

Configurar isso como header padrão no Brevo ou passar como campo adicional na API:

```typescript
// Em sendBrevoEmail, adicionar ao body:
replyTo: { email: 'suporte@microflow.com.br', name: 'MicroFlow Suporte' }
```

### Critérios de Aceite

- [ ] Domínio verificado no Brevo Dashboard (ícone verde)
- [ ] SPF e DKIM verificados
- [ ] E-mail de teste enviado pelo Brevo chega na caixa de entrada (não spam) no Gmail e Outlook
- [ ] Header `DMARC` configurado em modo monitoramento (`p=none`)

## TAREFA T5 — Limpar dependências do Lovable em `auth-email-hook`

**Arquivo:** `supabase/functions/auth-email-hook/index.ts`
**Pilar:** Manutenibilidade

### Contexto

O `auth-email-hook` ainda usa:
- `npm:@lovable.dev/email-js` (para `parseEmailWebhookPayload`)
- `npm:@lovable.dev/webhooks-js` (para `verifyWebhookRequest`, `WebhookError`)
- `LOVABLE_API_KEY` para verificar a assinatura do webhook

Esses pacotes são específicos do hook de e-mail do Lovable Platform. Se o projeto for movido para um Supabase standalone, esses hooks não funcionarão.

### O que fazer

**Verificar se o `auth-email-hook` é disparado pelo Supabase nativo ou pelo Lovable:**

Se for um **Auth Hook nativo do Supabase** (configurado em Authentication → Hooks), o payload é diferente e a verificação de assinatura usa o segredo HMAC do Supabase, não o `LOVABLE_API_KEY`.

Para migrar para Supabase Auth Hooks nativos:

```typescript
// Novo auth-email-hook usando Supabase Auth Hook nativo
// Documentação: https://supabase.com/docs/guides/auth/auth-hooks

Deno.serve(async (req) => {
  // Supabase assina o webhook com HMAC-SHA256
  const hookSecret = Deno.env.get('SUPABASE_AUTH_HOOK_SECRET')!;
  const signature = req.headers.get('x-supabase-signature') ?? '';

  const body = await req.text();

  // Verificar HMAC
  const key = await crypto.subtle.importKey(
    'raw',
    new TextEncoder().encode(hookSecret),
    { name: 'HMAC', hash: 'SHA-256' },
    false,
    ['sign', 'verify'],
  );
  const expectedSig = await crypto.subtle.sign('HMAC', key, new TextEncoder().encode(body));
  const expectedHex = Array.from(new Uint8Array(expectedSig))
    .map(b => b.toString(16).padStart(2, '0')).join('');

  if (signature !== expectedHex) {
    return new Response(JSON.stringify({ error: 'Invalid signature' }), { status: 401 });
  }

  const payload = JSON.parse(body);
  const emailType = payload.event_type; // 'signup', 'recovery', etc.
  // ... renderizar template e enfileirar como antes ...
});
```

**Observação:** Esta tarefa requer validação de qual mecanismo exato o projeto usa atualmente. Se o `auth-email-hook` estiver funcionando em produção via Lovable, priorizar primeiro as tarefas T1-T4 e deixar esta como refatoração pós-launch.

### Critérios de Aceite

- [ ] `auth-email-hook` funciona sem dependência de `LOVABLE_API_KEY` (ou a dependência está documentada como dívida técnica aceita)
- [ ] E-mails de signup, recovery e magic-link chegam normalmente após a migração
