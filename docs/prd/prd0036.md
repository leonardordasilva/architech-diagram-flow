PRD 0036 — Stripe Webhooks: Hardening para Produção
Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Gap de produção — eventos Stripe não mapeados e configuração não verificada Plataforma: Lovable Cloud / Supabase Edge Functions

## Contexto

A edge function `stripe-webhook` existe e trata `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted` e `invoice.payment_failed`. No entanto, há 4 problemas que impedem que o billing funcione de forma confiável em produção:

1. **`invoice.payment_failed` não rebaixa o plano** — apenas seta `status = 'past_due'`. Usuários com cartão recusado continuam com acesso Pro indefinidamente.
2. **`invoice.paid` não é tratado** — renovações mensais bem-sucedidas não atualizam `current_period_end` no banco, então após o primeiro ciclo o período de validade fica desatualizado.
3. **`customer.subscription.updated` com `cancel_at_period_end = true`** não é comunicado ao usuário — o usuário que cancela pelo portal não recebe feedback visual na aplicação.
4. **Configuração não documentada** — não há registro de que o endpoint foi registrado no Stripe Dashboard e que `STRIPE_WEBHOOK_SECRET` está configurado nos Supabase Secrets, criando risco de que o webhook não esteja ativo em produção.

## TAREFA T1 — Tratar `invoice.paid`

**Arquivo:** `supabase/functions/stripe-webhook/index.ts`
**Pilar:** Correctness

### Problema

Quando uma assinatura Pro renova mensalmente, o Stripe emite `invoice.paid`. Sem esse handler, `current_period_end` na tabela `subscriptions` fica com a data do checkout original, nunca atualizada. Isso afeta qualquer lógica de UI que mostre "sua assinatura vence em X" e potencialmente lógica de downgrades baseada em período.

### O que fazer

Adicionar o case `invoice.paid` no switch de eventos, ANTES do `default`:

```typescript
case "invoice.paid": {
  const invoice = event.data.object as Stripe.Invoice;

  // Apenas faturas de assinatura (não one-time)
  const subscriptionId = typeof invoice.subscription === "string"
    ? invoice.subscription
    : invoice.subscription?.id;
  if (!subscriptionId) break;

  // Buscar usuário pela assinatura Stripe
  const customerId = typeof invoice.customer === "string"
    ? invoice.customer
    : invoice.customer?.id || "";

  const { data: dbSub } = await serviceSupabase
    .from("subscriptions")
    .select("user_id, plan")
    .eq("stripe_customer_id", customerId)
    .maybeSingle();

  if (!dbSub?.user_id) {
    console.warn(JSON.stringify({ fn: 'stripe-webhook', msg: 'invoice.paid: no user found', customerId }));
    break;
  }

  // Buscar período atual da assinatura no Stripe
  const sub = await stripe.subscriptions.retrieve(subscriptionId);

  await serviceSupabase
    .from("subscriptions")
    .update({
      status: "active",                          // garante reativação após past_due
      current_period_start: new Date(sub.current_period_start * 1000).toISOString(),
      current_period_end: new Date(sub.current_period_end * 1000).toISOString(),
      updated_at: new Date().toISOString(),
    })
    .eq("user_id", dbSub.user_id);

  // Garantir que profiles.plan também está correto (segurança contra dessincronização)
  await serviceSupabase
    .from("profiles")
    .update({ plan: dbSub.plan })
    .eq("id", dbSub.user_id);

  console.log(JSON.stringify({ fn: 'stripe-webhook', msg: 'invoice.paid: period updated', userId: dbSub.user_id }));
  break;
}
```

### Critérios de Aceite

- [ ] Após renovação simulada no Stripe Dashboard (modo teste → "Create test invoice"), `current_period_end` no banco avança um mês
- [ ] Um usuário que estava `past_due` tem `status` revertido para `active` após pagamento bem-sucedido
- [ ] Log `invoice.paid: period updated` aparece nos logs da edge function

## TAREFA T2 — Grace period de 7 dias para `invoice.payment_failed`

**Arquivo:** `supabase/functions/stripe-webhook/index.ts`
**Pilar:** Correctness / Negócio

### Problema atual

```typescript
// ANTES — status muda para past_due mas plano não muda
case "invoice.payment_failed": {
  await serviceSupabase
    .from("subscriptions")
    .update({ status: "past_due", updated_at: ... })
    .eq("user_id", sub.user_id);
  break;
}
```

O usuário com cartão vencido continua com acesso Pro eternamente, pois `profiles.plan` nunca muda. Isso é um bug de billing.

### Estratégia

O Stripe já reencaminha automaticamente `invoice.payment_failed` com `attempt_count` crescente (1 → 2 → 3). Na terceira falha, o Stripe marca a assinatura como `canceled` e emite `customer.subscription.deleted`, que já faz o downgrade. No entanto, precisamos de uma política explícita:

- **1ª e 2ª falha:** apenas `past_due`, manter acesso (grace period para o usuário atualizar cartão)
- **3ª+ falha:** downgrade para `free` imediatamente (antes do `subscription.deleted` chegar)

```typescript
// DEPOIS
case "invoice.payment_failed": {
  const invoice = event.data.object as Stripe.Invoice;
  const customerId = typeof invoice.customer === "string"
    ? invoice.customer
    : invoice.customer?.id || "";
  if (!customerId) break;

  const { data: dbSub } = await serviceSupabase
    .from("subscriptions")
    .select("user_id")
    .eq("stripe_customer_id", customerId)
    .maybeSingle();

  if (!dbSub?.user_id) break;

  const attemptCount = invoice.attempt_count ?? 1;
  const newPlan = attemptCount >= 3 ? "free" : undefined; // downgrade apenas na 3ª tentativa

  const updateData: Record<string, unknown> = {
    status: "past_due",
    updated_at: new Date().toISOString(),
  };
  if (newPlan) {
    updateData.plan = newPlan;
  }

  await serviceSupabase
    .from("subscriptions")
    .update(updateData)
    .eq("user_id", dbSub.user_id);

  // Rebaixar plano no profiles apenas na 3ª falha
  if (newPlan) {
    await serviceSupabase
      .from("profiles")
      .update({ plan: "free" })
      .eq("id", dbSub.user_id);

    console.log(JSON.stringify({
      fn: 'stripe-webhook',
      msg: 'invoice.payment_failed: user downgraded after 3 attempts',
      userId: dbSub.user_id,
      attemptCount,
    }));
  }
  break;
}
```

### Critérios de Aceite

- [ ] Após 1ª falha: `subscriptions.status = 'past_due'`, `profiles.plan` permanece `'pro'`
- [ ] Após 3ª falha: `subscriptions.status = 'past_due'` e `profiles.plan = 'free'`
- [ ] O log `user downgraded after 3 attempts` aparece na 3ª falha

## TAREFA T3 — Comunicar `cancel_at_period_end`

**Arquivo:** `supabase/functions/stripe-webhook/index.ts`
**Pilar:** Correctness / UX

### Problema

Quando um usuário Pro clica em "cancelar assinatura" no portal Stripe, o Stripe emite `customer.subscription.updated` com `cancel_at_period_end = true`. O handler atual faz upsert normalmente mas não salva esse flag — o usuário não vê na aplicação que sua assinatura foi cancelada e vai expirar.

### O que fazer

**Passo 1 — Adicionar coluna `cancel_at_period_end` na tabela `subscriptions`:**

```sql
-- supabase/migrations/YYYYMMDD_prd0036_cancel_at_period_end.sql
ALTER TABLE subscriptions
  ADD COLUMN IF NOT EXISTS cancel_at_period_end boolean NOT NULL DEFAULT false;
```

**Passo 2 — Salvar o flag no handler `customer.subscription.updated`:**

Localizar a chamada `upsertSubscription` dentro de `customer.subscription.updated` e adicionar o campo:

```typescript
// Dentro do case "customer.subscription.updated":
// Após resolver plan, billingCycle, etc:

// Salvar cancel_at_period_end
const cancelAtPeriodEnd = sub.cancel_at_period_end ?? false;

// Incluir na função upsertSubscription:
await serviceSupabase
  .from("subscriptions")
  .upsert({
    ...record,               // campos existentes
    cancel_at_period_end: cancelAtPeriodEnd,
  }, { onConflict: "user_id" });
```

**Passo 3 — Atualizar `upsertSubscription` para aceitar o novo campo:**

```typescript
async function upsertSubscription(params: {
  userId: string;
  plan: string;
  status: string;
  billingCycle?: string;
  stripeCustomerId?: string;
  stripeSubscriptionId?: string;
  currentPeriodStart?: number;
  currentPeriodEnd?: number;
  cancelAtPeriodEnd?: boolean;   // novo
}) {
  const record: Record<string, unknown> = {
    // ... campos existentes ...
  };
  if (params.cancelAtPeriodEnd !== undefined) {
    record.cancel_at_period_end = params.cancelAtPeriodEnd;
  }
  // ... resto igual ...
}
```

**Passo 4 — Exibir aviso no `BillingModal.tsx` quando `cancel_at_period_end = true`:**

```typescript
// src/components/BillingModal.tsx
// Na seção onde exibe dados da assinatura Pro, adicionar:
{subscription?.cancel_at_period_end && (
  <div className="rounded-md border border-yellow-500/30 bg-yellow-500/10 px-4 py-3 text-sm text-yellow-700 dark:text-yellow-400">
    {t('billing.cancelScheduled', {
      date: subscription.current_period_end
        ? new Date(subscription.current_period_end).toLocaleDateString()
        : '—'
    })}
  </div>
)}
```

Adicionar chave de tradução em `pt-BR`:
```json
"billing.cancelScheduled": "Sua assinatura está cancelada e expirará em {{date}}. Você manterá acesso Pro até essa data."
```

### Critérios de Aceite

- [ ] Migration executada: coluna `cancel_at_period_end boolean DEFAULT false` existe
- [ ] Ao cancelar pelo portal Stripe (modo teste), `subscriptions.cancel_at_period_end = true`
- [ ] `BillingModal` exibe o aviso amarelo quando `cancel_at_period_end = true`
- [ ] Quando o período termina e Stripe emite `subscription.deleted`, `cancel_at_period_end` volta para `false` (via upsert do handler `subscription.deleted`)
