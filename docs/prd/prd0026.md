Tarefa F1-T3: Documentar ADR para style-src unsafe-inline (+0.5 pt)
Pilar: Segurança & Sessão Prioridade: Média Impacto na nota: +0.5

Contexto: A CSP contém style-src 'self' 'unsafe-inline'. Isso é uma concessão necessária porque Radix UI, React Flow e componentes com style prop injetam estilos inline dinamicamente. O Lovable Cloud não oferece controle sobre HTTP headers para injeção de nonces dinâmicos em styles. Esta decisão precisa ser documentada formalmente.

Requisitos:

Criar docs/adr/002-csp-unsafe-inline-styles.md com o seguinte conteúdo (adaptar datas e formato ao padrão do ADR-001 existente):
Status: Accepted
Contexto: A CSP do index.html inclui style-src 'self' 'unsafe-inline'. script-src já foi endurecida para 'self' apenas (sem unsafe-eval nem unsafe-inline). Para styles, a remoção de unsafe-inline quebraria Radix UI (injecta styles inline via JS), React Flow (posiciona nós com style attributes), Tailwind (classe utilitária, mas sem problema), e componentes com style prop.
Decisão: Manter unsafe-inline em style-src como concessão documentada. O risco de CSS injection é significativamente menor que script injection. A superfície de ataque é limitada porque: (1) script-src não permite inline, então CSS não pode escalar para XSS; (2) todos os inputs do usuário passam por DOMPurify; (3) a aplicação não renderiza CSS user-provided.
Alternativas rejeitadas: Nonces via HTTP header (requer controle de servidor, indisponível no Lovable Cloud). Hash-based CSP para styles (impraticável com Radix UI que gera styles dinâmicos). Shadow DOM isolation (reescrita arquitetural desproporcional).
Consequências: Aceita-se o risco residual de CSS injection como baixo. Revisitar se Lovable Cloud adicionar suporte a HTTP response headers customizados.

Criar docs/runbook.md com as seguintes seções:

Monitoramento — Como verificar saúde da aplicação: chamar GET /functions/v1/health e interpretar resposta (200 healthy, 503 degraded, campos checks.database e checks.encryption). Frequência recomendada: manual semanal ou via cron externo.

Manutenção de banco — Purge de soft-deleted diagrams: executado automaticamente via pg_cron (semanalmente, domingos às 3h). Para forçar manualmente: SELECT public.purge_old_soft_deleted_diagrams();. Purge de rate limits: SELECT public.purge_old_rate_limits(); (executado a cada 5 min via pg_cron). Verificar execução: SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 10;.

Backfill de criptografia — Quando executar: após migração de dados legados ou restore de backup. Como executar: POST /functions/v1/backfill-encryption com header Authorization: Bearer <service_role_key>. Interpretar resposta: { processed, skipped, errors, total }. Recomenda-se executar em horário de baixa atividade.

Troubleshooting comum — (a) "Canvas em branco após save": provável payload criptografado no realtime — o hook useRealtimeCollab ignora non-arrays, recarregar a página resolve. (b) "content_hash mismatch warning": indica possível corrupção — verificar logs, restaurar do backup se necessário. (c) "Rate limit 429": aguardar 60 segundos ou verificar se rate_limits tem entradas acumuladas. (d) "DIAGRAM_LIMIT_EXCEEDED": usuário atingiu limite do plano — upgrade ou deletar diagramas existentes.

Atualização de secrets — Lista de secrets no Supabase: DIAGRAM_ENCRYPTION_KEY, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, SUPABASE_SERVICE_ROLE_KEY, ALLOWED_ORIGINS, SITE_URL, ADMIN_USER_IDS (adicionado em F1-T2). Procedimento: atualizar via Supabase Dashboard > Edge Functions > Secrets. Após atualizar, as Edge Functions fazem hot-reload automático.

Feature flags — Como ativar/desativar uma flag: UPDATE public.feature_flags SET enabled = false WHERE key = 'atomic_save';. Criar nova flag: INSERT INTO public.feature_flags (key, enabled, description) VALUES ('new_feature', false, 'Description');. Flags são cacheadas no client por 10 minutos.
