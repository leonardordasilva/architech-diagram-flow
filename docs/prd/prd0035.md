PRD 0035 — Monitoramento de Erros em Produção (Sentry)
Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Gap de produção — ausência de observabilidade Plataforma: Lovable Cloud Premissas: Backend Supabase permanente. Sem CI/CD externo.

## Contexto

A aplicação possui um `ErrorBoundary` global (`src/components/ErrorBoundary.tsx`) e um handler de `unhandledrejection`, mas ambos apenas loguam no console do navegador. Em produção, nenhum erro de usuário real chega ao desenvolvedor — a aplicação é operada de forma completamente cega.

Sem monitoramento externo, erros críticos só serão descobertos quando:
- Um usuário reportar manualmente (baixíssima taxa de reporte)
- Dados inconsistentes forem percebidos no banco (tarde demais)
- A aplicação estiver completamente quebrada para todos os usuários

O Sentry é o padrão da indústria para rastreamento de erros frontend. A integração é feita em ~50 linhas e captura automaticamente: erros de render (via ErrorBoundary), rejeições de Promise não tratadas, erros de rede, e snapshots do estado da aplicação no momento do crash.

## TAREFA T1 — Instalar e configurar o SDK do Sentry

**Arquivo:** `src/main.tsx`, `vite.config.ts`
**Pilar:** Observabilidade

### O que fazer

**Passo 1 — Instalar o SDK:**

```bash
npm install @sentry/react @sentry/vite-plugin
```

**Passo 2 — Inicializar o Sentry em `src/main.tsx` ANTES do render:**

```typescript
// src/main.tsx
import * as Sentry from '@sentry/react';

Sentry.init({
  dsn: import.meta.env.VITE_SENTRY_DSN,
  environment: import.meta.env.MODE, // 'development' | 'production'
  enabled: import.meta.env.PROD,     // só ativo em produção
  release: import.meta.env.VITE_APP_VERSION ?? 'unknown',
  tracesSampleRate: 0.1,             // 10% das transações para performance
  replaysSessionSampleRate: 0,       // sem replay de sessão (privacidade + custo)
  replaysOnErrorSampleRate: 0,
  // Ignorar erros conhecidos que não são bugs do produto
  ignoreErrors: [
    'ResizeObserver loop limit exceeded',
    'ResizeObserver loop completed with undelivered notifications',
    'Network request failed',
    'Load failed',
    'ChunkLoadError',
  ],
  beforeSend(event) {
    // T4: Remover dados sensíveis antes de enviar
    if (event.user) {
      delete event.user.ip_address;
    }
    return event;
  },
});
```

**Passo 3 — Adicionar o plugin do Sentry ao `vite.config.ts` para upload de source maps:**

```typescript
// vite.config.ts
import { sentryVitePlugin } from '@sentry/vite-plugin';

export default defineConfig({
  // ... config existente ...
  build: {
    sourcemap: true, // necessário para stack traces legíveis no Sentry
  },
  plugins: [
    // ... plugins existentes ...
    ...(process.env.NODE_ENV === 'production' && process.env.SENTRY_AUTH_TOKEN
      ? [sentryVitePlugin({
          org: process.env.SENTRY_ORG,
          project: process.env.SENTRY_PROJECT,
          authToken: process.env.SENTRY_AUTH_TOKEN,
        })]
      : []),
  ],
});
```

**Observação:** `SENTRY_AUTH_TOKEN`, `SENTRY_ORG` e `SENTRY_PROJECT` são variáveis de build (não `VITE_*`) — ficam apenas no servidor de CI, nunca no bundle.

### Critérios de Aceite

- [ ] `npm run build` conclui sem erros com Sentry plugin ativo
- [ ] Em desenvolvimento (`import.meta.env.PROD === false`), o Sentry NÃO envia eventos
- [ ] Source maps são gerados (`dist/*.js.map` existem)

## TAREFA T2 — Integrar Sentry ao `ErrorBoundary` existente

**Arquivo:** `src/components/ErrorBoundary.tsx`
**Pilar:** Correctness

### Problema atual

```typescript
// ANTES — ErrorBoundary.tsx
componentDidCatch(error: Error, info: ErrorInfo) {
  console.error('[ErrorBoundary]', error, info.componentStack);
  toast({ ... });
}
```

O erro vai para o console do navegador e desaparece.

### O que fazer

```typescript
// DEPOIS — ErrorBoundary.tsx
import * as Sentry from '@sentry/react';

componentDidCatch(error: Error, info: ErrorInfo) {
  console.error('[ErrorBoundary]', error, info.componentStack);
  Sentry.captureException(error, {
    extra: { componentStack: info.componentStack },
    tags: { source: 'ErrorBoundary' },
  });
  toast({
    title: i18n.t('errors.unexpected'),
    description: i18n.t('errors.unexpectedDesc'),
    variant: 'destructive',
  });
}
```

Adicionalmente, melhorar a UI de fallback para usar o design system em vez de estilos inline:

```tsx
// DEPOIS — render() fallback
render() {
  if (this.state.hasError) {
    return (
      <div className="flex min-h-screen items-center justify-center bg-background">
        <div className="text-center space-y-4 p-8 max-w-md">
          <h2 className="text-xl font-semibold text-foreground">
            {i18n.t('errors.somethingWrong')}
          </h2>
          <p className="text-sm text-muted-foreground">
            {i18n.t('errors.unexpectedDesc')}
          </p>
          <button
            className="inline-flex items-center justify-center rounded-md bg-primary px-6 py-2 text-sm font-medium text-primary-foreground hover:bg-primary/90 transition-colors"
            onClick={() => this.setState({ hasError: false })}
          >
            {i18n.t('common.tryAgain')}
          </button>
        </div>
      </div>
    );
  }
  return this.props.children;
}
```

### Critérios de Aceite

- [ ] Um erro simulado no render (`throw new Error('test')`) aparece no dashboard do Sentry
- [ ] O fallback de UI usa classes Tailwind, não estilos inline
- [ ] O toast continua sendo exibido ao usuário

## TAREFA T3 — Integrar Sentry ao handler de `unhandledrejection`

**Arquivo:** `src/components/ErrorBoundary.tsx` (seção `window.addEventListener`)
**Pilar:** Correctness

### O que fazer

```typescript
// DEPOIS — handler no fim de ErrorBoundary.tsx
if (typeof window !== 'undefined') {
  window.addEventListener('unhandledrejection', (event) => {
    console.error('[UnhandledRejection]', event.reason);
    Sentry.captureException(event.reason instanceof Error
      ? event.reason
      : new Error(String(event.reason)), {
      tags: { source: 'unhandledRejection' },
    });
    toast({
      title: i18n.t('errors.networkOp'),
      description: event.reason?.message ?? i18n.t('errors.asyncFailed'),
      variant: 'destructive',
    });
  });
}
```

### Critérios de Aceite

- [ ] Uma Promise rejeitada não tratada (`Promise.reject(new Error('async-test'))` no console) aparece no Sentry
- [ ] O evento inclui a tag `source: unhandledRejection`

## TAREFA T4 — Configurar filtros de PII e rate limiting

**Arquivo:** `src/main.tsx` (dentro de `Sentry.init`)
**Pilar:** Segurança / Privacidade (LGPD)

### O que fazer

Complementar o `beforeSend` da T1 com remoção de e-mail e identificação mínima:

```typescript
Sentry.init({
  // ... config T1 ...
  beforeSend(event) {
    // Remover IP e identificadores pessoais
    if (event.user) {
      delete event.user.ip_address;
      delete event.user.email;         // não enviar e-mail ao Sentry
      // manter apenas event.user.id (UUID anônimo) para correlacionar sessões
    }
    // Remover query strings que possam conter tokens em URLs
    if (event.request?.url) {
      try {
        const url = new URL(event.request.url);
        url.search = '';               // remover ?token=... etc
        event.request.url = url.toString();
      } catch {
        // URL inválida — manter como está
      }
    }
    return event;
  },
  // Identificar usuário sem PII (só UUID) quando disponível
  // chamado após login em src/hooks/useAuth.tsx:
  // Sentry.setUser({ id: user.id });
  // chamado após logout:
  // Sentry.setUser(null);
});
```

Adicionar identificação do usuário no `useAuth`:

```typescript
// src/hooks/useAuth.tsx — dentro do handler de sessão autenticada
import * as Sentry from '@sentry/react';

// Após confirmar sessão ativa:
Sentry.setUser({ id: user.id });

// No signOut:
Sentry.setUser(null);
```

### Critérios de Aceite

- [ ] Eventos no Sentry contêm `user.id` (UUID) mas NÃO `user.email` nem `user.ip_address`
- [ ] URLs nos eventos não contêm query strings com tokens
- [ ] Após logout, `user` fica `null` nos novos eventos


```bash

VITE_APP_VERSION=1.0.0
```

## Impacto Esperado

| Situação | Antes | Depois |
|----------|-------|--------|
| Erro de render em produção | Invisível (console) | Alerta no Sentry em segundos |
| Promise rejeitada não tratada | Invisível | Capturado com stack trace |
| Diagnóstico de bug reportado por usuário | Horas/dias de investigação | Stack trace + contexto imediato |
| Dados de PII no log de erros | Risco de vazamento | E-mail e IP removidos antes do envio |
| Visibilidade do ambiente (dev/prod) | Nenhuma | Tag `environment` em cada evento |

## Configuração no Dashboard Sentry

Após deploy, configurar no dashboard do Sentry:
1. **Alertas:** criar regra "Novo erro único → notificação imediata por e-mail"
2. **Rate limiting:** 1000 eventos/hora por projeto (evitar custo com loops de erro)
3. **Inbound filters:** ativar filtro de bots e crawlers
4. **Data Scrubbing:** adicionar `password`, `token`, `access_token` à lista de campos sensíveis
