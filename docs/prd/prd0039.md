PRD 0039 — Resiliência de UI: ErrorBoundary por Rota e Estados de Erro Globais
Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Gap de produção — crash isolado derruba toda a aplicação Plataforma: Lovable Cloud Premissas: React 18, React Router v6, Suspense já em uso.

## Contexto

A aplicação possui um `ErrorBoundary` global em `src/App.tsx` que envolve toda a árvore de componentes. Isso significa que **um erro de render em qualquer componente (ex: canvas, modal, painel lateral) derruba completamente a aplicação** — o usuário vê a tela de fallback "algo deu errado" e perde o contexto de onde estava.

Além disso:
- O `NotFound.tsx` existente é funcional mas minimal (sem branding, sem botão "Voltar")
- A tela de fallback do `ErrorBoundary` usa estilos inline (já parcialmente corrigido no PRD-0035 T2), mas não tem opção de "Reportar problema" nem link para a página inicial
- Erros de carregamento de rota (ex: chunk load error em lazy imports) mostram spinner eterno em vez de mensagem de erro

Em produção com múltiplos usuários, o isolamento de erros é crítico: um bug num modal de billing não deve quebrar o canvas do usuário.

## TAREFA T1 — Criar `RouteErrorBoundary`

**Arquivo:** `src/components/RouteErrorBoundary.tsx` (novo)
**Pilar:** Correctness

### Problema

O `ErrorBoundary` global (`src/components/ErrorBoundary.tsx`) captura erros de qualquer nó da árvore. Se um erro ocorre no canvas, o fallback substitui TODA a aplicação. O usuário perde o estado não salvo e não pode recuperar.

### Solução

Criar um `ErrorBoundary` granular que envolve cada rota individualmente. Erros numa rota ficam contidos nela — as demais continuam funcionando.

```tsx
// src/components/RouteErrorBoundary.tsx
import { Component, type ErrorInfo, type ReactNode } from 'react';
import * as Sentry from '@sentry/react';
import { Link } from 'react-router-dom';
import i18n from '@/i18n';

interface Props {
  children: ReactNode;
  routeName?: string; // para logging contextual
}

interface State {
  hasError: boolean;
  errorId?: string;
}

export class RouteErrorBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(): State {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    const errorId = crypto.randomUUID();
    this.setState({ errorId });

    console.error(`[RouteErrorBoundary:${this.props.routeName ?? 'unknown'}]`, error, info.componentStack);

    // PRD-0035: Integração Sentry
    Sentry.captureException(error, {
      extra: { componentStack: info.componentStack, routeName: this.props.routeName },
      tags: { source: 'RouteErrorBoundary', route: this.props.routeName ?? 'unknown' },
    });
  }

  render() {
    if (this.state.hasError) {
      return (
        <div className="flex min-h-[60vh] flex-col items-center justify-center gap-4 p-8 text-center">
          <div className="space-y-2">
            <h2 className="text-lg font-semibold text-foreground">
              {i18n.t('errors.somethingWrong')}
            </h2>
            <p className="text-sm text-muted-foreground max-w-sm">
              {i18n.t('errors.unexpectedDesc')}
            </p>
            {this.state.errorId && (
              <p className="text-xs text-muted-foreground/60 font-mono">
                {i18n.t('errors.errorId', 'ID do erro:')} {this.state.errorId}
              </p>
            )}
          </div>
          <div className="flex gap-3">
            <button
              className="inline-flex items-center rounded-md border border-border bg-background px-4 py-2 text-sm font-medium hover:bg-accent transition-colors"
              onClick={() => this.setState({ hasError: false, errorId: undefined })}
            >
              {i18n.t('common.tryAgain')}
            </button>
            <Link
              to="/"
              className="inline-flex items-center rounded-md bg-primary px-4 py-2 text-sm font-medium text-primary-foreground hover:bg-primary/90 transition-colors"
            >
              {i18n.t('notFound.home', 'Ir para o início')}
            </Link>
          </div>
        </div>
      );
    }

    return this.props.children;
  }
}
```

### Por que exibir o `errorId`

O `errorId` (UUID) é o mesmo enviado ao Sentry. Quando um usuário reporta "tive um erro", basta pedir o ID e buscar no Sentry — sem precisar de reprodução.

### Critérios de Aceite

- [ ] Arquivo `src/components/RouteErrorBoundary.tsx` criado
- [ ] Um erro simulado numa rota filha não derruba o `ErrorBoundary` global
- [ ] O `errorId` aparece na tela E no Sentry como tag
- [ ] Botão "Tentar novamente" limpa o erro e remonta a rota

## TAREFA T2 — Envolver cada rota com `RouteErrorBoundary`

**Arquivo:** `src/App.tsx`
**Pilar:** Correctness

### O que fazer

```tsx
// src/App.tsx — DEPOIS
import { RouteErrorBoundary } from '@/components/RouteErrorBoundary';

// Envolver cada rota sensível individualmente:
<Routes>
  <Route path="/" element={<Landing />} />

  <Route path="/app" element={
    <RouteErrorBoundary routeName="canvas">
      <Index />
    </RouteErrorBoundary>
  } />

  <Route path="/reset-password" element={
    <RouteErrorBoundary routeName="reset-password">
      <ResetPassword />
    </RouteErrorBoundary>
  } />

  <Route path="/diagram/:shareToken" element={
    <RouteErrorBoundary routeName="shared-diagram">
      <SharedDiagram />
    </RouteErrorBoundary>
  } />

  <Route path="/billing" element={
    <ProtectedRoute>
      <RouteErrorBoundary routeName="billing">
        <Billing />
      </RouteErrorBoundary>
    </ProtectedRoute>
  } />

  <Route path="/workspace" element={
    <ProtectedRoute>
      <RouteErrorBoundary routeName="workspace">
        <Workspace />
      </RouteErrorBoundary>
    </ProtectedRoute>
  } />

  <Route path="/workspace/diagrams" element={
    <ProtectedRoute>
      <RouteErrorBoundary routeName="workspace-diagrams">
        <WorkspaceDiagrams />
      </RouteErrorBoundary>
    </ProtectedRoute>
  } />

  <Route path="/invite" element={
    <RouteErrorBoundary routeName="accept-invite">
      <AcceptInvite />
    </RouteErrorBoundary>
  } />

  <Route path="/admin/*" element={
    <AdminGuard>
      <RouteErrorBoundary routeName="admin">
        <AdminApp />
      </RouteErrorBoundary>
    </AdminGuard>
  } />

  <Route path="*" element={<NotFound />} />
</Routes>
```

**Nota:** A rota `/` (Landing) e `*` (NotFound) não precisam de `RouteErrorBoundary` — a Landing é uma página estática simples e o NotFound já é o fallback final.

### Critérios de Aceite

- [ ] Todas as 7 rotas com lógica de negócio têm `RouteErrorBoundary`
- [ ] Cada `routeName` é único e descritivo
- [ ] Um erro no `/billing` não afeta o canvas em `/app` (testar em duas abas)
- [ ] O `ErrorBoundary` global em `App.tsx` permanece como última linha de defesa

2. Sem logo/branding — parece uma página de erro genérica
3. Sem botão de "Voltar" para a página anterior
4. Sem sugestão de ação alternativa

### O que fazer

```tsx
// DEPOIS — src/pages/NotFound.tsx
import { Link, useNavigate, useLocation } from 'react-router-dom';
import { useEffect } from 'react';
import { useTranslation } from 'react-i18next';
import { useAuth } from '@/hooks/useAuth';

const NotFound = () => {
  const { t } = useTranslation();
  const { user } = useAuth();
  const location = useLocation();
  const navigate = useNavigate();

  useEffect(() => {
    console.error('404 Error: User attempted to access non-existent route:', location.pathname);
  }, [location.pathname]);

  return (
    <div className="flex min-h-screen flex-col items-center justify-center bg-background gap-6 p-8 text-center">
      {/* Logo ou nome do produto */}
      <div className="text-2xl font-bold text-primary">MicroFlow</div>

      <div className="space-y-2">
        <h1 className="text-6xl font-bold text-foreground">404</h1>
        <p className="text-lg text-muted-foreground">{t('notFound.title')}</p>
        <p className="text-sm text-muted-foreground/70">
          {t('notFound.subtitle', 'A página que você procura não existe ou foi movida.')}
        </p>
      </div>

      <div className="flex flex-col sm:flex-row gap-3">
        <button
          onClick={() => navigate(-1)}
          className="inline-flex items-center justify-center rounded-md border border-border bg-background px-5 py-2.5 text-sm font-medium hover:bg-accent transition-colors"
        >
          {t('common.goBack', '← Voltar')}
        </button>
        <Link
          to={user ? '/app' : '/'}
          className="inline-flex items-center justify-center rounded-md bg-primary px-5 py-2.5 text-sm font-medium text-primary-foreground hover:bg-primary/90 transition-colors"
        >
          {user ? t('notFound.goToApp', 'Ir para o app') : t('notFound.home', 'Ir para o início')}
        </Link>
      </div>
    </div>
  );
};

export default NotFound;
```

## TAREFA T4 — Tratar `ChunkLoadError`

**Arquivo:** `src/components/ErrorBoundary.tsx` e `src/App.tsx`
**Pilar:** Correctness

### Problema

Quando o Vite faz um novo deploy, os hashes dos chunks JavaScript mudam. Usuários com a versão antiga aberta no browser tentam carregar chunks que não existem mais → `ChunkLoadError: Loading chunk X failed`.

Com Suspense + lazy loading (já usado em `App.tsx`), esse erro se manifesta como:
1. O Suspense fallback (spinner) é mostrado
2. O import do chunk falha
3. O erro sobe para o ErrorBoundary — mas sem tratamento específico, parece um bug genérico

### O que fazer

**Detectar `ChunkLoadError` e recarregar automaticamente (uma vez):**

```typescript
// src/components/ErrorBoundary.tsx — dentro de componentDidCatch
componentDidCatch(error: Error, info: ErrorInfo) {
  // Detectar chunk load error e recarregar a página automaticamente
  const isChunkError = error.name === 'ChunkLoadError'
    || error.message.includes('Loading chunk')
    || error.message.includes('Failed to fetch dynamically imported module');

  if (isChunkError) {
    // Evitar loop de reload infinito com flag de sessão
    const alreadyReloaded = sessionStorage.getItem('__chunk_reload__');
    if (!alreadyReloaded) {
      sessionStorage.setItem('__chunk_reload__', '1');
      window.location.reload();
      return;
    }
    // Segunda vez: mostrar mensagem específica, não recarregar
    sessionStorage.removeItem('__chunk_reload__');
  }

  console.error('[ErrorBoundary]', error, info.componentStack);
  Sentry.captureException(error, {
    extra: { componentStack: info.componentStack },
    tags: {
      source: 'ErrorBoundary',
      isChunkError: String(isChunkError),
    },
  });
  toast({ ... });
}
```

**Adicionar mensagem específica no fallback para chunk errors:**

```tsx
// No estado do ErrorBoundary, adicionar flag:
interface State {
  hasError: boolean;
  isChunkError?: boolean;
}

// No fallback:
{this.state.isChunkError ? (
  <p className="text-sm text-muted-foreground">
    {i18n.t('errors.newVersionAvailable', 'Uma nova versão do app está disponível.')}
  </p>
) : (
  <p className="text-sm text-muted-foreground">
    {i18n.t('errors.unexpectedDesc')}
  </p>
)}
```

### Critérios de Aceite

- [ ] Ao simular `ChunkLoadError`, a página recarrega automaticamente (uma vez)
- [ ] Segundo `ChunkLoadError` na mesma sessão mostra mensagem "nova versão disponível"
- [ ] Nenhum loop infinito de reload

## TAREFA T5 — Fallback de Suspense com timeout visível

**Arquivo:** `src/App.tsx`, `src/components/LoadingSpinner.tsx`
**Pilar:** UX

### Problema

O `<Suspense fallback={<LoadingSpinner />}>` em `App.tsx` mostra um spinner genérico enquanto carrega os chunks lazy. Se o chunk demorar mais de 5 segundos (rede lenta, chunk grande), o usuário vê apenas um spinner sem nenhuma indicação de que algo pode estar errado.

### O que fazer

Criar um wrapper de Suspense que exibe mensagem adicional após 5 segundos:

```tsx
// src/components/SuspenseWithTimeout.tsx
import { Suspense, useState, useEffect, type ReactNode } from 'react';
import LoadingSpinner from './LoadingSpinner';
import { useTranslation } from 'react-i18next';

function TimeoutMessage() {
  const { t } = useTranslation();
  const [showMessage, setShowMessage] = useState(false);

  useEffect(() => {
    const timer = setTimeout(() => setShowMessage(true), 5000);
    return () => clearTimeout(timer);
  }, []);

  return (
    <div className="flex flex-col items-center gap-3">
      <LoadingSpinner />
      {showMessage && (
        <p className="text-xs text-muted-foreground animate-in fade-in">
          {t('common.loadingSlowNetwork', 'Carregando... Isso está demorando mais que o esperado.')}
        </p>
      )}
    </div>
  );
}

export function SuspenseWithTimeout({ children }: { children: ReactNode }) {
  return (
    <Suspense fallback={<TimeoutMessage />}>
      {children}
    </Suspense>
  );
}
```

Substituir em `App.tsx`:

```tsx
// ANTES
<Suspense fallback={<LoadingSpinner />}>
  <Routes>...</Routes>
</Suspense>

// DEPOIS
<SuspenseWithTimeout>
  <Routes>...</Routes>
</SuspenseWithTimeout>
```

### Critérios de Aceite

- [ ] Após 5 segundos de loading, mensagem "está demorando mais que o esperado" aparece
- [ ] Mensagem tem `animate-in fade-in` (não aparece bruscamente)
- [ ] Quando o chunk carrega normalmente (< 5s), a mensagem nunca é exibida

## Impacto Esperado

| Cenário | Antes | Depois |
|---------|-------|--------|
| Erro de render no canvas | Tela inteira quebra | Apenas o canvas quebra; billing/sidebar ok |
| Erro no modal de billing | Tela inteira quebra | Apenas o modal quebra; canvas preservado |
| Usuário acessa URL inexistente | 404 minimal sem navegação | 404 com branding, "voltar" e "ir para app" |
| Deploy novo enquanto usuário está online | Spinner eterno ou erro genérico | Reload automático com mensagem de nova versão |
| Carregamento de rota > 5s | Spinner sem feedback | Mensagem "demorando mais que o esperado" |
| Bug reportado por usuário | "Tive um erro" — sem como diagnosticar | "Tive um erro — ID: abc123" → busca imediata no Sentry |

## Dependência com PRD-0035

As tarefas T1 e T2 deste PRD referenciam `Sentry.captureException`. A integração do Sentry é definida no PRD-0035. **Este PRD deve ser implementado após o PRD-0035**, ou as chamadas ao Sentry devem ser condicionadas com `if (typeof Sentry !== 'undefined')` até que o PRD-0035 seja aplicado.
