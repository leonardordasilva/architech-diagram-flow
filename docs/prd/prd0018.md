### 3.1 A1 — Ativar pg_cron para purge automático semanal

**Arquivos afetados:**
- Nova migration SQL: `supabase/migrations/<timestamp>_enable_pg_cron_purge.sql`

**Situação atual:**

O PRD-17 criou a função `purge_old_soft_deleted_diagrams()` e o índice parcial
`idx_diagrams_deleted_at`, mas o agendamento via pg_cron não foi ativado. A
função só pode ser executada manualmente via SQL Editor. Com o tempo, registros
soft-deleted acumulam-se indefinidamente.

**Contexto técnico:**

O Supabase oferece a extensão `pg_cron` como add-on habilitável no dashboard
(Database → Extensions). Uma vez habilitada, jobs podem ser criados via SQL com
`cron.schedule()`. O schedule `'0 3 * * 0'` corresponde a "todos os domingos às
03:00 UTC", um horário de baixo tráfego.

**Impacto no score:** +1 ponto (pilar Banco & Escalabilidade)

**Correção esperada:**

Criar migration: `supabase/migrations/<timestamp>_enable_pg_cron_purge.sql`

```sql
-- A1: Habilitar pg_cron e agendar purge semanal de diagramas soft-deleted
--
-- PRÉ-REQUISITO: A extensão pg_cron deve estar habilitada no projeto Supabase.
-- No dashboard: Database → Extensions → procurar "pg_cron" → Enable.
-- Se já estiver habilitada, o CREATE EXTENSION abaixo é um no-op.

CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Agendar execução semanal: domingos às 03:00 UTC
-- Remove diagramas com deleted_at > 30 dias e seus diagram_shares associados
SELECT cron.schedule(
  'purge-deleted-diagrams',           -- nome único do job
  '0 3 * * 0',                        -- cron expression: domingo 03:00 UTC

  $$SELECT public.purge_old_soft_deleted_diagrams()$$
);

-- Comentário para documentação
COMMENT ON EXTENSION pg_cron IS
  'Agendador de jobs. Job ativo: purge-deleted-diagrams (dom 03:00 UTC). '
  'Verificar jobs: SELECT * FROM cron.job; '
  'Verificar execuções: SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 10;';
```

**Ação manual necessária antes de rodar a migration:**

1. Acessar o Supabase Dashboard do projeto
2. Navegar para **Database → Extensions**
3. Buscar `pg_cron`
4. Clicar em **Enable** (se ainda não estiver habilitado)
5. Aguardar a ativação (pode levar alguns segundos)
6. Rodar a migration normalmente

**Critérios de aceite:**
- A extensão `pg_cron` está habilitada no projeto Supabase.
- O job `purge-deleted-diagrams` aparece em `SELECT * FROM cron.job`.
- Executar `SELECT public.purge_old_soft_deleted_diagrams()` manualmente retorna o count esperado.
- Após a primeira execução agendada (domingo seguinte), verificar em `SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 5` que o job executou com sucesso (status `succeeded`).
- Diagramas com `deleted_at` < 30 dias NÃO são removidos.
- Diagramas com `deleted_at` > 30 dias e seus `diagram_shares` associados são removidos permanentemente.

### 3.2 A2 — Corrigir comentário de contrato de segurança em `saveSharedDiagram`

**Arquivo afetado:** `src/services/diagramService.ts`

**Situação atual:**

O JSDoc da função `saveSharedDiagram` contém um comentário de contrato de
segurança que referencia uma política RLS desatualizada:

```typescript
/**
 * Política RLS esperada (tabela diagrams, operação UPDATE):
 *   auth.uid() = owner_id  OR  share_token IS NOT NULL
 */
```

A política RLS real, após as correções dos PRDs 16 e 17, é:

```sql
CREATE POLICY "diagrams_update_policy"
  ON public.diagrams FOR UPDATE TO authenticated
  USING (
    deleted_at IS NULL
    AND (
      owner_id = auth.uid()
      OR EXISTS (
        SELECT 1 FROM public.diagram_shares
        WHERE diagram_shares.diagram_id = diagrams.id
          AND diagram_shares.shared_with_id = auth.uid()
      )
    )
  )
  ...
```

A discrepância entre o comentário (`share_token IS NOT NULL`) e a realidade
(`EXISTS(diagram_shares)`) pode induzir a erros em futuras auditorias ou
refatorações feitas via vibe coding.

**Impacto no score:** +1 ponto (pilar Segurança & Sessão — correção documental)

**Correção esperada:**

Substituir o bloco de JSDoc completo da função `saveSharedDiagram` em
`src/services/diagramService.ts`:

```typescript
/**
 * Salva alterações em um diagrama compartilhado.
 *
 * CONTRATO DE SEGURANÇA: esta função NÃO verifica owner_id no lado
 * cliente intencionalmente. A autorização é delegada inteiramente
 * à política RLS da tabela `diagrams` no Supabase, que garante
 * que somente colaboradores autorizados possam executar UPDATE.
 *
 * Política RLS atual (tabela diagrams, operação UPDATE):
 *   USING:
 *     deleted_at IS NULL
 *     AND (
 *       owner_id = auth.uid()
 *       OR EXISTS (
 *         SELECT 1 FROM diagram_shares
 *         WHERE diagram_shares.diagram_id = diagrams.id
 *           AND diagram_shares.shared_with_id = auth.uid()
 *       )
 *     )
 *
 * WITH CHECK:
 *     owner_id = auth.uid()
 *     OR EXISTS (
 *       SELECT 1 FROM diagram_shares
 *       WHERE diagram_shares.diagram_id = diagrams.id
 *         AND diagram_shares.shared_with_id = auth.uid()
 *     )
 *
 * Ou seja: o UPDATE só é permitido se o diagrama NÃO está soft-deleted
 * E o usuário autenticado é o proprietário OU possui um registro
 * explícito em diagram_shares.
 *
 * Nunca remova esta nota sem auditar as políticas RLS primeiro.
 * Última auditoria: 21/03/2026 (PRD-18, Score 93/100).
 */
```

**Critérios de aceite:**
- O comentário reflete exatamente a política RLS vigente.
- A referência `share_token IS NOT NULL` não existe mais no código.
- A data da última auditoria está documentada no comentário.
- `npm run build` passa sem erros.
- Nenhuma lógica da função foi alterada — apenas o JSDoc.

interface ProtectedRouteProps {
  children: React.ReactNode;
  redirectTo?: string;
}

export default function ProtectedRoute({ children, redirectTo = '/' }: ProtectedRouteProps) {
  const { user, loading } = useAuth();
  const { t } = useTranslation();

  if (loading) {
    return (
      <div className="flex h-screen items-center justify-center text-muted-foreground">

      </div>
    );
  }

  if (!user) return <Navigate to={redirectTo} replace />;

  return <>{children}</>;
}

A implementação completa do `RecoveryBanner` com i18n:

```typescript
import { useState, useEffect } from 'react';
import { getAutoSave, clearAutoSave, type AutoSaveData } from '@/hooks/useAutoSave';
import { useDiagramStore } from '@/store/diagramStore';
import { DbDiagramNodesSchema, DbDiagramEdgesSchema } from '@/schemas/diagramSchema';
import type { DiagramNode, DiagramEdge } from '@/types/diagram';
import { toast } from '@/hooks/use-toast';
import { X, FolderOpen } from 'lucide-react';
import { Button } from '@/components/ui/button';
import { useTranslation } from 'react-i18next';

const RECOVERY_FLAG_KEY = 'microflow_show_recovery';

export function triggerRecoveryBanner() {
  sessionStorage.setItem(RECOVERY_FLAG_KEY, '1');
}

export default function RecoveryBanner() {
  const { t } = useTranslation();
  const [savedData, setSavedData] = useState<AutoSaveData | null>(null);
  const [dismissed, setDismissed] = useState(false);
  const nodes = useDiagramStore((s) => s.nodes);

  useEffect(() => {
    const flag = sessionStorage.getItem(RECOVERY_FLAG_KEY);
    if (flag && nodes.length === 0 && !dismissed) {
      getAutoSave().then((data) => {
        setSavedData(data);
      });
      sessionStorage.removeItem(RECOVERY_FLAG_KEY);
    }
  }, []);

  if (!savedData || dismissed || nodes.length > 0) return null;

  const formattedDate = new Date(savedData.savedAt).toLocaleString(
    navigator.language.startsWith('pt') ? 'pt-BR' : 'en-US',
    { day: '2-digit', month: '2-digit', year: 'numeric', hour: '2-digit', minute: '2-digit' },
  );

  const handleRestore = () => {
    const nodesParsed = DbDiagramNodesSchema.safeParse(savedData.nodes);
    const edgesParsed = DbDiagramEdgesSchema.safeParse(savedData.edges);
    if (!nodesParsed.success || !edgesParsed.success) {
      toast({ title: t('recovery.corruptedData'), variant: 'destructive' });
      clearAutoSave();
      setDismissed(true);
      return;
    }
    const { loadDiagram, setDiagramName } = useDiagramStore.getState();
    loadDiagram(nodesParsed.data as DiagramNode[], edgesParsed.data as DiagramEdge[]);
    if (savedData.title) setDiagramName(savedData.title);
    setDismissed(true);
  };

  const handleDiscard = () => {
    clearAutoSave();
    setDismissed(true);
  };

  return (
    <div className="absolute top-2 left-1/2 -translate-x-1/2 z-50 animate-in slide-in-from-top-2 duration-300">
      <div className="flex items-center gap-3 rounded-lg border border-amber-500/30 bg-amber-900/80 px-4 py-2.5 shadow-lg backdrop-blur-sm">
        <FolderOpen className="h-4 w-4 text-amber-300 shrink-0" />
        <span className="text-sm text-amber-100">
          {t('recovery.title')}: "<strong>{savedData.title}</strong>" — {t('recovery.savedAt')} {formattedDate}
        </span>
        <div className="flex items-center gap-1.5 ml-2">
          <Button
            size="sm"
            variant="outline"
            className="h-7 text-xs border-amber-500/40 bg-amber-800/50 text-amber-100 hover:bg-amber-700/60"
            onClick={handleRestore}
          >
            {t('recovery.restore')}
          </Button>
          <Button
            size="sm"
            variant="ghost"
            className="h-7 text-xs text-amber-300 hover:bg-amber-800/50 hover:text-amber-100"
            onClick={handleDiscard}
          >
            <X className="h-3.5 w-3.5 mr-1" />
            {t('recovery.discard')}
          </Button>
        </div>
      </div>
    </div>
  );
}
```

Após a execução completa deste PRD, os únicos riscos remanescentes serão:

| Risco | Status | Justificativa |
|-------|--------|---------------|
| `style-src 'unsafe-inline'` no CSP | Aceito | Inerente ao Tailwind CSS + Radix UI. Removê-lo exigiria SSR ou migração de framework de estilos — custo desproporcional. O `unsafe-eval` já foi removido (PRD-16). |
| Sem landing page | Decisão de produto | A infraestrutura técnica suporta. Depende de definição de copy, design e estratégia comercial. |
| Migração i18n incompleta | Em progresso | Os componentes core foram migrados. Componentes complexos (Toolbar, DiagramCanvas, MyDiagrams, modais) ficam para PRDs futuros. A infraestrutura está validada em produção. |

Nenhum desses itens representa risco técnico significativo.

## 8. Critérios de Aceitação Globais

Após a execução de todos os apontamentos implementáveis, os seguintes critérios devem ser atendidos sem nenhuma regressão funcional:

- `npm run build` executa sem erros
- `npm run test` passa em TODOS os testes existentes (~56 testes)
- `npx tsc --noEmit` compila sem erros TypeScript
- A extensão pg_cron está habilitada no Supabase
- O job `purge-deleted-diagrams` aparece em `SELECT * FROM cron.job`
- O comentário JSDoc de `saveSharedDiagram` reflete a política RLS vigente
- Os componentes migrados exibem strings via `useTranslation()` em vez de hardcoded
- Alternar idioma para `en` via `i18n.changeLanguage('en')` reflete em todos os componentes migrados
- Nenhum comportamento funcional visível ao usuário foi alterado em pt-BR
- Nenhum componente visual mudou de aparência no idioma padrão (pt-BR)
