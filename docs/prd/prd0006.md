Épico 1 – Debounce no diagramName para preservar histórico de undo/redo Prioridade: 🟠 ALTA | Arquivos: src/components/DiagramCanvas.tsx, src/components/Toolbar.tsx, src/store/diagramStore.ts
Problema detalhado: O campo <input> no Toolbar.tsx chama onDiagramNameChange(e.target.value) a cada keystroke. Essa chamada aciona storeActions.setDiagramName, que executa set({ diagramName }) no Zustand. Como diagramName está incluído no partialize do zundo, cada caractere digitado adiciona uma entrada independente no histórico. Com um nome de 20 caracteres, o usuário esgota 20 das 50 entradas disponíveis. Pressionar Ctrl+Z reverte letra por letra em vez de desfazer a última ação de diagrama.
Raiz do problema no código atual:
Copy// Toolbar.tsx — dispara setDiagramName a cada tecla
<input
  value={diagramName}
  onChange={(e) => onDiagramNameChange(e.target.value)}  // ← sem debounce
  ...
/>

// diagramStore.ts — setDiagramName registrado pelo zundo
setDiagramName: (diagramName) => set({ diagramName }),  // ← grava no histórico
Solução — três partes coordenadas:
Parte 1: O Toolbar deve manter estado local para o valor visível do input (localName) e propagar para o store somente no onBlur ou Enter, nunca no onChange. Isso garante que o histórico receba apenas uma entrada por renomeação completa, não uma por caractere:
Copy// src/components/Toolbar.tsx
import { useState, useEffect } from 'react';

// Dentro do componente Toolbar, substituir o <input> atual por:
const [localName, setLocalName] = useState(diagramName);

// Sincronizar quando o store muda externamente (ex: loadDiagram, RecoveryBanner)
useEffect(() => {
  setLocalName(diagramName);
}, [diagramName]);

const commitName = () => {
  const trimmed = localName.trim();
  if (trimmed && trimmed !== diagramName) {
    onDiagramNameChange(trimmed);
  } else {
    // Revert local state if empty or unchanged
    setLocalName(diagramName);
  }
};

// JSX:
<input
  className="w-40 bg-transparent px-2 text-sm font-semibold text-foreground outline-none placeholder:text-muted-foreground"
  value={localName}
  onChange={(e) => setLocalName(e.target.value)}        // ← apenas estado local
  onBlur={commitName}                                    // ← propaga ao store
  onKeyDown={(e) => {
    if (e.key === 'Enter') {
      commitName();
      (e.target as HTMLInputElement).blur();
    }
    if (e.key === 'Escape') {
      setLocalName(diagramName);                         // ← descarta edição
      (e.target as HTMLInputElement).blur();
    }
  }}
  maxLength={100}
  placeholder="Nome do diagrama"
  aria-label="Nome do diagrama"
/>
{localName.length > 80 && (
  <span className="absolute -bottom-4 right-0 text-[10px] text-muted-foreground">
    {localName.length}/100
  </span>
)}
Copy
Parte 2: O DiagramCanvas.tsx passa onDiagramNameChange={storeActions.setDiagramName} para o Toolbar. Esta ligação já está correta e não precisa de alteração — a mudança no Toolbar isola o comportamento.
Parte 3: Adicionar teste em src/store/diagramStore.test.ts para garantir que múltiplas chamadas a setDiagramName com nomes diferentes geram entradas separadas no histórico (verificando o comportamento correto do zundo), e que uma única chamada gera uma única entrada:
Copydescribe('setDiagramName undo/redo', () => {
  it('reverte nome completo com um único undo', () => {
    getState().setDiagramName('Diagrama Antigo');
    useDiagramStore.temporal.getState().clear();

    // Uma única chamada ao store (simulando commit no blur)
    getState().setDiagramName('Diagrama Novo');
    expect(getState().diagramName).toBe('Diagrama Novo');

    useDiagramStore.temporal.getState().undo();
    expect(getState().diagramName).toBe('Diagrama Antigo');
  });

  it('duas renomeações distintas geram dois passos de undo', () => {
    getState().setDiagramName('Nome A');
    getState().setDiagramName('Nome B');
    expect(getState().diagramName).toBe('Nome B');

    useDiagramStore.temporal.getState().undo();
    expect(getState().diagramName).toBe('Nome A');

    useDiagramStore.temporal.getState().undo();
    expect(getState().diagramName).toBe('Novo Diagrama'); // valor inicial do resetStore
  });
});
Critérios de aceite:
•	Digitar um nome completo e pressionar Enter ou clicar fora gera uma única entrada no histórico de undo
•	Ctrl+Z após renomear reverte o nome inteiro, não letra a letra
•	Pressionar Escape durante a edição descarta a mudança e restaura o nome anterior
•	useEffect em Toolbar sincroniza o campo quando o store muda externamente (ex.: ao carregar diagrama salvo)
•	npm run test passa com os dois novos casos

Épico 2 – Substituir <input> nativo por <Input> Radix no Toolbar Prioridade: 🟡 MÉDIA | Arquivo: src/components/Toolbar.tsx
Problema detalhado: O Toolbar.tsx usa um <input> HTML nativo com classes Tailwind manuais, enquanto toda a aplicação usa o componente <Input> de @/components/ui/input (wrapper Radix). A inconsistência causa divergência visual em temas e no dark mode, pois o <input> nativo não herda automaticamente as variáveis CSS do tema Radix (--input, --ring, --foreground) aplicadas pelo <Input> padronizado.
Solução: Substituir o <input> nativo pelo componente <Input> do Radix, mantendo o comportamento implementado no Épico 1 (estado local + commit no blur/Enter/Escape):
Copy// src/components/Toolbar.tsx
import { Input } from '@/components/ui/input';

// Substituir o <input> nativo por:
<div className="relative">
  <Input
    className="h-8 w-40 text-sm font-semibold bg-transparent border-transparent 
               focus-visible:border-border focus-visible:ring-1 px-2"
    value={localName}
    onChange={(e) => setLocalName(e.target.value)}
    onBlur={commitName}
    onKeyDown={(e) => {
      if (e.key === 'Enter') {
        commitName();
        (e.target as HTMLInputElement).blur();
      }
      if (e.key === 'Escape') {
        setLocalName(diagramName);
        (e.target as HTMLInputElement).blur();
      }
    }}
    maxLength={100}
    placeholder="Nome do diagrama"
    aria-label="Nome do diagrama"
  />
  {localName.length > 80 && (
    <span className="absolute -bottom-4 right-0 text-[10px] text-muted-foreground">
      {localName.length}/100
    </span>
  )}
</div>
Copy
Observação: O estilo border-transparent focus-visible:border-border mantém o visual integrado ao toolbar (sem borda visível em repouso) enquanto respeita o sistema de design Radix em foco e no dark mode.
Critérios de aceite:
•	Campo renderiza com o componente <Input> do Radix, sem <input> nativo remanescente no Toolbar
•	Visual em repouso idêntico ao anterior (sem borda visível)
•	Em foco, exibe borda e ring do tema atual (claro e escuro)
•	Comportamento de commit/Escape do Épico 1 funciona normalmente

Épico 3 – Feedback visual de embedding no SpawnFromNodeModal Prioridade: 🟡 MÉDIA | Arquivo: src/components/SpawnFromNodeModal.tsx
Problema detalhado: Quando o usuário seleciona "Banco de Dados → Oracle" com um nó do tipo service como origem, o comportamento é embutir o Oracle dentro do nó de serviço (não cria um novo nó independente no canvas). Da mesma forma, selecionar "Microserviço" com um service como origem embute um serviço interno. O modal não comunica esse comportamento diferenciado: o usuário clica em "Criar", não vê nenhum novo nó no canvas, e fica confuso achando que houve um erro. A nota informativa atual ("O nó será criado próximo à origem...") fica abaixo do campo "Quantidade" em texto simples e não diferencia os casos.
Solução: Detectar os casos de embedding e exibir um Alert descritivo com ícone Info que aparece dinamicamente conforme a seleção do usuário:
Copy// src/components/SpawnFromNodeModal.tsx
import { Info } from 'lucide-react';
import { Alert, AlertDescription } from '@/components/ui/alert';

// Lógica de detecção de embedding — calcular dentro do componente:
const isEmbeddingOracle =
  isService && effectiveType === 'database' && subType === 'Oracle';
const isEmbeddingService =
  isService && effectiveType === 'service';
const isEmbedding = isEmbeddingOracle || isEmbeddingService;

// Mensagem dinâmica baseada no caso:
const embeddingMessage = isEmbeddingOracle
  ? `O Oracle será adicionado como banco interno do nó "${sourceNodeLabel}". Nenhum nó separado será criado no canvas.`
  : `O microserviço será adicionado como serviço interno do nó "${sourceNodeLabel}". Nenhum nó separado será criado no canvas.`;

const connectionMessage =
  `O nó será criado próximo a "${sourceNodeLabel}". Conecte-os manualmente arrastando entre os handles desejados.`;
Substituir a <p> informativa atual e adicionar o Alert condicional no corpo do modal, posicionado antes do campo "Quantidade" para garantir visibilidade:
Copy{/* Alert de embedding — exibido antes do campo Quantidade */}
{isEmbedding ? (
  <Alert className="border-blue-500/30 bg-blue-500/5">
    <Info className="h-4 w-4 text-blue-500" />
    <AlertDescription className="text-xs text-muted-foreground">
      {embeddingMessage}
    </AlertDescription>
  </Alert>
) : (
  <Alert className="border-muted bg-muted/30">
    <Info className="h-4 w-4 text-muted-foreground" />
    <AlertDescription className="text-xs text-muted-foreground">
      {connectionMessage}
    </AlertDescription>
  </Alert>
)}

{/* Campo de Quantidade — sem a <p> antiga */}
<div className="space-y-2">
  <Label>Quantidade</Label>
  <Input
    type="number"
    min={1}
    max={20}
    value={count}
    onChange={(e) => setCount(Math.max(1, parseInt(e.target.value) || 1))}
    // Desabilitar quantidade > 1 não faz sentido para embedding, mas manter habilitado
  />

Critérios de aceite:
•	Selecionar "Banco de Dados → Oracle" com origem service exibe Alert azul com mensagem de embedding do Oracle
•	Selecionar "Microserviço" com origem service exibe Alert azul com mensagem de embedding de serviço interno
•	Qualquer outra seleção exibe Alert neutro com mensagem de conexão manual
•	Trocar o tipo no Select atualiza o Alert dinamicamente sem fechar o modal
•	Aparência correta em dark mode com as classes dark: do Tailwind

Épico 5 – Testes E2E com Playwright (fluxo crítico) Prioridade: 🟢 BAIXA | Arquivos: e2e/ (nova pasta), package.json, playwright.config.ts (novos)
Problema: Após 6 ciclos, toda a cobertura é unitária. Nenhum teste cobre o fluxo crítico de ponta a ponta que integra IA, protocolo, canvas e persistência. Embora de baixa prioridade isoladamente, a ausência de testes E2E representa risco real para regressões silenciosas nos fluxos de maior valor para o usuário.
Escopo mínimo para o ciclo v6.0: Instalar Playwright e implementar um único teste E2E cobrindo o fluxo mais crítico da aplicação: adição manual de dois nós e criação de conexão com seleção de protocolo.

Copy// e2e/diagram-flow.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Fluxo básico de diagrama', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/');
    // Aguardar canvas carregar (página de autenticação ou canvas direto)
    await page.waitForLoadState('networkidle');
  });

  test('adiciona dois nós ao canvas via toolbar', async ({ page }) => {
    // Clicar no botão de Microserviço (Box icon) na toolbar
    await page.getByRole('button', { name: 'Microserviço' }).click();
    await page.getByRole('button', { name: 'Microserviço' }).click();

    // Verificar StatusBar mostra 2 nós
    await expect(page.getByText('2 nós')).toBeVisible();
  });

  test('campo de nome do diagrama faz commit apenas no blur', async ({ page }) => {
    const nameInput = page.getByRole('textbox', { name: 'Nome do diagrama' });

    // Digitar novo nome
    await nameInput.clear();
    await nameInput.type('Meu Diagrama Teste');

    // Antes do blur, o store ainda não foi atualizado (estado local)
    // Após blur, o store é atualizado
    await nameInput.blur();

    // Verificar que o nome aparece no overlay do canvas
    await expect(page.getByText('Meu Diagrama Teste')).toBeVisible();
  });

  test('undo após renomear reverte nome completo (não letra a letra)', async ({ page }) => {
    const nameInput = page.getByRole('textbox', { name: 'Nome do diagrama' });

    await nameInput.clear();
    await nameInput.type('Nome Completo');
    await nameInput.press('Enter');

    // Aguardar commit
    await expect(page.getByText('Nome Completo')).toBeVisible();

    // Ctrl+Z uma única vez deve reverter o nome inteiro
    await page.keyboard.press('Control+z');

    // Nome deve voltar ao valor anterior, não ficar como 'Nome Complet'
    await expect(page.getByText('Nome Completo')).not.toBeVisible();
  });
});
