PRD 0029 — Correções Críticas de Segurança (Code Review)
Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Code Review interno — Itens bloqueantes de segurança Plataforma: Lovable Cloud Premissas: Backend Supabase permanente. Sem CI/CD externo. Testes via Vitest + jsdom + MSW.

## Contexto

O code review identificou 2 vulnerabilidades críticas de segurança que devem ser corrigidas antes de qualquer release pública:

1. **Admin Guard por e-mail em variável de ambiente** — `VITE_*` é exposta no bundle do cliente, tornando o e-mail do admin visível a qualquer pessoa pelo DevTools.
2. **Information leakage nas mensagens de erro de autenticação** — mapear mensagens específicas do Supabase (ex: `'User already registered'`) permite *user enumeration* por atacantes.

Ambos os itens têm impacto direto na superfície de ataque da aplicação e não requerem mudança de arquitetura grande — apenas ajustes focados.

## TAREFA T1 — Admin check via campo `is_admin` no banco de dados

**Arquivo principal:** `src/pages/admin/AdminGuard.tsx`
**Pilar:** Segurança
**Risco que resolve:** Admin access controlado por variável de ambiente visível no bundle

### Problema

```typescript
// AdminGuard.tsx — situação atual
const adminEmail = import.meta.env.VITE_ADMIN_EMAIL?.trim().toLowerCase();
const userEmail = user?.email?.trim().toLowerCase();
if (!user || !adminEmail || userEmail !== adminEmail) {
  return <Navigate to="/" replace />;
}
```

Variáveis `VITE_*` são embutidas no bundle JavaScript durante o build. Qualquer pessoa pode abrir o DevTools, inspecionar os arquivos JS e descobrir o e-mail do administrador. Além disso, a proteção depende exclusivamente do endereço de e-mail — não há verificação server-side.

### O que fazer

**Passo 1 — Adicionar coluna `is_admin` na tabela `profiles`:**

Executar no Supabase SQL Editor:

```sql
-- Adicionar coluna is_admin à tabela profiles
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS is_admin BOOLEAN NOT NULL DEFAULT FALSE;

-- RLS: Apenas o próprio usuário pode ler seu is_admin
CREATE POLICY "Users can read own is_admin"
  ON profiles FOR SELECT
  USING (auth.uid() = id);

-- Apenas o service_role pode escrever is_admin (nunca o cliente)
CREATE POLICY "Only service_role can set is_admin"
  ON profiles FOR UPDATE
  USING (auth.role() = 'service_role');

-- Promover o admin atual (substituir pelo UUID real)
-- UPDATE profiles SET is_admin = TRUE WHERE email = 'admin@example.com';
```

**Passo 2 — Criar hook `useIsAdmin` em `src/hooks/useIsAdmin.ts`:**

```typescript
import { useQuery } from '@tanstack/react-query';
import { supabase } from '@/integrations/supabase/client';
import { useAuth } from '@/hooks/useAuth';

export function useIsAdmin(): boolean {
  const { user } = useAuth();

  const { data } = useQuery({
    queryKey: ['isAdmin', user?.id],
    queryFn: async () => {
      if (!user?.id) return false;
      const { data, error } = await supabase
        .from('profiles')
        .select('is_admin')
        .eq('id', user.id)
        .single();
      if (error) return false;
      return data?.is_admin === true;
    },
    enabled: !!user?.id,
    staleTime: 5 * 60 * 1000, // 5 minutos
  });

  return data ?? false;
}
```

**Passo 3 — Atualizar `AdminGuard.tsx` para usar o hook:**

```typescript
// ANTES
import { useAuth } from '@/hooks/useAuth';

const adminEmail = import.meta.env.VITE_ADMIN_EMAIL?.trim().toLowerCase();
const userEmail = user?.email?.trim().toLowerCase();
if (!user || !adminEmail || userEmail !== adminEmail) {
  return <Navigate to="/" replace />;
}

// DEPOIS
import { useIsAdmin } from '@/hooks/useIsAdmin';

const isAdmin = useIsAdmin();
if (!user || !isAdmin) {
  return <Navigate to="/" replace />;
}
```

**Passo 4 — Remover `VITE_ADMIN_EMAIL` de todos os arquivos:**
- `.env`, `.env.defaults`, `.env.example` — remover a variável
- `AdminGuard.tsx` — remover import e uso de `import.meta.env.VITE_ADMIN_EMAIL`

### Critérios de Aceite

- [ ] Coluna `is_admin` existe na tabela `profiles` com default `FALSE`
- [ ] RLS impede que o cliente altere `is_admin` diretamente
- [ ] `useIsAdmin` retorna `true` apenas para o usuário com `is_admin = TRUE` no banco
- [ ] `AdminGuard.tsx` não contém mais referência a `VITE_ADMIN_EMAIL`
- [ ] Nenhuma variável `VITE_ADMIN_EMAIL` existe nos arquivos `.env*`
- [ ] Rota `/admin` redireciona usuários não-admin para `/`
- [ ] Adicionar teste em `useIsAdmin.test.ts` cobrindo: usuário admin, usuário comum, usuário não autenticado

## TAREFA T2 — Generalizar mensagens de erro de autenticação

**Arquivo principal:** `src/pages/Auth.tsx`
**Pilar:** Segurança
**Risco que resolve:** User enumeration via mensagens de erro específicas

### Problema

```typescript
// Auth.tsx — situação atual
const msgMap: Record<string, string> = {
  'Invalid login credentials': t('auth.invalidCredentials'),
  'User already registered': t('auth.emailAlreadyUsed'),  // ← user enumeration
};
```

A mensagem `'User already registered'` (mapeada para uma tradução visível) confirma ao atacante que um determinado e-mail possui conta na plataforma. Isso viola a recomendação OWASP de não diferenciar "e-mail não existe" de "senha incorreta" nas respostas ao usuário.

### O que fazer

**Passo 1 — Atualizar o mapa de mensagens em `Auth.tsx`:**

```typescript
// ANTES
const msgMap: Record<string, string> = {
  'Invalid login credentials': t('auth.invalidCredentials'),
  'User already registered': t('auth.emailAlreadyUsed'),
};
const description = msgMap[authError.message] ?? authError.message;

// DEPOIS
// Mapeamento interno apenas para erros não-sensíveis de UX
const msgMap: Record<string, string> = {
  'Invalid login credentials': t('auth.invalidCredentials'),
  'Email not confirmed': t('auth.emailNotConfirmed'),
  'Too many requests': t('auth.tooManyRequests'),
};
// User already registered → mensagem genérica para evitar user enumeration
const description = msgMap[authError.message] ?? t('auth.genericError');
```

**Passo 2 — Adicionar chaves de tradução em todos os arquivos de locale (`src/i18n/`):**

```json
{
  "auth": {
    "genericError": "Não foi possível concluir a operação. Tente novamente.",
    "emailNotConfirmed": "Confirme seu e-mail antes de fazer login.",
    "tooManyRequests": "Muitas tentativas. Aguarde alguns minutos e tente novamente."
  }
}
```

**Passo 3 — Verificar o fluxo de signup:**

No handler de signup, se o objetivo for informar que o e-mail já existe sem vazar a informação, usar o fluxo "magic link silencioso" — enviar e-mail de confirmação normalmente sem indicar se a conta existe. O Supabase suporta este padrão nativamente quando `confirmations` estão habilitadas.

### Critérios de Aceite

- [ ] A mensagem `'User already registered'` não é mais mapeada para texto específico na UI
- [ ] Erros de autenticação desconhecidos caem em `t('auth.genericError')`
- [ ] Todos os locales têm a chave `auth.genericError` traduzida
- [ ] Fluxo de login com e-mail inexistente e com senha errada exibe a **mesma** mensagem ao usuário
- [ ] Teste em `Auth.test.tsx` cobre o comportamento de fallback para mensagem genérica
