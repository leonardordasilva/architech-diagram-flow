# PRD-0019 — Finalização i18n: camadas não-UI + correção de typo

**Projeto:** MicroFlow Architect  
**Data:** 21/03/2026  
**Autor:** Auditoria automatizada  
**Origem:** Relatório de auditoria pós-PRD-18 (score 98/100)  
**Objetivo:** Eliminar as 6 últimas ocorrências de strings hard-coded em português fora da camada de componentes React, corrigir typo visual em `ServiceNode`, e adicionar chaves i18n correspondentes nos dois locales. Score projetado: **98 → 99/100.**

---

## Contexto

A migração i18n executada no PRD-18 cobriu **todos os 28 componentes de UI** e **2 hooks** que utilizam `useTranslation()`. Restam, porém, **6 pontos** em camadas que não são componentes React (Zustand store, utility functions, hook sem acesso a hook React) onde o texto é hard-coded em português. Nesses contextos, a solução correta é importar a instância `i18next` diretamente (`import i18n from '@/i18n'`) e chamar `i18n.t('chave')` em tempo de execução — sem usar o hook `useTranslation`, que é exclusivo de componentes React.

Adicionalmente, o `ServiceNode.tsx` exibe `"Microservio"` (typo — faltam `ç` e `o`) como label estático do tipo do nó; a correção é substituir por chamada i18n.

## Ações

### A1 — Migrar `ServiceNode.tsx` para i18n (corrige typo)

**Arquivo:** `src/components/nodes/ServiceNode.tsx`

**Problema:** Linha ~100 contém `<div>Microservio</div>` — string hard-coded com typo (deveria ser "Microserviço").

**Solução:**
1. Adicionar `import { useTranslation } from 'react-i18next';` ao componente `ServiceNode`.
2. Dentro do corpo do `memo`, extrair `const { t } = useTranslation();`.
3. Substituir `Microservio` por `{t('nodes.service')}`.

**Chaves i18n utilizadas:** `nodes.service` — já existe em ambos os locales (`"Microserviço"` / `"Microservice"`).

**Critérios de aceitação:**
- O nó Service exibe "Microserviço" em pt-BR e "Microservice" em en.
- O typo "Microservio" não aparece mais em nenhum arquivo do projeto.
- Build e testes passam.

### A2 — Migrar `diagramStore.ts` para i18n (labels default + nome do diagrama)

**Arquivo:** `src/store/diagramStore.ts`

**Problemas identificados (3 ocorrências):**
1. **Linha ~68** — Estado inicial `diagramName: 'Novo Diagrama'` hard-coded.
2. **Linha ~125 (`addNode`)** — `labelMap` com `service: 'Microserviço'`.
3. **Linha ~160 (`addNodesFromSource`)** — Labels `'Biblioteca X'`, `'Microserviço'` (duas ocorrências de `labelMap` e `newSvcs.push`).

**Solução:**
1. Adicionar `import i18n from '@/i18n';` no topo do arquivo (importação direta da instância, sem hook).
2. Substituir `diagramName: 'Novo Diagrama'` por `diagramName: i18n.t('diagram.newDiagram')`.
3. Nos dois `labelMap` (em `addNode` e `addNodesFromSource`), substituir `'Microserviço'` por `i18n.t('nodes.service')`.
4. No embed de biblioteca (`addNodesFromSource`, bloco `subType === 'library'`), substituir `` `Biblioteca ${...}` `` por `` `${i18n.t('nodePanel.libDefault', { n: currentSvcs.length + i + 1 })}` `` (a chave `nodePanel.libDefault` já existe: `"DB {{n}}"` / `"Library {{n}}"`).

**Novas chaves i18n:** Nenhuma. Todas as chaves já existem nos locales.

**Critérios de aceitação:**
- `diagramName` inicial exibe "Novo Diagrama" em pt-BR e "New Diagram" em en.

### A3 — Migrar `connectionRules.ts` para i18n

**Arquivo:** `src/utils/connectionRules.ts`

**Problema:** A função `connectionErrorMessage` contém um `names: Record<NodeType, string>` com valores em português hard-coded: `'Microserviço'`, `'Banco de Dados'`, `'Fila/Mensageria'`, `'API/Protocolo'`. A string de template também está em português.

**Solução:**
1. Adicionar `import i18n from '@/i18n';` no topo.
2. Reescrever `connectionErrorMessage`:

```typescript
export function connectionErrorMessage(sourceType: NodeType, targetType: NodeType): string {
  const names: Record<NodeType, string> = {
    service: i18n.t('nodes.service'),
    database: i18n.t('nodes.database'),
    queue: i18n.t('nodes.queue'),
    external: i18n.t('nodes.external'),
  };
  return i18n.t('connectionError.message', {
    source: names[sourceType],
    target: names[targetType],
  });
}
```

3. Adicionar chave nova nos locales:

**`pt-BR.json`** — adicionar ao nível raiz:
```json
"connectionError": {
  "message": "{{source}} não pode se conectar a {{target}}"
}
```

**`en.json`** — adicionar ao nível raiz:
```json
"connectionError": {
  "message": "{{source}} cannot connect to {{target}}"
}
```

**Critérios de aceitação:**
- `connectionErrorMessage('database', 'queue')` retorna `"Banco de Dados não pode se conectar a Fila/Mensageria"` em pt-BR e `"Database cannot connect to Queue/Messaging"` em en.
- Testes existentes em `connectionRules.test.ts` continuam passando (ajustar assertions de string se necessário).
- Build compila sem erros.

### A4 — Migrar `useAutoSave.ts` para i18n

**Arquivo:** `src/hooks/useAutoSave.ts`

**Problemas identificados (2 ocorrências):**
1. **Linha ~112 (catch `QuotaExceededError`)** — Toast com `title: 'Armazenamento local cheio'` e `description: 'O diagrama muito grande para salvar localmente...'` hard-coded (também com typo — falta "é" em "é muito grande" e acento em "não").
2. **Linha ~148 (`getAutoSave`, fallback legado)** — `title: parsed.data.title ?? 'Sem título'` com fallback hard-coded (também com typo — `'Sem ttulo'` sem acento).

**Solução:**
1. Adicionar `import i18n from '@/i18n';` no topo do arquivo.
2. Substituir o toast:
```typescript
toast({
  title: i18n.t('autoSave.storageFull'),
  description: i18n.t('autoSave.storageFullDesc'),
  variant: 'destructive',
});
```
3. Substituir fallback legado:
```typescript
title: parsed.data.title ?? i18n.t('autoSave.untitled'),
```

4. Adicionar chaves nos locales:

**`pt-BR.json`** — adicionar namespace:
```json
"autoSave": {
  "storageFull": "Armazenamento local cheio",
  "storageFullDesc": "O diagrama é muito grande para salvar localmente. Salve na nuvem para não perder seu trabalho.",
  "untitled": "Sem título"
}
```

**`en.json`** — adicionar namespace:
```json
"autoSave": {
  "storageFull": "Local storage full",
  "storageFullDesc": "The diagram is too large to save locally. Save to cloud to preserve your work.",
  "untitled": "Untitled"
}
```

**Critérios de aceitação:**
- Toast de quota exceeded exibe texto traduzido no idioma ativo.
- Fallback de título legado usa texto traduzido.
- Typos corrigidos ("é muito grande", "não perder", "Sem título").
- Build compila sem erros.

## Impacto em testes

Os seguintes arquivos de teste podem necessitar ajuste de assertions de string:

1. **`connectionRules.test.ts`** — Os testes que verificam a mensagem de erro retornada por `connectionErrorMessage` comparam com strings em português. Após a migração, o valor dependerá do idioma ativo do i18n nos testes. **Recomendação:** Garantir que o `i18n` esteja configurado com `lng: 'pt-BR'` no ambiente de teste (o que já ocorre pelo fallback) ou ajustar as assertions para usar `i18n.t()`.

2. **`diagramStore.test.ts`** — Testes que verificam `diagramName` inicial ou labels de nós criados comparam com `'Novo Diagrama'`, `'Microserviço'`, etc. Mesma recomendação.

**Ação de mitigação:** Adicionar um `beforeAll` ou setup global nos testes que inicializa `i18n` com `lng: 'pt-BR'` para manter compatibilidade com as assertions existentes.
