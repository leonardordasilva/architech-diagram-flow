TAREFA 3 — Eliminação de unsafe-inline na CSP e implementação de nonces reais
Pilar: Segurança & Sessão Impacto estimado: +2 pontos Prioridade: Alta Risco que resolve: A CSP atual com script-src 'self' 'unsafe-inline' e style-src 'self' 'unsafe-inline' permite execução de scripts e estilos inline arbitrários, enfraquecendo significativamente a proteção contra XSS.

Especificação:

O vite.config.ts já configura html.cspNonce: 'placeholder', mas o index.html não usa nonces — usa unsafe-inline diretamente. A correção consiste em:

No index.html, substituir a CSP meta tag para usar nonces: script-src 'self' 'nonce-%%CSP_NONCE%%'; style-src 'self' 'nonce-%%CSP_NONCE%%'

Configurar o Vite para substituir %%CSP_NONCE%% no build. O plugin @vitejs/plugin-react-swc já suporta injeção de nonce em scripts inline gerados pelo build. A meta tag de CSP no index.html deve referenciar o nonce que o Vite injeta.

Atenção com Lovable Cloud: Se o Lovable Cloud controla o build, verificar se o pipeline de build do Lovable suporta a injeção de nonces. Se não suportar nonces dinâmicos (que idealmente seriam gerados por request no servidor), a alternativa é usar hashes de CSP: calcular o SHA-256 dos scripts inline gerados pelo Vite e adicioná-los como 'sha256-...' na CSP. Isso é estático e funciona com hosting estático.

Para estilos inline do Tailwind/Radix: Muitos componentes Radix e estilos inline do React (atributo style={}) requerem unsafe-inline para styles. A alternativa é usar style-src 'self' 'unsafe-inline' apenas para styles (risco menor que scripts) e remover unsafe-inline apenas do script-src. Isso já seria uma melhoria significativa.

Abordagem pragmática recomendada:

Copy<meta http-equiv="Content-Security-Policy"
  content="default-src 'self';
           script-src 'self';
           style-src 'self' 'unsafe-inline';
           font-src 'self';
           connect-src 'self' https://*.supabase.co wss://*.supabase.co;
           img-src 'self' data: blob: https://storage.googleapis.com https://*.supabase.co;
           worker-src 'self' blob:;
           frame-ancestors 'self';" />
Remover 'unsafe-inline' de script-src e manter apenas em style-src (necessário para Radix/inline styles). Testar se o Vite em modo production não injeta scripts inline — com @vitejs/plugin-react-swc, os scripts são module-type em arquivo, então não devem necessitar de unsafe-inline.

Critérios de aceite:

script-src não contém 'unsafe-inline'
A aplicação funciona em produção sem erros de CSP no console (nenhuma violação reportada)
frame-ancestors 'self' está presente (ver Tarefa 4)
Testes manuais: login, criar diagrama, salvar, exportar PNG/SVG, importar JSON — todos funcionam sem violação de CSP

TAREFA 6 — Rate limiter persistente nas Edge Functions
Pilar: Segurança & Sessão Impacto estimado: +1 ponto Prioridade: Média Risco que resolve: O rate limiter in-memory (Map<string, ...>) nas Edge Functions (diagram-crypto, share-diagram) se perde a cada cold start, tornando-o ineficaz contra ataques distribuídos.

Especificação:

Substituir o rateLimitMap in-memory por uma verificação contra o banco de dados. Dado que o Supabase não oferece Redis nativo, a abordagem mais pragmática é:

Opção A (recomendada) — Tabela rate_limits no Postgres:

Criar uma tabela:

CopyCREATE TABLE public.rate_limits (
  key         TEXT PRIMARY KEY,   -- 'crypto:{userId}' ou 'share:{userId}'
  count       INTEGER NOT NULL DEFAULT 1,
  window_start TIMESTAMPTZ NOT NULL DEFAULT now()
);
No início de cada Edge Function, executar uma RPC check_rate_limit(p_key, p_limit, p_window_seconds) que faz UPSERT atômico: se o window expirou, reseta; se não, incrementa; se count > limit, retorna false.

O custo é uma query a mais por request, mas a tabela é minúscula e a query é um simples UPSERT com PK lookup — sub-milissegundo.

Adicionar um job pg_cron para limpar entradas antigas: DELETE FROM rate_limits WHERE window_start < now() - interval '1 hour'.

Opção B — Manter in-memory mas aceitar como limitação documentada:

Se o overhead de 1 query extra por request for considerado inaceitável, documentar formalmente a limitação no ADR e adicionar um header X-RateLimit-Remaining na resposta para transparência ao cliente. Isso não resolve totalmente o risco, mas demonstra consciência.

Critérios de aceite:

Rate limit funciona corretamente mesmo após cold start da Edge Function
Um usuário que excede 60 req/min no diagram-crypto recebe HTTP 429
A limpeza de registros antigos está agendada
Testes manuais: fazer mais de 60 requests em 1 minuto e validar que o 61º é rejeitado
TAREFA 7 — Consolidação da criptografia + persistência para reduzir round-trips
Pilar: Performance & Requisições Impacto estimado: +1 ponto Prioridade: Média Risco que resolve: Cada save executa 2 chamadas de rede sequenciais: diagram-crypto (encrypt) → supabase.from('diagrams').update(). Em redes lentas, a latência total pode ser >200ms.

Especificação:

Criar uma nova Edge Function save-diagram que recebe o payload completo (title, nodes, edges, diagramId, ownerId, workspaceId) e internamente: (1) criptografa nodes e edges, (2) computa content_hash, (3) faz o UPSERT na tabela usando service_role. Retorna o registro salvo.

Isso elimina um round-trip completo: o browser faz 1 request em vez de 2.

A Edge Function deve usar SUPABASE_SERVICE_ROLE_KEY para o write, mas validar JWT do usuário e verificar ownership/permissão antes de escrever (equivalente ao que o RLS faz, mas server-side, já que service_role bypassa RLS).

Atenção: Essa abordagem move a lógica de autorização de RLS para código na Edge Function. Para manter a defesa em profundidade, a Edge Function deve explicitamente verificar: (a) o user é owner do diagrama, OU (b) o user tem share ativo, OU (c) o user é membro do workspace com role editor/owner. Replicar a mesma lógica do RLS diagrams_update_policy.

Alternativa mais simples: manter 2 calls mas torná-las paralelas onde possível. Neste caso, como encrypt precisa ocorrer ANTES do update, a paralelização não é possível. Portanto, a consolidação em 1 Edge Function é a abordagem correta.

O diagramService.ts no cliente seria refatorado: saveDiagram() e saveSharedDiagram() passariam a chamar supabase.functions.invoke('save-diagram', ...) em vez de encryptDiagramData() + supabase.from('diagrams').update().

Critérios de aceite:

O save de diagrama faz exatamente 1 request de rede (Edge Function)
Latência média de save reduzida em pelo menos 30% (medido no Network tab)
Testes existentes (useSaveDiagram.test.ts, diagramService.test.ts) adaptados e passando
Fallback: se a nova Edge Function estiver indisponível, o fluxo antigo (2 calls) continua funcional

TAREFA 9 — Refino da política RLS share_token IS NOT NULL
Pilar: Banco & Escalabilidade Impacto estimado: +1 ponto Prioridade: Alta Risco que resolve: A política SELECT de diagrams inclui OR share_token IS NOT NULL, o que torna todo diagrama com share_token legível por qualquer usuário autenticado. Isso pode causar vazamento de dados.

Especificação:

A intenção original é permitir que diagramas compartilhados sejam acessíveis via token. Contudo, a cláusula share_token IS NOT NULL na política SELECT não verifica se o usuário de fato possui ou conhece o token — apenas que o diagrama tem um token gerado.

Correção: Remover OR share_token IS NOT NULL da política RLS diagrams_select_policy. O acesso via share_token deve ser feito exclusivamente pela RPC get_diagram_by_share_token, que já existe e recebe o token como parâmetro. A RPC é SECURITY DEFINER, então ela pode ler o diagrama independente das políticas RLS do caller.

Para acesso anônimo (visitantes não autenticados vendo diagrama compartilhado), a RPC já suporta isso — basta dar GRANT EXECUTE ... TO anon se não estiver dado (verificar no migration existente).

Migration SQL:

Copy-- Remove share_token IS NOT NULL da policy SELECT
DROP POLICY IF EXISTS "diagrams_select_policy" ON public.diagrams;

CREATE POLICY "diagrams_select_policy"
  ON public.diagrams FOR SELECT
  USING (
    deleted_at IS NULL AND (
      owner_id = auth.uid()
      OR EXISTS (
        SELECT 1 FROM public.diagram_shares ds
        WHERE ds.diagram_id = id AND ds.shared_with_id = auth.uid()
      )
      OR (
        workspace_id IS NOT NULL AND EXISTS (
          SELECT 1 FROM public.workspace_members wm
          WHERE wm.workspace_id = diagrams.workspace_id
            AND wm.user_id = auth.uid()
            AND wm.accepted_at IS NOT NULL
        )
      )
    )
  );
Critérios de aceite:

Um usuário autenticado que NÃO é owner, NÃO tem share, e NÃO é membro do workspace NÃO consegue listar o diagrama via .from('diagrams').select()
O acesso via share link (/diagram/:shareToken) continua funcionando via RPC
Testes do SharedDiagram page continuam passando

TAREFA 11 — Backfill de criptografia e content_hash para dados legados
