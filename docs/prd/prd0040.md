### T1 — [BLOCKING B1] Habilitar source maps + Sentry Vite Plugin

**Problema:** `vite.config.ts` não tem `build.sourcemap: true` e o plugin `@sentry/vite-plugin` não está instalado. Isso torna os stack traces no Sentry completamente ilegíveis em produção (referências a código minificado).

**Arquivos:**
- `vite.config.ts`
- `package.json`

**Mudanças em `vite.config.ts`:**

1. Importar o plugin no topo do arquivo:
```typescript
import { sentryVitePlugin } from "@sentry/vite-plugin";
```

2. Adicionar `build.sourcemap: true` no objeto de configuração (dentro do `return {}`):
```typescript
build: {
  sourcemap: true,  // ← ADICIONAR
  rollupOptions: {
    output: {
      manualChunks: {
        "react-flow": ["@xyflow/react"],
        "dagre-layout": ["dagre"],
        "export-utils": ["html-to-image"],
        supabase: ["@supabase/supabase-js"],
      },
    },
  },
},
```

3. Adicionar o plugin Sentry **após** o plugin `react()`, dentro do array `plugins`. O plugin só deve rodar quando `SENTRY_AUTH_TOKEN` estiver definido (evita falha em builds locais sem o token):
```typescript
plugins: [
  react(),
  ...(env.SENTRY_AUTH_TOKEN
    ? [
        sentryVitePlugin({
          org: env.SENTRY_ORG || "microflow",
          project: env.SENTRY_PROJECT || "microflow-architect",
          authToken: env.SENTRY_AUTH_TOKEN,
          telemetry: false,
        }),
      ]
    : []),
],
```

**Instalação do pacote:**
```
npm install --save-dev @sentry/vite-plugin
```

**Variáveis de ambiente a adicionar no Lovable (Secrets):**
- `SENTRY_AUTH_TOKEN` — token de auth do Sentry (Settings > Auth Tokens)
- `SENTRY_ORG` — slug da organização no Sentry
- `SENTRY_PROJECT` — slug do projeto no Sentry

**Critério de aceite:** Build de produção sobe os source maps para o Sentry. Stack traces no Sentry mostram linha/coluna real do código TypeScript original.

### T2 — [BLOCKING B2] Corrigir parser regex em `brevo.ts`

**Problema:** Quando `payload.from` não contém `<email>` (ex: `"noreply@microflow.com.br"` sem display name), a linha:
```typescript
const fromMatch = payload.from.match(/^(.*?)\s*<(.+)>$/) ?? [null, payload.from, payload.from];
```
Faz `fromMatch[1]` ser `"noreply@microflow.com.br"` (truthy), portanto `senderName` fica com o endereço de e-mail como nome. O Brevo rejeita ou exibe nomes incorretos.

**Arquivo:** `supabase/functions/_shared/brevo.ts`

**Correção — substituir as linhas 22-24:**

Substituir:
```typescript
const fromMatch = payload.from.match(/^(.*?)\s*<(.+)>$/) ?? [null, payload.from, payload.from];
const senderName = fromMatch[1]?.trim() || 'MicroFlow';
const senderEmail = fromMatch[2]?.trim() || payload.from;
```

Por:
```typescript
const fromMatch = payload.from.match(/^(.*?)\s*<(.+)>$/);
const senderName = fromMatch?.[1]?.trim() || 'MicroFlow';
const senderEmail = fromMatch?.[2]?.trim() || payload.from;
```

**Critério de aceite:** `sendBrevoEmail({ from: 'noreply@microflow.com.br', ... })` envia com `sender.name = 'MicroFlow'` e `sender.email = 'noreply@microflow.com.br'`.

### T3 — [IMPORTANT I1] Não silenciar `ChunkLoadError` no Sentry

**Problema:** `src/main.tsx` tem `ignoreErrors: ['ChunkLoadError', ...]`. Isso impede que erros de chunk load cheguem ao Sentry com `level: 'warning'`, perdendo visibilidade de deploys com cache stale.

**Arquivo:** `src/main.tsx`

**Correção:** Remover `'ChunkLoadError'` do array `ignoreErrors` em `Sentry.init(...)`.

Localizar no `ignoreErrors`:
```typescript
'ChunkLoadError',
```
E remover essa linha.

**Contexto:** O `ErrorBoundary.tsx` já trata o ChunkLoadError com auto-reload + flag de sessão. Mas queremos que o Sentry o registre como `warning` para monitoramento de cache stale em produção, não como erro silenciado.

**Critério de aceite:** ChunkLoadError aparece no Sentry com severity `warning` (não `error`) em vez de ser descartado.

### T4 — [IMPORTANT I2] Corrigir resolução de plano em `invoice.paid`

**Problema:** O handler `invoice.paid` em `stripe-webhook/index.ts` usa `dbSub.plan` (lido do banco) para sincronizar `profiles.plan`. Se o usuário fez upgrade/downgrade via Stripe Portal antes da renovação, o plano no banco pode estar desatualizado. A fonte de verdade é o `priceId` da assinatura do Stripe.

**Arquivo:** `supabase/functions/stripe-webhook/index.ts`

**Localizar o handler `invoice.paid`** (aproximadamente linha 251-294):

O trecho atual que deve ser **substituído**:
```typescript
const sub = await stripe.subscriptions.retrieve(subscriptionId);

await serviceSupabase
  .from("subscriptions")
  .update({
    status: "active",
    current_period_start: new Date(sub.current_period_start * 1000).toISOString(),
    current_period_end: new Date(sub.current_period_end * 1000).toISOString(),
    updated_at: new Date().toISOString(),
  })
  .eq("user_id", dbSub.user_id);

// Ensure profiles.plan is in sync
await serviceSupabase
  .from("profiles")
  .update({ plan: dbSub.plan })
  .eq("id", dbSub.user_id);
```

**Substituir por:**
```typescript
const sub = await stripe.subscriptions.retrieve(subscriptionId);

// Resolve plan from actual price ID — reliable even after portal upgrades/downgrades
const renewedPriceId = sub.items.data[0]?.price?.id ?? "";
const resolvedPlan = renewedPriceId
  ? resolvePlanFromPriceId(renewedPriceId)
  : dbSub.plan;

await serviceSupabase
  .from("subscriptions")
  .update({
    status: "active",
    plan: resolvedPlan,
    current_period_start: new Date(sub.current_period_start * 1000).toISOString(),
    current_period_end: new Date(sub.current_period_end * 1000).toISOString(),
    updated_at: new Date().toISOString(),
  })
  .eq("user_id", dbSub.user_id);

// Ensure profiles.plan is in sync with the resolved plan
await serviceSupabase
  .from("profiles")
  .update({ plan: resolvedPlan })
  .eq("id", dbSub.user_id);
```

**Critério de aceite:** Após upgrade via Stripe Portal seguido de renovação, `invoice.paid` persiste o plano correto (ex: `team`) em vez do plano anterior (ex: `pro`).

### T5 — [IMPORTANT I3] Corrigir mensagem de log `isForbidden` em `process-email-queue`

**Problema:** O log/DLQ reason diz `'Emails disabled for this project'` — mensagem legada do provider anterior (Lovable). Com Brevo, um 403 significa API key inválida ou plano excedido.

**Arquivo:** `supabase/functions/process-email-queue/index.ts`

**Localizar (aproximadamente linha 305-311):**
```typescript
if (isForbidden(error)) {
  await moveToDlq(supabase, queue, msg, 'Emails disabled for this project')
  return new Response(
    JSON.stringify({ processed: totalProcessed, stopped: 'emails_disabled' }),
    { headers: { 'Content-Type': 'application/json' } }
  )
}
```

**Substituir por:**
```typescript
if (isForbidden(error)) {
  await moveToDlq(supabase, queue, msg, 'Brevo API key invalid or plan exceeded (403)')
  return new Response(
    JSON.stringify({ processed: totalProcessed, stopped: 'brevo_forbidden' }),
    { headers: { 'Content-Type': 'application/json' } }
  )
}
```

**Critério de aceite:** Logs e DLQ reason refletem a causa real (Brevo 403) em vez da mensagem do provider anterior.

**Problema:** `src/pages/Privacy.tsx` contém o placeholder `CNPJ a definir` / `CNPJ to be defined`. Páginas legais com placeholders visíveis passam má impressão e podem gerar problemas de conformidade.

**Arquivo:** `src/pages/Privacy.tsx`

**Localizar (aproximadamente linha 33-34):**
```typescript
? 'MicroFlow Architect é operado por MicroFlow Tecnologia Ltda. (CNPJ a definir). Encarregado de Proteção de Dados (DPO): privacidade@microflow.dev'
: 'MicroFlow Architect is operated by MicroFlow Tecnologia Ltda. (CNPJ to be defined). Data Protection Officer (DPO): privacidade@microflow.dev'
```

**Substituir** `CNPJ a definir` e `CNPJ to be defined` pelo número de CNPJ real da empresa quando disponível.

**Critério de aceite:** Página de Privacidade não exibe "a definir" ou "to be defined" para o usuário final.

### T8 — [MINOR N4] Adicionar `replyTo` no `sendBrevoEmail`

**Problema:** E-mails enviados pelo sistema não têm `Reply-To` configurado. Usuários que respondem a e-mails transacionais (ex: confirmação de pagamento) recebem erro de endereço inválido.

**Arquivo:** `supabase/functions/_shared/brevo.ts`

**Modificar o objeto `body`** para incluir `replyTo` quando não definido explicitamente:

1. Adicionar campo opcional na interface `BrevoEmailPayload`:
```typescript
export interface BrevoEmailPayload {
  to: string;
  from: string;
  subject: string;
  html: string;
  text?: string;
  messageId?: string;
  replyTo?: string;  // ← ADICIONAR
}
```

2. No objeto `body` dentro de `sendBrevoEmail`, adicionar após o campo `to`:
```typescript
const body = {
  sender: { name: senderName, email: senderEmail },
  to: [{ email: payload.to }],
  replyTo: { email: payload.replyTo ?? 'suporte@microflow.dev' },  // ← ADICIONAR
  subject: payload.subject,
  htmlContent: payload.html,
  ...(payload.text ? { textContent: payload.text } : {}),
  ...(payload.messageId ? { headers: { 'Message-ID': payload.messageId } } : {}),
};
```

**Critério de aceite:** E-mails transacionais têm `Reply-To: suporte@microflow.dev` por padrão, permitindo respostas diretas dos usuários.
