Projeto: MicroFlow Architect (architech-diagram-flow) Data: 09/04/2026 Origem: Code Review interno — Itens menores (recomendados) Plataforma: Lovable Cloud Premissas: Backend Supabase permanente. Sem CI/CD externo. Testes via Vitest + jsdom + MSW.

## Contexto

O code review identificou 4 pontos de melhoria de manutenibilidade e qualidade que não bloqueam funcionalidades, mas acumulam débito técnico e podem causar bugs sutis no futuro:

1. **`unescape()` depreciado em `useAutoSave`** — API pré-ES6 que pode ser removida de engines modernas.
2. **Comparação de timestamp como string ISO no Realtime** — múltiplos saves no mesmo milissegundo podem causar updates descartados.
3. **`InternalItemSchema` com 4 variações** — dificulta manutenção e onboarding.
4. **`DiagramCanvas.tsx` com 367 linhas** — baixa coesão, difícil de testar e revisar.

## TAREFA T1 — Substituir `unescape()` por `TextEncoder` em `useAutoSave`

**Arquivo:** `src/hooks/useAutoSave.ts`
**Pilar:** Manutenibilidade
**Risco que resolve:** Uso de API depreciada que pode ser removida de engines futuras

### Problema

```typescript
// useAutoSave.ts — situação atual
async function compressString(input: string): Promise<string> {
  if (!SUPPORTS_COMPRESSION) {
    return btoa(unescape(encodeURIComponent(input))); // ← unescape() é depreciado
  }
  // ...
}
```

`unescape()` foi depreciado no ECMAScript 3 e está na lista de remoção de engines modernas. A combinação `btoa(unescape(encodeURIComponent(str)))` é um hack legacy para converter strings UTF-8 para base64.

### O que fazer

```typescript
// DEPOIS — fallback moderno usando TextEncoder
async function compressString(input: string): Promise<string> {
  if (!SUPPORTS_COMPRESSION) {
    // Converter string UTF-8 para Uint8Array e depois para base64
    const bytes = new TextEncoder().encode(input);
    const binString = Array.from(bytes, (byte) => String.fromCodePoint(byte)).join('');
    return btoa(binString);
  }
  // ... código CompressionStream existente permanece igual
}

async function decompressString(input: string): Promise<string> {
  if (!SUPPORTS_COMPRESSION) {
    // Reverso: base64 → Uint8Array → string UTF-8
    const binString = atob(input);
    const bytes = Uint8Array.from(binString, (ch) => ch.codePointAt(0)!);
    return new TextDecoder().decode(bytes);
  }
  // ... código DecompressionStream existente permanece igual
}
```

**Atenção:** Verificar se há dados comprimidos com o formato antigo em IndexedDB. Se sim, a função `decompressString` precisa detectar o formato e fazer fallback para o decode antigo antes de migrar completamente.

### Critérios de Aceite

- [ ] `unescape()` não existe mais em nenhum arquivo de produção
- [ ] `compressString` e `decompressString` usam `TextEncoder`/`TextDecoder`
- [ ] Dados salvos com o formato antigo continuam sendo lidos corretamente (backward compat)
- [ ] Teste em `useAutoSave.test.ts` cobre compress → decompress round-trip com caracteres UTF-8 (acentos, emoji)

## TAREFA T2 — Adicionar `revision_number` para comparação de updates no Realtime

**Arquivo:** `src/hooks/useRealtimeCollab.ts`
**Pilar:** Correctness
**Risco que resolve:** Updates descartados quando múltiplos saves ocorrem no mesmo milissegundo

### Problema

```typescript
// useRealtimeCollab.ts — situação atual
const lastUpdatedAtRef = useRef<string>('');
if (remoteUpdatedAt && remoteUpdatedAt === lastUpdatedAtRef.current) {
  return; // descarta update se timestamp igual
}
```

Timestamps ISO (`2026-04-09T12:00:00.000Z`) têm precisão de milissegundo. Se dois saves ocorrem no mesmo milissegundo (ex: autosave + save manual simultâneos), o segundo update é descartado.

### O que fazer

**Passo 1 — Adicionar coluna `revision_number` na tabela `diagrams`:**

```sql
ALTER TABLE diagrams ADD COLUMN IF NOT EXISTS revision_number BIGINT NOT NULL DEFAULT 0;

-- Trigger para auto-incrementar a cada update
CREATE OR REPLACE FUNCTION increment_revision_number()
RETURNS TRIGGER AS $$
BEGIN
  NEW.revision_number = OLD.revision_number + 1;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER diagrams_revision_trigger
BEFORE UPDATE ON diagrams
FOR EACH ROW EXECUTE FUNCTION increment_revision_number();
```

**Passo 2 — Atualizar `useRealtimeCollab.ts` para usar `revision_number`:**

```typescript
// DEPOIS
const lastRevisionRef = useRef<number>(-1);

// No handler de UPDATE:
const remoteRevision: number = payload.new.revision_number ?? -1;
if (remoteRevision <= lastRevisionRef.current) {
  return; // já temos versão igual ou mais recente
}
lastRevisionRef.current = remoteRevision;
// ... aplicar update normalmente
```

**Passo 3 — Manter o check de `updated_at` como fallback:**

```typescript
// Se revision_number não estiver disponível (ex: diagrama criado antes da migração)
if (remoteRevision === -1) {
  if (remoteUpdatedAt && remoteUpdatedAt === lastUpdatedAtRef.current) return;
  lastUpdatedAtRef.current = remoteUpdatedAt ?? '';
}
```

### Critérios de Aceite

- [ ] Coluna `revision_number` existe em `diagrams` com trigger de auto-incremento
- [ ] `useRealtimeCollab` usa `revision_number` como critério primário de deduplicação
- [ ] Diagramas sem `revision_number` (criados antes da migração) continuam funcionando via fallback de `updated_at`
- [ ] Teste cobre o caso de dois updates com mesmo `updated_at` mas `revision_number` diferente

## TAREFA T3 — Padronizar `InternalItemSchema` para 2 formatos

**Arquivo:** `src/schemas/diagramSchema.ts`
**Pilar:** Manutenibilidade
**Risco que resolve:** 4 variações do mesmo schema dificultam manutenção e leitura

### Problema

```typescript
// diagramSchema.ts — situação atual
const InternalItemSchema = z.union([
  z.string().min(1),                                          // formato 1: string
  z.object({ label: z.string(), dbType: z.string() }),        // formato 2: objeto com dbType
  z.object({ id: z.string(), label: z.string() })             // formato 3: objeto com id (legado)
    .transform((obj) => obj.label),
  z.object({ label: z.string() })                             // formato 4: objeto só com label (legado)
    .transform((obj) => obj.label),
]);
```

Os formatos 3 e 4 são legados (dados antigos no banco). Manter 4 variações aumenta a complexidade cognitiva e o risco de introduzir bugs em refatorações futuras.

### O que fazer

**Passo 1 — Consolidar formatos legados em um único transform:**

```typescript
// DEPOIS — 2 formatos canônicos + transform unificado para legados
const InternalItemSchema = z.union([
  // Formato canônico 1: string simples
  z.string().min(1),

  // Formato canônico 2: objeto com label e dbType opcional
  z.object({
    label: z.string(),
    dbType: z.string().optional(),
    id: z.string().optional(),
  }).transform((obj) => {
    // Se tiver dbType, manter como objeto (formato atual)
    if (obj.dbType) return { label: obj.label, dbType: obj.dbType };
    // Legado: objetos com apenas label ou id+label → normalizar para string
    return obj.label;
  }),
]);
```

**Passo 2 — Adicionar comentário explicando a estratégia de backward compat:**

```typescript
/**
 * Schema para items internos de nós (fields de Database, endpoints de API, etc.)
 *
 * Formato canônico atual:
 *   - string: label simples
 *   - { label: string; dbType: string }: campo de banco com tipo
 *
 * Formatos legados (dados criados antes de 01/2026) são normalizados pelo transform.
 * Migração server-side planejada para PRD-0031-T3.
 * @see docs/prd/prd0031.md
 */
```

**Passo 3 (Opcional, se viável) — Script de migração de dados legados:**

```sql
-- Normalizar objetos { id, label } e { label } para strings em nodes existentes
-- ATENÇÃO: executar com backup ativo
-- UPDATE diagrams SET nodes = ... WHERE nodes::text LIKE '%"id":"%"label":%';
-- (implementação depende do formato exato dos dados legados)
```

### Critérios de Aceite

- [ ] `InternalItemSchema` aceita no máximo 2 variantes de entrada (não 4)
- [ ] Dados legados (formato 3 e 4) continuam sendo lidos corretamente via transform
- [ ] Testes de snapshot existentes continuam passando
- [ ] Comentário JSDoc documenta os formatos aceitos e a estratégia de migração

## TAREFA T4 — Refatorar `DiagramCanvas.tsx` em sub-componentes

**Arquivo:** `src/components/DiagramCanvas.tsx`
**Pilar:** Manutenibilidade
**Risco que resolve:** 367 linhas em um único componente, difícil de testar e revisar

### Problema

`DiagramCanvas.tsx` concentra responsabilidades demais: configuração do ReactFlow, toolbar, header, handlers de teclado, overlays de colaboração e recovery banner. Com 367 linhas, qualquer mudança requer leitura e compreensão de contextos não relacionados.

### O que fazer

**Passo 1 — Identificar blocos a extrair:**

| Bloco atual | Novo arquivo | Responsabilidade |
|-------------|--------------|-----------------|
| `<header>` com Toolbar + WorkspaceContextSelector + DiagramHeader | `src/components/diagram/DiagramToolbarStrip.tsx` | Barra superior com todas as ações |
| Recovery banner + status overlay | `src/components/diagram/DiagramStatusOverlays.tsx` | Overlays de estado (saving, recovery, offline) |
| Configuração de `nodeTypes` e `edgeTypes` | `src/components/diagram/diagramFlowConfig.ts` | Mapa de tipos (objeto estático, fora do componente) |

**Passo 2 — Estrutura de arquivos resultante:**

```
src/components/
  DiagramCanvas.tsx              (~180 linhas — orquestrador)
  diagram/
    DiagramToolbarStrip.tsx      (~80 linhas)
    DiagramStatusOverlays.tsx    (~60 linhas)
    diagramFlowConfig.ts         (~30 linhas — nodeTypes, edgeTypes, defaultEdgeOptions)
```

**Passo 3 — `DiagramCanvas.tsx` resultante deve conter apenas:**

```typescript
export function DiagramCanvas() {

```

**Passo 4 — Verificar se `nodeTypes` e `edgeTypes` estão definidos fora do render:**

```typescript
// diagramFlowConfig.ts — definir fora do componente para evitar re-criação a cada render
export const nodeTypes: NodeTypes = {
```

Se já estiverem fora do componente, apenas mover para o arquivo de config sem mudança de comportamento.

### Critérios de Aceite

- [ ] `DiagramCanvas.tsx` tem no máximo 200 linhas após a refatoração
