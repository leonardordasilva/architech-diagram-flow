## TAREFA T1 — Corrigir estado inconsistente em `useSaveDiagram`

**Arquivo:** `src/hooks/useSaveDiagram.ts`
**Pilar:** Correctness
**Risco que resolve:** `diagramId` pode ficar dessincronizado se `saveDiagram` falhar após persistir no banco

### Problema

```typescript
// useSaveDiagram.ts — trecho atual (aproximado, linhas 67-94)
try {
  lastSaveTimestampRef.current = Date.now();
  const record = await saveDiagram(diagramName, nodes, edges, user.id, diagramId, workspaceId);
  setDiagramId(record.id);          // ← só executado se não lançar
  if (isNewDiagram) {
    queryClient.invalidateQueries({ queryKey: ['diagrams', user.id] });
  }
} catch (err: any) {
  toast({ ... });
}
```

Se `saveDiagram` lançar erro **depois** de persistir no banco (ex: falha ao descriptografar a resposta), o banco tem o novo `id` mas o estado React ainda tem o `id` antigo (ou `undefined`). Na próxima tentativa de save, o sistema tenta criar um novo registro em vez de atualizar o existente, podendo gerar duplicatas.

### O que fazer

**Passo 1 — Introduzir variável de resultado antes do bloco try/catch:**

```typescript
// DEPOIS — useSaveDiagram.ts
let savedRecord: { id: string } | null = null;

try {
  lastSaveTimestampRef.current = Date.now();
  if (isCollaborator && diagramId) {
    await saveSharedDiagram(diagramId, nodes, edges);
  } else {
    savedRecord = await saveDiagram(diagramName, nodes, edges, user.id, diagramId, workspaceId);
  }
} catch (err: unknown) {
  toast({ title: t('errors.saveFailed'), description: getErrorMessage(err), variant: 'destructive' });
  return;
}

// Atualizar estado apenas após confirmação de sucesso
if (savedRecord) {
  setDiagramId(savedRecord.id);
  if (isNewDiagram) {
    queryClient.invalidateQueries({ queryKey: ['diagrams', user.id] });
  }
}
```

**Passo 2 — Verificar se `saveDiagram` em `diagramService.ts` é idempotente:**

Confirmar que, se chamada com o mesmo `diagramId`, a função executa um `upsert` (não `insert`). Se não for idempotente, adicionar guard:

```typescript
// diagramService.ts — verificar se existe upsert por id
const { data, error } = await supabase
  .from('diagrams')
  .upsert({ id: diagramId ?? undefined, ...payload }, { onConflict: 'id' })
  .select()
  .single();
```

### Critérios de Aceite

- [ ] `setDiagramId` só é chamado após confirmação de que `saveDiagram` retornou com sucesso
- [ ] Um erro em `saveDiagram` não altera o estado de `diagramId`
- [ ] Nenhuma duplicata de diagrama é criada após falha de save

## TAREFA T2 — Resolver pipeline de criptografia no canal Realtime

**Arquivo:** `src/hooks/useRealtimeCollab.ts`
**Pilar:** Correctness
**Risco que resolve:** Colaboradores perdem mudanças remotas silenciosamente quando dados chegam criptografados

### Problema

```typescript
// useRealtimeCollab.ts — workaround atual (linhas ~193-196)
// FIX: After save, DB stores encrypted envelopes (objects with iv/ciphertext).
// Postgres Realtime delivers these raw encrypted objects. Pushing them into the
// store would corrupt React Flow, causing "e is not iterable"
if (!Array.isArray(remoteNodes) || !Array.isArray(remoteEdges)) {
  return; // silently ignores encrypted updates
}
```

O canal Realtime do Supabase entrega o estado atual da linha no banco — que contém os dados criptografados (`{ iv, ciphertext }`). O hook detecta que os dados não são arrays e descarta o update silenciosamente. Resultado: mudanças feitas por colaboradores em diagramas criptografados nunca chegam aos outros participantes em tempo real.

### O que fazer

**Opção A (Recomendada) — Descriptografar no cliente ao receber o update Realtime:**

```typescript
// useRealtimeCollab.ts — após receber payload do canal
import { decryptDiagramData } from '@/services/cryptoService';

// Dentro do handler de UPDATE:
let remoteNodes = payload.new.nodes;
let remoteEdges = payload.new.edges;

// Se dados chegarem criptografados, descriptografar antes de usar
if (remoteNodes && !Array.isArray(remoteNodes)) {
  try {
    const decrypted = await decryptDiagramData(remoteNodes, remoteEdges);
    remoteNodes = decrypted.nodes;
    remoteEdges = decrypted.edges;
  } catch {
    console.warn('[useRealtimeCollab] Falha ao descriptografar update remoto — ignorando');
    return;
  }
}

if (!Array.isArray(remoteNodes) || !Array.isArray(remoteEdges)) {
  return;
}
```

**Opção B (Alternativa) — Usar Broadcast em vez de Realtime DB para colaboração:**

Em vez de escutar mudanças na tabela (`supabase.channel(...).on('postgres_changes', ...)`), usar o canal de broadcast do Supabase para enviar os dados já descriptografados diretamente entre clientes:

```typescript
// Ao salvar, fazer broadcast dos dados plaintext para o canal
channel.send({
  type: 'broadcast',
  event: 'diagram_update',
  payload: { nodes, edges, updatedAt: new Date().toISOString() },
});
```

Esta opção elimina o problema de criptografia no Realtime, mas requer que todos os participantes estejam online no momento do broadcast (sem persistência da mensagem).

**Decisão:** Verificar qual `decryptDiagramData` está disponível em `cryptoService.ts` e aplicar a Opção A como fix imediato, mantendo a Opção B como refatoração futura.

### Critérios de Aceite

- [ ] Mudanças feitas por colaboradores chegam em tempo real para outros participantes em diagramas criptografados
- [ ] O `console.warn` do workaround anterior é substituído por lógica de descriptografia
- [ ] Se a descriptografia falhar, o update é descartado com log de aviso (sem crash)
- [ ] Teste de integração cobre o fluxo: save criptografado → Realtime → outro cliente recebe dados plaintext

## TAREFA T3 — Adicionar testes para `useSaveDiagram` e `useRealtimeCollab`

**Arquivos:** `src/hooks/useSaveDiagram.ts`, `src/hooks/useRealtimeCollab.ts`
**Pilar:** Qualidade / Testes
**Risco que resolve:** Regressões silenciosas em hooks de save e colaboração

### Problema

`useSaveDiagram` é o hook mais crítico da aplicação (lida com persistência), mas não possui arquivo de testes. `useRealtimeCollab` tem cobertura superficial e não testa o caminho de dados criptografados.

### O que fazer

**Criar `src/hooks/useSaveDiagram.test.ts` com os seguintes casos:**

```typescript
describe('useSaveDiagram', () => {
  // Caminho feliz
  it('salva diagrama novo e atualiza diagramId no store')
  it('atualiza diagrama existente sem criar duplicata')
  
  // Limites de plano
  it('bloqueia save quando limite de diagramas do plano é atingido')
  it('exibe toast de upgrade quando limite é atingido')
  
  // Diagrama compartilhado
  it('usa saveSharedDiagram quando usuário é colaborador')
  it('usa saveDiagram quando usuário é o dono')
  
  // Tratamento de erros
  it('não atualiza diagramId quando saveDiagram lança erro')
  it('exibe toast de erro quando save falha')
  
  // Throttle / cooldown
  it('não executa save se chamado dentro do SAVE_COOLDOWN_MS')
  
  // Invalidação de query
  it('invalida query de diagramas apenas para diagrama novo')
})
```

**Atualizar `src/hooks/useRealtimeCollab.test.ts` adicionando:**

```typescript
describe('useRealtimeCollab — dados criptografados', () => {
  it('descriptografa nodes/edges criptografados recebidos via Realtime')
  it('descarta update quando descriptografia falha')
  it('não aplica update remoto quando updatedAt não mudou')
  it('pausa undo/redo durante aplicação de update remoto')
})
```

**Usar MSW para mockar:**
- `supabase.from('diagrams').upsert(...)` → resposta de sucesso e de erro
- Canal Realtime → simular evento `UPDATE` com dados criptografados e plaintext

- [ ] `useSaveDiagram.test.ts` criado com no mínimo 8 casos de teste passando
